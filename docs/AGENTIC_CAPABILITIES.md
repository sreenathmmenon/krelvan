# Agentic Capabilities — Research, Roadmap, and Sample Plugins

> **Thesis.** A capability is *agentic* only when an LLM plans, uses tools, observes real environment feedback, reflects, and re-decides in a loop until a stopping condition it reasons about — choosing each step rather than walking a human-authored path. That is the dividing line between a *workflow* (predefined code paths) and an *agent* (the model dynamically directs its own process). Everything in this catalog clears that bar; everything excluded — webhook glue, when-X-do-Y triggers, CRUD connectors, chat bots — does not, because it has no loop, no environment-grounded re-decision, and no reasoning about when to stop or escalate. The cross-source signal is loud and convergent: intelligence is now cheap and abundant, so the binding constraint has moved to **verification and coordination** — agent throughput already exceeds human oversight. That is *exactly* where Krelvan's ledger-first, verifiable model is uniquely positioned: deny-by-default admission, supervisor co-sign, secret broker, full audit trail, reserved-then-settled integer-cent budgets, crash-safe idempotent replay, counterfactual safety, and six side-effect classes that drive automatic approval gates are not features bolted onto an agent — they are the precise infrastructure the entire industry has identified as the missing layer. Krelvan is therefore uniquely suited to *host risky agentic capabilities safely*: the riskier the action, the more it depends on the assurance stack Krelvan already treats as primary.

## 1. What makes a capability agentic

Genuinely agentic means **an LLM that plans, uses tools, observes real environment feedback, reflects, and re-decides in a loop until a stopping condition** — choosing each step rather than following a human-authored path. This is Anthropic's explicit dividing line between a *workflow* (predefined code paths) and an *agent* (the model dynamically directs its own process). The agent owns the control flow: it decides what to do next, when it has enough, when to escalate, and when to stop.

### The distinction at a glance

| Genuinely agentic | Pre-agentic (excluded) |
| --- | --- |
| Generated, revisable step graph | Hardcoded step graph |
| Re-decides from observed environment feedback | Fires on a fixed trigger |
| Reasons about when to stop / escalate | No stopping/escalation reasoning |
| Open-ended action space (discovered at runtime) | Pre-wired connectors only |
| Loop: plan → act → observe → reflect → re-decide | Linear, one-shot, or static rule |

### Explicitly excluded (NOT agentic)

- **Webhook glue** — passing payloads between systems with no reasoning loop.
- **When-X-do-Y triggers** — fixed condition→action rules; the step graph is authored by a human, not generated by the model.
- **CRUD connectors** — read/write a record on command; no investigation, no re-decision.
- **Chat bots** — converse without acting on the world, observing feedback, or re-planning.

These are excluded because they have no loop, no environment-grounded re-decision, and no reasoning about when to stop or escalate. Where a thin connector *does* appear in the catalog below (e.g. `http.wrap_api`), it is included strictly as a no-code on-ramp to genuinely agentic *composition* — the agent selecting and composing it at runtime is the agentic part, not the wrapper itself.

### Why Krelvan is the right host

De-duplicating five independent source areas (Anthropic research, a16z theses, labs/VC reports, academic papers, vertical case studies) yields a remarkably convergent stack:

1. **Perception–action substrate** — `code.execute`, `browser.act`: run novel code or drive any GUI and observe the result.
2. **Reasoning layer** — `plan.decompose`, `orchestrate.delegate`, `research.investigate`, `data.analyze`, agentic RAG: decompose goals, fan out sub-agents, self-direct investigation.
3. **Memory / self-improvement layer** — `memory.persist`, reflect/consolidate, `skills.acquire_reuse`: learn across sessions, grow the action repertoire.
4. **Extensibility layer** — `tools.discover`, MCP admission, YAML API-wrap: discover the action space at runtime.
5. **Assurance-and-effect layer** — verify (cite/critique/counterfactual), `hitl.escalate`, durable replay, and the real-world-effect ladder (`act.execute_effect` → `spend` → `provision_irreversible` → `identity`): gate, budget, and audit consequential actions.

The capabilities form a **risk ladder** that maps cleanly onto Krelvan's six side-effect classes: `read → write-reversible → write-irreversible → spend → message-human → identity-mutation`. a16z's "Know Your Agent" thesis — the agent economy's bottleneck is *identity, not intelligence* — lands directly on Krelvan's `identity-mutation` class plus its signed execution record, which **is** the verifiable proof a counterparty needs. The frontier is no longer demonstrating that agents can act (browser use, coding, research are table-stakes among labs) but proving they acted *correctly, within authority, and within budget* — verifiably, replayably, and reversibly where possible. Krelvan's job is not to race to ship the flashiest action; it is to be the only platform where the flashiest actions are safe to delegate.

## 2. The catalog, by category

Each capability lists: **what it does**, **why it's agentic**, **side-effect class**, **default autonomy**, **authoring path**, and **sources**. The *Krelvan fit* note explains how the platform's trust model governs it.

### Computer / Browser Use

#### `browser.act`
- **What it does:** Drives any GUI (web browser, desktop app, CLI) via an observe-plan-act loop: takes screenshots or reads the DOM/accessibility tree, then clicks, types, scrolls, fills forms, and navigates multi-page flows to complete tasks on software that exposes no API. Backed by a sandboxed VM or Playwright MCP; falls back to human takeover for credential/payment entry.
- **Why agentic:** The agent perceives an unknown GUI at runtime and reasons step-by-step about what to do next, recovering from misclicks and adapting to dynamic pages. It acts on real software as a human would when no API exists — the antithesis of a pre-wired connector. Only viable with vision-capable reasoning models. WebArena/OSWorld measure exactly this open-ended capability.
- **Side-effect class:** `write-irreversible` · **Default autonomy:** `act-with-veto` · **Authoring path:** TypeScript
- **Krelvan fit:** A flagship capability backed by a sandboxed VM. Each browser action is ledger-logged; the supervisor co-signs before form submission/checkout. The credential-entry takeover maps to the secret broker (plugin drives the field, never sees the raw secret). Screenshots become replayable audit artifacts. High side-effect (a submitted order is irreversible) so it parks for approval by default — Operator/Mariner already gate order submission, validating the design.
- **Sources:** Anthropic computer use; Operator, Project Mariner, Gemini 2.5 Computer Use; OSWorld, WebArena, Surfer 2.

### Autonomous Code

#### `code.engineer`
- **What it does:** Takes a high-level engineering goal or GitHub issue, clones a repo into a sandbox, explores it, makes multi-file edits, runs tests/CI, reads tracebacks, iterates until green, and opens a reviewer-gated PR. Long-running and asynchronous (minutes to hours), using a purpose-built agent-computer interface rather than raw shell.
- **Why agentic:** Closes the write→run→observe→repair loop autonomously over a long horizon using execution feedback (not human review) as the verification signal. SWE-agent showed interface design rivals model capability. Impossible for trigger-action glue that cannot reason over test output it has never seen.
- **Side-effect class:** `write-reversible` · **Default autonomy:** `act-with-veto` · **Authoring path:** TypeScript
- **Krelvan fit:** Edits are write-reversible under version control; the PR merge/push escalates to write-irreversible and parks for approval (reviewer-gated PR = act-with-veto). Per-run sandbox cost reserved-then-settled in integer cents. The CI signal becomes the supervisor's co-sign input. Durable checkpointing lets a multi-hour run survive crashes via idempotent replay.
- **Sources:** Anthropic autonomous engineering; Open SWE (reviewer-gated PRs); SWE-agent (NeurIPS 2024); SWE-bench Verified.

#### `code.execute`
- **What it does:** Spins up an isolated container with a persistent Python/Bash session, stateful shell, and file editor where the agent writes and runs novel code, installs packages, generates files (charts, data outputs), reads back results/errors, and self-corrects in a single session.
- **Why agentic:** The agent authors code to solve a problem it was not pre-programmed for, observes the actual execution result, and revises — generative computation in a closed perception-action loop. It is the substrate beneath data analysis, deep research, and autonomous coding; the agent decides when text reasoning is insufficient and offloads to deterministic computation.
- **Side-effect class:** `write-reversible` · **Default autonomy:** `full` · **Authoring path:** core
- **Krelvan fit:** The shared substrate under `code.engineer`, `data.analyze`, and research. Container is gated, budgeted per container-hour (reserved-then-settled integer cents), and audited. Side-effects are isolated inside the sandbox, which is what makes `full` autonomy safe here. Programmatic Tool Calling can run inside it.
- **Sources:** Anthropic code execution tool; Code Interpreter; E2B, Bedrock AgentCore; Daytona, OpenHands, Modal.

### Planning & Orchestration

#### `plan.decompose`
- **What it does:** Breaks a high-level outcome into ordered subgoals, sequences them, tracks progress, maps each subgoal to concrete actions, and replans mid-execution when intermediate results invalidate the plan.
- **Why agentic:** Long-horizon decomposition with mid-execution replanning is a core reasoning-model behavior. Pre-agentic flows hardcode the step graph; here the step graph is generated and revised by the agent from observed feedback. It is the core agent loop (plan, act, observe, re-decide) that every node's autonomy setting governs.
- **Side-effect class:** `read` · **Default autonomy:** `full` · **Authoring path:** core
- **Krelvan fit:** The runtime each Krelvan node's autonomy setting actually governs — the loop with environment feedback and stopping conditions. Pure planning is read-class; the actions it emits inherit their own side-effect classes and gates. A foundational core primitive, not a plugin.
- **Sources:** Anthropic agent loop; a16z deep multistep; Lilian Weng, ReAct, Generative Agents.

#### `orchestrate.delegate`
- **What it does:** A lead/orchestrator agent dynamically decomposes a goal it cannot predict in advance, spawns 3–5 (up to thousands) specialized worker subagents each with its own context window, tool set, objective, output format and task boundaries, runs them in parallel, validates and synthesizes their condensed results, and iterates by spawning more.
- **Why agentic:** The subtasks are NOT predefined — the orchestrator invents them at runtime from the input. Anthropic's production research system spawns 3–5 subagents and beat single-agent Opus by 90.2%; parallelism cut research time up to 90%. a16z calls recursive fan-out of 5,000 sub-tasks the new bottleneck (coordination, not computation). Reasoning-driven division of labor, not a static DAG.
- **Side-effect class:** `write-reversible` · **Default autonomy:** `act-with-veto` · **Authoring path:** core
- **Krelvan fit:** Maps to parent-node delegation where each subagent is a separately-budgeted, separately-autonomy-gated node. Per-node budget reservation prevents runaway fan-out spend; the supervisor co-signs the aggregated result before it settles. Coordination/locking/state across parallel nodes is exactly Krelvan's orchestration value — the moat a16z identifies.
- **Sources:** Anthropic orchestrator-subagents; a16z recursive fan-out; CrewAI, OpenAI Agents SDK handoffs; AutoGen, MetaGPT.

#### `plan.orchestrate_as_code`
- **What it does:** The agent writes a script that calls multiple discovered tools with loops, conditionals, `asyncio.gather` parallelism, and error handling inside the sandbox, filtering/transforming intermediate results locally and pausing only when it needs a tool result — returning only what matters to the model context.
- **Why agentic:** The agent expresses its multi-tool orchestration AS CODE and filters intermediate data before reasoning (Anthropic: 200KB→1KB, 98.7% context reduction). Control flow over tools is generated by the model, not authored by a human as a flowchart — the defining difference from trigger-action chains.
- **Side-effect class:** `write-reversible` · **Default autonomy:** `act-with-veto` · **Authoring path:** TypeScript
- **Krelvan fit:** A node that emits an executable plan over other Krelvan capability plugins. Each tool call the script makes is still individually classified, budgeted, and gated by the trust layer — so code-driven orchestration does not bypass approval gates. Runs inside `code.execute`. Lets one node compound many capabilities while preserving the ledger.
- **Sources:** Anthropic programmatic orchestration; Code execution with MCP.

### Deep Research

#### `research.investigate`
- **What it does:** Given an open question, plans a multi-step research trajectory: generates its own search queries, fetches and reads sources, runs code over data/PDFs, uses interleaved thinking to evaluate source quality, decides breadth vs depth and when it has enough evidence, pivots on new findings, and synthesizes a structured report.
- **Why agentic:** The agent writes its OWN queries and adaptively decides when to search vs read and when it has enough — search is a means it controls, not a fixed step. RL-trained end-to-end planning reacts to information mid-task (backtracking, re-querying). A research analyst, not a search connector.
- **Side-effect class:** `read` · **Default autonomy:** `full` · **Authoring path:** core
- **Krelvan fit:** A composite read-class capability over `web.search` + `web.fetch` + `code.execute`. Read-class means `full` autonomy is safe — no irreversible effects. Pairs mandatorily with `verify.cite_sources` before the result is accepted, making verifiability a first-class output that fits the ledger model.
- **Sources:** Anthropic multistep research; a16z deep multistep; OpenAI Deep Research.

### Self-Verification

#### `verify.cite_sources`
- **What it does:** A dedicated verification pass (a separate agent from the writer) reads the raw source documents AND the produced report, attributes every claim to a specific source location, and flags unsupported claims.
- **Why agentic:** Verification is treated as a distinct agentic role checking against ground truth, avoiding the "game of telephone" of self-recollection. Adversarial fact-grounding by a separate agent is a genuinely agentic assurance primitive, not validation plumbing.
- **Side-effect class:** `read` · **Default autonomy:** `full` · **Authoring path:** core
- **Krelvan fit:** This IS the Krelvan supervisor co-sign, made concrete for research output. Per-claim source attribution becomes a ledger artifact — exactly the "execution records + cryptographic guarantees" a16z says verification demands once throughput exceeds human oversight.
- **Sources:** Anthropic evaluator-optimizer (CitationAgent); cite-claims-against-sources literature.

#### `verify.critique_and_revise`
- **What it does:** A generator-evaluator loop: the agent produces a candidate, a separate evaluator (or the model with tool-grounded feedback) critiques it against criteria, generates its own test cases, runs verification, and the generator revises in a loop until checks pass — before any output or high-side-effect action commits.
- **Why agentic:** Self-critique and adversarial checking of the agent's OWN output is a reasoning behavior (Reflexion/CRITIC/GenRM). Execution-grounded verification let agents self-correct 70.3% of the time. The system improves an answer through iterative judgment, not by re-running a fixed script. a16z: "verification is the binding constraint."
- **Side-effect class:** `read` · **Default autonomy:** `full` · **Authoring path:** core
- **Krelvan fit:** The mandatory critique node before any write-irreversible / spend / identity-mutation action settles — the natural enforcement point for required assurance. Literature warns models over-trust their own outputs, which is exactly why Krelvan uses a supervisor co-sign (cross-checker) rather than pure self-review. Counterfactual-safety lives here.
- **Sources:** Anthropic evaluator-optimizer; a16z supervisor co-sign; Sequoia, CrewAI manager validation; Reflexion, CRITIC, GenRM; ReVeal, VerifiAgent.

#### `simulate.counterfactual_dryrun`
- **What it does:** Runs a hypothetical/what-if scenario before committing: dry-runs an action, simulates multiple candidate plans, or models outcomes to predict consequences before taking a real-world action.
- **Why agentic:** Reasoning over hypothetical futures to choose a plan — the agent evaluates consequences counterfactually before acting, rather than acting and observing harm.
- **Side-effect class:** `read` · **Default autonomy:** `full` · **Authoring path:** core
- **Krelvan fit:** Directly implements Krelvan's counterfactual safety: `estimateCents` and a dry-run before any spend or write-irreversible call. The dry-run result feeds the budget reservation and the supervisor co-sign, making the cost-and-reversibility reasoning auditable in the ledger.
- **Sources:** Generative Agents lookahead planning; Anthropic estimate-before-commit.

### Data & Analysis

#### `data.analyze`
- **What it does:** Given a business question, the agent plans the analytical workflow, writes and runs SQL/Python over a dataset in a persistent session, performs anomaly detection / segmentation / root-cause, produces charts, and outputs reproducible analysis code plus a narrative report.
- **Why agentic:** Goes from natural-language question to executed analysis to findings with no human writing code at each step — selecting methods, iterating on results, interpreting output. The agent reasons about what analysis to run, not rendering a fixed dashboard query.
- **Side-effect class:** `read` · **Default autonomy:** `full` · **Authoring path:** core
- **Krelvan fit:** Read-class (analysis over data, no mutation) so `full` autonomy is appropriate; runs on `code.execute`. Reproducible code output is a ledger artifact — the same run can be idempotently replayed and re-verified. A high-leverage horizontal capability for a self-hostable platform.
- **Sources:** Databricks Agentic Analytics; "Many AI Analysts, One Dataset."

### Agentic RAG & Knowledge

#### `rag.retrieve_evaluate_refine`
- **What it does:** Delegates the retrieval strategy itself to an agent that chooses among multiple backends, reformulates queries, performs multi-hop traversal, and runs a retrieve→evaluate→refine loop over private corpora until the context is judged sufficient.
- **Why agentic:** Retrieval becomes a planned, self-correcting investigation rather than a single similarity lookup; the agent judges sufficiency and iterates. Reflection + planning + tool use applied to knowledge access (+15–25% on HotpotQA multi-hop), far beyond flat RAG plumbing.
- **Side-effect class:** `read` · **Default autonomy:** `full` · **Authoring path:** YAML
- **Krelvan fit:** A read-class composite capability over a vector/graph backend. High-leverage for a self-hostable platform: every org has a private corpus. Authorable as YAML (wrap a retrieval HTTP API) for the simple case or TypeScript for custom multi-hop logic. Sufficiency-judgment loop is ledger-logged.
- **Sources:** Agentic RAG Survey; Graph RAG.

#### `knowledge.build_graph`
- **What it does:** Reads a private corpus and builds a structured knowledge graph (entities + typed relations) enabling explicit multi-hop traversal and compositional reasoning, then queries, validates, and optimizes its own graph search strategies.
- **Why agentic:** The agent imposes structure on unstructured knowledge and reasons compositionally over it — knowledge organization as an autonomous reasoning act, supporting disambiguation and multi-hop answers a flat index cannot.
- **Side-effect class:** `write-reversible` · **Default autonomy:** `full` · **Authoring path:** TypeScript
- **Krelvan fit:** Writing graph nodes/edges is write-reversible and ledger-tracked. Complements `rag.retrieve_evaluate_refine` (graph is the backend it traverses). The graph build is reproducible and idempotently replayable. TypeScript plugin since it needs custom extraction logic.
- **Sources:** Graph-Based Agentic RAG; self-correcting Agentic Graph RAG.

### Memory & Self-Improvement

#### `memory.persist`
- **What it does:** A file/store-backed memory the agent writes, reads, updates, and deletes OUTSIDE the context window — episodic (timestamped experiences) and semantic (vector recall over unlimited history) — retrieved by relevance/recency/importance, paired with context editing that auto-prunes stale tool calls.
- **Why agentic:** Gives the agent self-managed working context and continual learning across sessions so it can run long-horizon tasks and improve over time. Anthropic measured +39% on complex multi-step tasks; the lead research agent saves its plan to memory to survive past 200K tokens. State the agent curates itself, not a CRUD record.
- **Side-effect class:** `write-reversible` · **Default autonomy:** `full` · **Authoring path:** core
- **Krelvan fit:** Writes are write-reversible and ledger-tracked, user-scoped. `full` autonomy is fine because writes are reversible and auditable. Foundational for long-horizon nodes; the saved plan also aids crash-safe idempotent replay (the agent resumes from its own memory).
- **Sources:** Anthropic persistent cross-session memory; a16z memory/personalize; Letta/MemGPT Memory Blocks; Generative Agents, Lilian Weng.

#### `memory.reflect_consolidate`
- **What it does:** A background/sleep-time process runs asynchronously during idle time to synthesize raw memories into higher-level insights, resolve contradictions between stored facts, abstract recurring patterns and failure modes, and restructure the agent's own memory.
- **Why agentic:** Continual self-improvement and reflection decoupled from any user request — the agent improves itself offline ("thinks during downtime"). Genuinely post-agentic; no automation tool reflects on its own experience. Produced emergent coordinated behavior in Generative Agents with no explicit instruction.
- **Side-effect class:** `write-reversible` · **Default autonomy:** `full` · **Authoring path:** core
- **Krelvan fit:** Reversible writes to the agent's own memory, fully ledger-logged so consolidation decisions are auditable and reversible. A differentiator for a self-hostable platform: agents that compound competence on the owner's data without retraining. Runs as a scheduled background node.
- **Sources:** Letta Sleep-time Compute; Generative Agents; Claude Dreaming offline consolidation.

#### `skills.acquire_reuse`
- **What it does:** The agent packages a learned workflow/expertise as a reusable skill (folder of instructions + scripts + reference material, or executable code), self-verifies it via execution feedback, stores it indexed by embedding, and later discovers, loads on demand via progressive disclosure, and composes skills into complex workflows.
- **Why agentic:** Open-ended continual learning: the agent grows its OWN action repertoire, never forgetting, compounding competence and self-selecting which to apply. Skills became a cross-platform open standard (Claude, Cursor, Codex). Voyager outperformed AutoGPT precisely because of its skill library + self-verification + curriculum.
- **Side-effect class:** `write-reversible` · **Default autonomy:** `act-with-veto` · **Authoring path:** manifest-template (signed JSON)
- **Krelvan fit:** The bridge to Krelvan's signed-JSON-manifest and YAML authoring paths — shareable, capability-bearing packages an agent self-selects. An agent authoring and registering a new capability the platform can recall later is self-extension. Admission stays deny-by-default; a newly authored skill that performs side-effects parks for approval (act-with-veto) on first use.
- **Sources:** Anthropic Agent Skills; SKILL.md, skills.sh; Voyager; Skill Creator, MUSE-Autoskill.

### Dynamic Tooling & Extensibility

#### `tools.discover`
- **What it does:** At runtime the agent queries available tools/servers (e.g. via MCP `tools/list` or a searchable tool library), retrieves only the tool definitions relevant to the current subtask from a large pool, loads them via progressive disclosure, and decides how to invoke them — with servers notifying it when capabilities change.
- **Why agentic:** The agent's action space is not fixed at build time; it expands by discovery and the model reasons over tool descriptions to choose among thousands of tools (Anthropic: 72K→8.7K context, 95% window preserved; 98.7% reduction via code execution). Runtime tool acquisition is a planning act — the antithesis of hardcoded connectors.
- **Side-effect class:** `read` · **Default autonomy:** `full` · **Authoring path:** core
- **Krelvan fit:** Makes Krelvan's capability-plugin registry queryable so an agent finds and admits capabilities mid-task. Discovery is gated by deny-by-default admission (the agent can discover but admission requires policy); the secret broker means a discovered tool is invoked without the agent ever seeing raw credentials. Core to a self-hostable, extensible platform.
- **Sources:** Anthropic Tool Search Tool; a16z tool discovery; MCP; Toolformer, ToolLLM, AnyTool; ScaleMCP, Dynamic ReAct.

#### `tools.admit_mcp_server`
- **What it does:** Admits an MCP server as a first-class capability surface — a bundle of tools (executable functions), resources (data), and prompts (workflows) — with automatic discovery and error handling, treating each as a `CapabilityPlugin`.
- **Why agentic:** MCP is the open standard ("USB-C for AI") giving a reasoning agent an extensible, discoverable action space across databases, browsers, design tools, even 3D printers. The value is the agent choosing among a discovered toolset, not a single hardcoded integration.
- **Side-effect class:** `write-reversible` · **Default autonomy:** `act-with-veto` · **Authoring path:** core
- **Krelvan fit:** Krelvan admits an MCP server as a bundle of `CapabilityPlugin`s, each individually classified into one of the six side-effect classes under the same trust model. MCP tool-poisoning (MCPTox) is exactly what deny-by-default admission, supervisor co-sign, and the secret broker defend against. This makes the entire MCP ecosystem available without bypassing Krelvan's gates.
- **Sources:** Anthropic MCP; Code execution with MCP; MCPTox.

#### `http.wrap_api`
- **What it does:** Wraps an arbitrary HTTP API as a capability the agent can reason about and invoke, declared purely in YAML (endpoint, params, side-effect class) with no custom code, so a self-hoster exposes a private/internal service to agents in minutes.
- **Why agentic:** It only becomes agentic in combination with `tools.discover` and `plan.decompose` — the agent selects and composes the wrapped API at runtime based on intent, rather than a human pre-wiring a when-X-do-Y rule. The wrapper is plumbing; the reasoning over it is the point. Included strictly as the no-code authoring on-ramp to genuinely agentic composition, **not** as a headline trigger-action connector.
- **Side-effect class:** `write-reversible` · **Default autonomy:** `act-with-veto` · **Authoring path:** YAML
- **Krelvan fit:** The YAML authoring path made concrete — the lowest-friction way for self-hosters to extend Krelvan. The author declares the side-effect class in YAML, which auto-drives the approval gate and budget. Secret broker injects credentials so the YAML never contains raw secrets. The on-ramp that makes the platform broadly extensible.
- **Sources:** Anthropic custom tools under approval; a16z tool discovery.

### Multi-Agent Collaboration

#### `collab.debate`
- **What it does:** Multiple agents or roles mutually critique and refine each other's outputs through structured debate/deliberation, often with tool use to verify facts and resolve disagreement, converging on a better answer — with role specialization driven by SOPs.
- **Why agentic:** Adversarial collaboration among reasoning agents surfaces errors a single agent misses — an emergent quality mechanism, not a linear handoff. Role specialization (MetaGPT PM/engineer) injects domain expertise; debate-to-deliberation improves collective reasoning.
- **Side-effect class:** `read` · **Default autonomy:** `full` · **Authoring path:** core
- **Krelvan fit:** Read-class deliberation among Krelvan nodes that improves answer quality before any action settles — a multi-agent extension of the supervisor co-sign. Each debating agent is a separately-budgeted node; the consensus is logged to the ledger as the assurance trail.
- **Sources:** AutoGen, MetaGPT, From Debate to Deliberation; CrewAI inter-agent discussion.

#### `collab.negotiate_counterparty`
- **What it does:** Agents representing different stakeholders negotiate with each other (or a human's agent) within authorized parameters — proposing terms, evaluating counter-offers against constraints/objectives, converging on agreement (commercial/supply-chain/legal), and flagging asymmetries or out-of-bounds terms for human review.
- **Why agentic:** Goal-directed strategic interaction requiring planning, theory-of-mind, and concession reasoning over many turns. a16z frames this as the leap beyond solo retrieval to counterparty AIs that negotiate and flag asymmetries — where AI finally gets network effects. No pre-agentic analogue.
- **Side-effect class:** `message-human` · **Default autonomy:** `act-with-veto` · **Authoring path:** TypeScript
- **Krelvan fit:** Negotiation within parameters is the autonomy envelope; any negotiated term that exceeds it parks for human review automatically (message-human + act-with-veto). When a deal is actually executed it escalates to spend/write-irreversible with full assurance. The signed audit trail of every offer/counter-offer is the verifiable record a counterparty's KYA check needs.
- **Sources:** a16z negotiate counterparty; Pactum-style, NEGOTIATIONGYM; Workday, Sirion AI Redline, CoCounsel.

### Human-in-the-Loop

#### `hitl.escalate`
- **What it does:** Treats human approval as a first-class node: the agent reasons about when it lacks authority/information or is about to take a high-side-effect action, pauses, surfaces the proposed action with evidence/context, asks multiple-choice clarifying questions if needed, and resumes exactly where it left off on approve/edit/reject — with escalation conditional on the action's arguments.
- **Why agentic:** The agent itself decides when to interrupt and hand control to a human — escalation is a planned action in the loop, not an exception handler. This is the runtime governance layer of an acting agent, enabling calibrated high-stakes autonomy. Every production agentic SOC/RCM deployment runs HITL for consequential decisions.
- **Side-effect class:** `message-human` · **Default autonomy:** `act-with-veto` · **Authoring path:** core
- **Krelvan fit:** Krelvan's autonomy model made concrete: `act-with-veto` and `suggest` modes, plus the automatic parking of spend / write-irreversible / identity-mutation calls for approval. Predicate gating on tool args maps to per-node side-effect-driven gates. Durable checkpointing lets the parked run resume idempotently after the human decides. The load-bearing safety primitive.
- **Sources:** Anthropic AskUserQuestion; LangGraph interrupt, Operator; LangGraph HITL.

### Runtime & Durability

#### `runtime.durable_replay`
- **What it does:** Executes long-running, multi-day tasks durably: checkpoints full agent state at every super-step, survives crashes/restarts, and resumes from exactly where it left off without redoing completed work or re-firing side-effects.
- **Why agentic:** Long-horizon autonomy over open-ended tasks (the next major evolution per Menlo) requires durable statefulness — an agent runtime property a trigger-action tool neither needs nor has. It is what makes multi-hour browser/coding/research runs viable.
- **Side-effect class:** `read` · **Default autonomy:** `full` · **Authoring path:** core
- **Krelvan fit:** Directly implements Krelvan's crash-safe idempotent replay. Combined with the reserved-then-settled budget, a crash mid-action does not double-spend (reservation persists; settlement is idempotent). The checkpoint stream is the ledger. Core infrastructure under every long-horizon capability.
- **Sources:** LangGraph checkpointers, Menlo long-horizon agents; idempotent replay literature.

### Real-World Effects (the effect ladder)

#### `act.execute_effect`
- **What it does:** Takes consequential actions in real services — calendars, docs, repos, payments, deployments, systems of record — turning a passive database into an active workflow engine. Reads, writes, and reasons across operational data end-to-end, with the effect verified, metered, and high-stakes classes parked for approval.
- **Why agentic:** a16z's whole thesis: "from chat to action" — AI stops being something you ask and becomes something that does. The agent decides which action to take from intent and operational context, not a pre-wired when-X-do-Y rule. The agent owns the workflow; the SaaS UI is incidental. Bessemer's "systems of action" put whole processes on autopilot.
- **Side-effect class:** `write-reversible` (irreversible variants escalate) · **Default autonomy:** `act-with-veto` · **Authoring path:** YAML
- **Krelvan fit:** The family of plugins that mutate real downstream state. Reversible mutations (update a deal, edit a doc) are write-reversible/act-with-veto; irreversible ones (issue an invoice, submit a filing) escalate to write-irreversible and park. Reserved-then-settled integer-cent budget + ledger audit = "autonomy with accountability," precisely the supervised-effect regime Krelvan's classes implement.
- **Sources:** a16z act-on-software; Anthropic real-world effects under approval; Bessemer systems of action, Menlo; SOC, SRE, finance close, prior-auth RCM, support verticals.

#### `act.spend`
- **What it does:** Agent-initiated commerce: the agent pays for services, allocates capital, executes financial operations (procurement, refunds, payments), and transacts agent-to-agent (e.g. paying "headless merchants" per call) within authorization limits defined at the policy level.
- **Why agentic:** Autonomous spend under a budget is the irreducible agentic act. a16z documents real agent-to-agent rails (Stripe/Tempo 34k transactions week one; headless merchants priced per call) and argues agents need wallets to act agentically. Support agents that decide a refund is owed and move the money apply policy judgment per case — not a webhook.
- **Side-effect class:** `spend` · **Default autonomy:** `act-with-veto` · **Authoring path:** YAML
- **Krelvan fit:** The canonical `spend` side-effect class: reserved-then-settled integer-cent budgets (no floats), automatic parking above thresholds, counterfactual dry-run before commit. Per-call pricing of headless merchants maps perfectly to `estimateCents → reserve → invoke → settle claimedCostCents`. The ledger is the spend audit trail.
- **Sources:** a16z spend autonomously; Anthropic real-world effects under approval; support + finance verticals.

#### `act.provision_irreversible`
- **What it does:** Acts in the physical/operational world with irreversible effect: provisioning/deploying infrastructure, executing a remediation runbook, submitting a regulatory filing, driving an edge/DePIN node, or merging/shipping code — chosen by a reasoning agent under the strongest gates.
- **Why agentic:** Root-cause reasoning over noisy production signals and selecting/executing the right irreversible remediation is inference under uncertainty (one deployment handled 25 incidents autonomously; MTTR 45min→5min). a16z cites autonomous edge systems and DePIN nodes "with minimal human intervention." Irreversible real-world effects chosen by reasoning — the antithesis of a scripted connector.
- **Side-effect class:** `write-irreversible` · **Default autonomy:** `act-with-veto` · **Authoring path:** TypeScript
- **Krelvan fit:** The write-irreversible class requiring the strongest approval gates, mandatory supervisor co-sign + counterfactual safety check, and crash-safe idempotent replay (a half-applied provision must not double-apply on resume). High-stakes verticals (SRE, healthops prior-auth) prove the gate is load-bearing, not optional.
- **Sources:** a16z physical provision; verified-effect literature; SRE incident remediation, prior-auth RCM.

#### `identity.act_as_principal`
- **What it does:** The agent carries cryptographically signed credentials proving who it represents (its principal), what it is permitted to do, and its reputation — "Know Your Agent" — so a counterparty can trust and bound it; mutating or asserting that identity is the highest-assurance act.
- **Why agentic:** a16z declares the agent-economy bottleneck is "identity, not intelligence" and pushes KYA: signed credentials linking agents to principals, permissions, reputation. This exists only because agents act on others' behalf autonomously — the highest-assurance, identity-mutating class.
- **Side-effect class:** `identity-mutation` · **Default autonomy:** `suggest` · **Authoring path:** core
- **Krelvan fit:** The identity-mutation side-effect class — the rarest and most-gated. Defaults to `suggest` (always surface to the human; never silently change who the agent is or what it is authorized as). Krelvan's signed audit trail and ledger ARE the verifiable execution record KYA requires; a counterparty verifying a Krelvan agent reads exactly this. Uniquely enabled by the ledger-first model.
- **Sources:** a16z identity/KYA (KYA, MetaMask/AgentKit).

### 4.1 `verify.cosign` — the binding-constraint capability *(read)*

**Showcase rationale.** The canonical instance of Krelvan's whole thesis in ~120 lines. It turns the abstract "supervisor co-signs results" primitive into a concrete capability you actually call; it is dependency-light (only the shipped `getLLMClient`/`estimateCostCents` adapters and real ledger helpers); it is uniquely ledger-enabled (the assurance record is content-addressed over the exact subject bytes and emitted via an `EffectResult` the *supervisor* signs, so it is tamper-evident, replayable, unforgeable); and it is the perfect template shape — `sideEffect: "read"` producing evidence that *gates* expensive/irreversible nodes. It teaches the pattern every serious Krelvan agent copies: **the gate must be safer and cheaper than the act it gates; verification is a first-class, ledger-anchored artifact.**

```typescript
// capabilities/verify-cosign.ts
//
// verify.cosign — the binding-constraint capability.
// a16z 2026: "intelligence is cheap, verification is the binding constraint."
// Krelvan's answer: make supervisor co-sign a REAL, reusable capability that
// critiques a draft claim-by-claim, demands a citation per claim, revises once,
// and emits a per-claim, ledger-anchored ASSURANCE RECORD. High-side-effect
// nodes refuse to fire until an assurance record over their exact payload exists.
//
// sideEffect = "read": it touches no external world. It only produces evidence.
// That is the point — the gate must be cheaper and safer than the act it gates.

import type { CapabilityPlugin, EffectCall } from "../src/core/capability/capability.js";
import { getLLMClient, estimateCostCents } from "../src/adapters/llm-client.js";
import { canonicalize } from "../src/core/ledger/canonical.js";
import { contentAddress } from "../src/core/ledger/crypto.js";
import { getLogger } from "../src/core/observability/logger.js";

const log = getLogger("verify-cosign");

// ── what one verified claim looks like ────────────────────────────────────────
interface ClaimVerdict {
  claim: string;            // the atomic assertion the draft makes
  verdict: "supported" | "unsupported" | "contradicted";
  citation: string | null;  // source the claim is anchored to (URL/doc id/quote)
  note: string;             // one-line critique
}
interface AssuranceRecord {
  subject: string;          // content-hash of the EXACT payload being gated
  verdicts: ClaimVerdict[];
  revised: boolean;         // did critique force a rewrite?
  revisedDraft: string | null;
  unsupportedCount: number;
  pass: boolean;            // true only if every claim is supported AND cited
  assuranceId: string;      // content-hash of this whole record → ledger anchor
}

function provider() {
  return (process.env["KRELVAN_LLM_PROVIDER"] ?? "anthropic") as
    "anthropic" | "openai" | "ollama";
}
function model() {
  return process.env["KRELVAN_VERIFY_MODEL"]
    ?? process.env["KRELVAN_LLM_MODEL"]
    ?? "claude-haiku-4-5-20251001";
}

// strict JSON-out helper (same defensive parse pattern as llm_route)
async function ask(system: string, user: string) {
  const client = getLLMClient();
  const r = await client.complete({
    system, messages: [{ role: "user", content: user }],
    model: model(), maxTokens: 900, temperature: 0,
  });
  let t = r.text;
  const fence = t.match(/```(?:json)?\s*([\s\S]*?)```/);
  if (fence?.[1]) t = fence[1].trim();
  if (!t.startsWith("{") && !t.startsWith("[")) {
    const s = Math.min(...["{", "["].map(c => t.indexOf(c)).filter(i => i >= 0));
    const e = Math.max(t.lastIndexOf("}"), t.lastIndexOf("]"));
    if (Number.isFinite(s) && e > s) t = t.slice(s, e + 1);
  }
  return { text: t, inputTokens: r.inputTokens, outputTokens: r.outputTokens };
}

export const verifyCosignCapability: CapabilityPlugin = {
  name: "verify.cosign",
  sideEffect: "read", // produces evidence only; never mutates the world

  // two LLM passes (critique + conditional revise) — budget for the worst case.
  estimateCents: () => 30,

  async invoke(call: EffectCall) {
    const input = call.input as Record<string, unknown>;
    // the draft to be gated and (optionally) sources the agent already gathered.
    const draft = String(input["draft"] ?? input[`${call.nodeId}.draft`] ?? "").trim();
    const sources = String(input["sources"] ?? input[`${call.nodeId}.sources`] ?? "").trim();
    if (!draft) throw new Error("verify.cosign: 'draft' input is required");

    // subject = the exact bytes we are vouching for. The downstream gate compares
    // its own payload hash to THIS; you cannot pass a verified hash and then ship
    // different bytes.
    const subject = contentAddress(canonicalize({ draft }));

    // ── PASS 1: adversarial critique, one verdict + one citation per claim ──────
    const critiqueSys = [
      "You are a verification supervisor. Split the DRAFT into atomic claims.",
      "For EACH claim decide: supported | unsupported | contradicted, using ONLY",
      "the SOURCES provided. A claim with no matching source is 'unsupported'.",
      "Every 'supported' claim MUST carry a citation (a source id or exact quote).",
      'Reply ONLY: {"verdicts":[{"claim","verdict","citation","note"}]}',
    ].join("\n");
    const c = await ask(critiqueSys,
      `SOURCES:\n${sources || "(none provided)"}\n\nDRAFT:\n${draft}`);
    let verdicts: ClaimVerdict[] = [];
    try { verdicts = (JSON.parse(c.text).verdicts ?? []) as ClaimVerdict[]; }
    catch { verdicts = [{ claim: draft, verdict: "unsupported",
      citation: null, note: "critique unparsable — fail closed" }]; }

    const unsupported = verdicts.filter(v => v.verdict !== "supported");

    // ── PASS 2: revise ONCE, only if critique found problems ───────────────────
    let revisedDraft: string | null = null;
    let revInTok = 0, revOutTok = 0;
    if (unsupported.length > 0) {
      const reviseSys = [
        "Rewrite the DRAFT so every remaining claim is supported by the SOURCES.",
        "DROP claims that cannot be cited. Do not invent sources.",
        'Reply ONLY: {"revisedDraft":"..."}',
      ].join("\n");
      const r = await ask(reviseSys,
        `SOURCES:\n${sources || "(none)"}\n\nDRAFT:\n${draft}\n\n` +
        `PROBLEMS:\n${unsupported.map(v => `- ${v.claim}: ${v.note}`).join("\n")}`);
      revInTok = r.inputTokens; revOutTok = r.outputTokens;
      try { revisedDraft = String(JSON.parse(r.text).revisedDraft ?? "").trim() || null; }
      catch { revisedDraft = null; }
    }

    // pass ONLY if nothing is unsupported/contradicted AND every claim is cited.
    const pass = verdicts.length > 0
      && unsupported.length === 0
      && verdicts.every(v => !!v.citation);

    const record: AssuranceRecord = {
      subject, verdicts, revised: revisedDraft !== null,
      revisedDraft, unsupportedCount: unsupported.length, pass,
      assuranceId: "", // filled below from the canonical record
    };
    record.assuranceId = contentAddress(canonicalize({
      subject: record.subject, verdicts: record.verdicts,
      revised: record.revised, pass: record.pass,
    }));

    // cost: real token cost of both passes, integer cents (ledger invariant).
    const cost = estimateCostCents(provider(), model(),
      c.inputTokens + revInTok, c.outputTokens + revOutTok);

    log.info({ nodeId: call.nodeId, subject, pass,
      unsupported: unsupported.length }, "verify.cosign: assurance produced");

    // NOTE: this plugin returns the record as data. The SUPERVISOR co-signs the
    // EffectResult that carries it (capability.ts: the plugin never self-signs),
    // so the assurance record lands in the ledger under the supervisor's key,
    // tamper-evident and replayable — the part no plugin can forge.
    return {
      output: {
        assurance: record,
        // spread for declarative edge gating, e.g.
        //   when: { op:"eq", left:{op:"var",key:"verify.assurance_pass"},
        //           right:{op:"const",value:true} }
        assurance_pass: record.pass,
        assurance_id: record.assuranceId,
        verified_subject: record.subject,
      },
      claimedCostCents: Math.max(cost, 1),
    };
  },
};

/* ── HOW IT GATES A HIGH-SIDE-EFFECT ACTION (manifest sketch) ──────────────────
 nodes:
   verify  (autonomy: full)        caps: [verify.cosign     read]
   pay     (autonomy: act-with-veto) caps: [stripe.charge   spend]   // HIGH
 edges:
   - { from: verify, to: pay,
       when: { op:"eq", left:{op:"var",key:"verify.assurance_pass"},
                        right:{op:"const",value:true} } }
   - { from: verify, to: halt,
       when: { op:"eq", left:{op:"var",key:"verify.assurance_pass"},
                        right:{op:"const",value:false} } }
 The spend literally cannot be reached unless a supervisor-co-signed assurance
 record over its exact subject exists in the ledger. Verification is the gate. */
```

### 4.2 `hitl.escalate` — the load-bearing safety primitive *(message-human)*

**Showcase rationale.** The one capability every other consequential capability sits behind, so the highest-leverage thing to copy. It is genuinely agentic (an agent reasons, then deliberately hands a decision to a human and the run parks durably) and faithful to Krelvan's real contract: the four `CapabilityPlugin` members are exact, `sideEffect` is one of the six classes, costs are integer cents with `claimedCostCents` marked untrusted, and it leans on the *actual* autonomy gate (`needsApproval` in `capability.ts`) and the *actual* parking event (`AwaitRequested` in `engine.ts`) rather than reinventing them. The predicate-gating-on-args + assurance ladder (notify / block-until-ack / two-person) is the pattern people extend for SOC, RCM, and coding agents.

```typescript
// capabilities/hitl-escalate.ts
//
// hitl.escalate — the load-bearing safety primitive.
// An agent calls this to hand a consequential decision to a human and PARK
// the run durably until a person responds. It is declared "message-human",
// so the SIX-CLASS autonomy gate forces it through approval automatically
// under "suggest" and "act-with-veto" — you write zero gating code here.
//
// What this plugin actually does (honest, dependency-light):
//   1. Validates + normalizes the decision packet the agent proposes.
//   2. Runs a PREDICATE over the args to decide the assurance level required
//      ("notify" vs "block-until-ack" vs "two-person") — this is the
//      side-effect-class-driven automatic parking policy, made explicit.
//   3. Returns a structured, audit-ready decision record as `output`.
//
// What it does NOT do: it does not itself deliver the message or wait. The
// kernel does the parking (AwaitRequested) and the channel adapter does the
// delivery. >>> EXTERNAL CALL is clearly marked and behind a port; in this
// reference build it is a no-op stub, NOT a fake "real" Slack/email API. <<<

import type { CapabilityPlugin, EffectCall } from "../src/core/capability/capability.js";

// --- The decision packet an agent proposes when it escalates ---------------
type Assurance = "notify" | "block-until-ack" | "two-person";

interface EscalateInput {
  /** what the agent wants to do, in one line a human can judge fast. */
  summary: string;
  /** the side-effect class of the action being gated (drives the predicate). */
  proposedEffect:
    | "read" | "write-reversible" | "write-irreversible"
    | "spend" | "message-human" | "identity-mutation";
  /** structured args of the gated action, so the predicate can reason on them. */
  args: Record<string, unknown>;
  /** optional explicit override the agent can request; predicate may raise it. */
  requested?: Assurance;
}

/**
 * PREDICATE GATING ON ARGS — the heart of "automatic parking driven by
 * side-effect class". The required assurance is the MAX of the class floor
 * and any arg-derived escalation. You can only ever ratchet UP, never down.
 */
function requiredAssurance(input: EscalateInput): Assurance {
  const order: Assurance[] = ["notify", "block-until-ack", "two-person"];
  const max = (a: Assurance, b: Assurance) =>
    order[Math.max(order.indexOf(a), order.indexOf(b))];

  // 1) floor from the side-effect class of the action being gated
  let level: Assurance =
    input.proposedEffect === "spend" ||
    input.proposedEffect === "write-irreversible" ||
    input.proposedEffect === "identity-mutation"
      ? "block-until-ack"
      : "notify";

  // 2) predicate over args — concrete, copyable examples
  const amount = Number(input.args.amountCents ?? 0);
  if (input.proposedEffect === "spend" && amount >= 50_00) level = max(level, "two-person");
  if (input.args.target === "production") level = max(level, "two-person");
  if (input.proposedEffect === "identity-mutation") level = max(level, "two-person");

  // 3) agent may REQUEST more assurance, never less
  if (input.requested) level = max(level, input.requested);
  return level;
}

export const HitlEscalatePlugin: CapabilityPlugin = {
  name: "hitl.escalate",
  // message-human => the autonomy gate parks this for a human under
  // "suggest" and "act-with-veto". Declaring it honestly is the whole point.
  sideEffect: "message-human",

  // Escalation is compute + one outbound message; integer cents, no floats.
  estimateCents(_call: EffectCall): number {
    return 0;
  },

  async invoke(call: EffectCall): Promise<{ output: unknown; claimedCostCents: number }> {
    const input = call.input as EscalateInput;
    if (typeof input?.summary !== "string" || input.summary.length === 0) {
      throw new Error("hitl.escalate: 'summary' must be a non-empty string");
    }
    if (typeof input.proposedEffect !== "string") {
      throw new Error("hitl.escalate: 'proposedEffect' (side-effect class) is required");
    }

    const assurance = requiredAssurance(input);

    // >>> EXTERNAL CALL BOUNDARY (clearly marked, not a fake real API) <<<
    // In production a ChannelAdapter delivers this; the secret broker mints a
    // scoped, short-lived token so this plugin never sees a raw credential.
    // Reference build: no-op. We return the packet; the kernel parks the run
    // (AwaitRequested) and resumes durably on AwaitResolved after a human acts.
    //   const token = broker.mint(destination, allowed, now(), 60);  // scoped
    //   await channel.deliver(token, decisionRecord);                // external

    const decisionRecord = {
      kind: "human-decision-required",
      summary: input.summary,
      proposedEffect: input.proposedEffect,
      assurance,                 // notify | block-until-ack | two-person
      args: input.args ?? {},
      // correlationId is the kernel's idempotency key for this call; the
      // engine emits AwaitRequested with it and matches AwaitResolved on resume.
      raisedBy: call.nodeId,
    };

    // claimedCostCents is explicitly UNTRUSTED — the supervisor co-signs the
    // observed cost. We claim 0 (no spend, no metered egress in this build).
    return { output: decisionRecord, claimedCostCents: 0 };
  },
};

export default HitlEscalatePlugin;

/* ──────────────────────────────────────────────────────────────────────────
 HOW THE SIX-CLASS AUTONOMY GATE PRODUCES PARKING (already in core — quoted so
 readers see this plugin needs no gating code of its own):

   // src/core/capability/capability.ts  (verbatim)
   export function needsApproval(autonomy, effect) {
```

### 4.3 `runtime.durable_job` — long-horizon, crash-safe, reserved-then-settled job runner *(spend)*

**Showcase rationale.** The canonical template for any long-running `spend`-class capability (browser automation, coding agents, deep research, render/transcode farms). It is the smallest honest example that *uses* — rather than re-implements — Krelvan's two hardest-won guarantees: crash-safe idempotent replay and reserved-then-settled integer-cent budgeting. It teaches three reusable patterns: `estimateCents` returns the worst-case reservation and rejects floats inline; `invoke()` re-serves finished sub-steps from durable checkpoints instead of re-running them; `claimedCostCents` reports the real observed cost (always ≤ reservation) and is treated as untrusted data the supervisor co-signs.

```typescript
/**
 * runtime.durable_job — a long-horizon, crash-safe, reserved-then-settled job runner.
 *
 * WHY THIS IS THE FOUNDATION CAPABILITY:
 *   It does not add a new external integration. It SHOWCASES the two pieces of
 *   Krelvan platform infrastructure that every other "spend"-class capability sits
 *   on: (1) crash-safe idempotent replay, and (2) reserved-then-settled integer-cent
 *   budgeting. A multi-hour browser/coding/research job is modelled as a list of
 *   priced sub-steps. Each completed sub-step is appended as a durable checkpoint.
 *   On resume after a crash, finished sub-steps are RE-SERVED from the checkpoint
 *   (never re-executed, never re-charged); only unfinished work runs. The plugin
 *   reserves the worst-case cost up front and SETTLES the real observed cost — the
 *   supervisor, not this plugin, signs that cost into the ledger.
 *
 * HONESTY NOTE: every line that would touch a paid external system (a browser farm,
 * a coding sandbox, an LLM) is marked `// EXTERNAL (illustrative stub)`. Nothing here
 * pretends a fake API is real. Swap the stub for a real client + broker.mint() token.
 */
import type { CapabilityPlugin, EffectCall } from "../src/core/capability/capability.js";

// ---- the durable substrate this capability is built to exercise -----------------
// Krelvan gives idempotent replay for the WHOLE call for free (keyed on
// nodeId+capability). For a multi-HOUR job we want SUB-STEP durability too, so we
// append our own fine-grained checkpoints under the call's idempotency key. The
// kernel's crash-hole detection still protects the outer call; this just makes a
// resumed call cheap instead of restarting from step 0.
interface CheckpointStore {
  // append-only; returns all sub-steps already durably committed for this run.
  loadCompleted(idemKey: string): Promise<Array<{ stepId: string; settledCents: number }>>;
  commit(idemKey: string, step: { stepId: string; settledCents: number }): Promise<void>;
}

interface JobStep { stepId: string; worstCaseCents: number; }      // integer cents only
interface JobInput { idemKey: string; steps: JobStep[]; }

export function makeDurableJobPlugin(store: CheckpointStore): CapabilityPlugin {
  return {
    name: "runtime.durable_job",
    sideEffect: "spend",                  // forces approval gate unless autonomy:"full"

    // RESERVE: worst-case sum of all steps. Integer cents, no floats. The kernel
    // reserves exactly this before invoke() runs; an over-ceiling job is denied
    // pre-dispatch, so this plugin can never cause overspend.
    estimateCents(call: EffectCall): number {
      const { steps } = call.input as JobInput;
      return steps.reduce((sum, s) => {
        if (!Number.isInteger(s.worstCaseCents) || s.worstCaseCents < 0) {
          throw new Error(`durable_job: worstCaseCents must be a non-negative integer (got ${s.worstCaseCents})`);
        }
        return sum + s.worstCaseCents;
      }, 0);
    },

    async invoke(call: EffectCall): Promise<{ output: unknown; claimedCostCents: number }> {
      const { idemKey, steps } = call.input as JobInput;

      // RESUME: re-serve finished steps from the durable log. They are NOT re-run
      // and NOT re-charged — this is what lets a crashed 3-hour job continue.
      const done = await store.loadCompleted(idemKey);
      const doneIds = new Set(done.map(d => d.stepId));
      let settledCents = done.reduce((s, d) => s + d.settledCents, 0);
      const resumed: string[] = [...doneIds];
      const executed: string[] = [];

      for (const step of steps) {
        if (doneIds.has(step.stepId)) continue;            // already durable → skip

        // EXTERNAL (illustrative stub): real work goes here — drive a browser farm,
        // run a coding sandbox turn, call an LLM. Use broker.mint(destination,...)
        // for a scoped token; the plugin never sees the raw secret. The stub charges
        // the worst case so the example settles deterministically without a network.
        const observedCents = step.worstCaseCents;         // replace with real metered cost

        // COMMIT the checkpoint BEFORE moving on. If the process dies on the next
        // step, this one is already durable and will be re-served, not re-charged.
        await store.commit(idemKey, { stepId: step.stepId, settledCents: observedCents });
        settledCents += observedCents;
        executed.push(step.stepId);
      }

      // SETTLE: report the REAL total spent. claimedCostCents is explicitly untrusted
      // input to the supervisor, which co-signs the EffectResult; any drift from the
      // reservation is on the record. Always <= the reserved worst case.
      return {
        output: { completedSteps: steps.length, resumedSteps: resumed, executedSteps: executed },
        claimedCostCents: settledCents,
      };
    },
  };
}

// Register exactly like text-transform.ts:
//   const plugins = new Map([["runtime.durable_job", makeDurableJobPlugin(store)]]);
//   const supervisor = new Supervisor(plugins);
// Grant in the node manifest (note sideEffect + a real ceiling):
//   { "name": "runtime.durable_job", "sideEffect": "spend", "budgetCents": 500 }
```

### 4.4 `tools.admit` — the admission + discovery + http-wrap bundle *(read / identity-mutation / write-irreversible)*

**Showcase rationale.** The one capability that makes Krelvan a platform instead of a fixed tool set, composing primitives Krelvan already ships rather than adding new trust primitives. It is the canonical *meta-capability* — a plugin whose output is more plugins — and it demonstrates the single most important authoring decision honestly: choosing the side-effect class (discovery is `read`, admitting a server is `identity-mutation`, minting a wrapped API is `write-irreversible`). It turns the abstract "deny-by-default admission + secret broker" claims into runnable code against the real source, and it contains tool-poisoning (MCPTox) structurally — a poisoned server cannot self-admit, cannot widen its declared side-effect class past a human veto, and never touches a raw secret.

```typescript
// capabilities/tools-admit.ts
// A SHOWCASE bundle: discover third-party tools, admit them under deny-by-default,
// and wrap any HTTP API as a YAML capability — all behind the secret broker.
//
// Grounded in real Krelvan source:
//   - CapabilityPlugin / EffectCall          src/core/capability/capability.ts
//   - McpRegistry (discover + connect)       src/core/mcp/mcp-client.ts
//   - compileYamlCapability / validate       src/core/extensions/yaml-capability.ts
//   - SecretBrokerPort {validate, resolve}   src/core/plugins/ports.ts
//   - PluginLifecycleService (snapshot swap) src/core/plugins/lifecycle-service.ts
//
// HONESTY MARKERS:
//   * EXTERNAL CALL — McpRegistry.connect() spawns/contacts a third-party MCP
//     server. Network/process boundary. Marked below.
//   * NO FAKE APIs — http.wrap_api compiles a descriptor; it does not pretend any
//     specific vendor endpoint exists.

import type { CapabilityPlugin, EffectCall } from "../core/capability/capability.js";
import { McpRegistry, type McpServerConfig } from "../core/mcp/mcp-client.js";
import {
  validateYamlCapability,
  compileYamlCapability,
} from "../core/extensions/yaml-capability.js";
import type { SecretBrokerPort } from "../core/plugins/ports.js";
import type { PluginLifecycleService } from "../core/plugins/lifecycle-service.js";

// One shared registry + an explicit allowlist = DENY-BY-DEFAULT admission.
// A discovered tool is INERT until an operator/owner admits it by name.
export interface AdmissionDeps {
  mcp: McpRegistry;
  broker: SecretBrokerPort;
  lifecycle: PluginLifecycleService; // owns the live Supervisor snapshot swap
  // names an owner has explicitly cleared for admission (empty = nothing admitted)
  admittedAllowlist: Set<string>;
}

export function makeToolAdmissionBundle(deps: AdmissionDeps): CapabilityPlugin[] {
  // ── 1) tools.discover — pure-ish READ: list what a server OFFERS, admit nothing
  const discover: CapabilityPlugin = {
    name: "tools.discover",
    sideEffect: "read", // discovery changes no state; it only enumerates
    estimateCents: () => 0,
    async invoke(call: EffectCall) {
      const cfg = call.input as McpServerConfig;
      // EXTERNAL CALL — contacts the third-party MCP server to list its tools.
      const res = await deps.mcp.connect(cfg);
      if (!res.ok) return { output: { ok: false, error: res.error }, claimedCostCents: 0 };
      // Return the proposed tools WITH their inferred side-effect class so a human
      // can see "this server wants 'send_email' = message-human" before admitting.
      const proposed = deps.mcp.allPlugins()
        .filter((p) => p.name.startsWith(cfg.name.toLowerCase().replace(/[^a-z0-9]/g, "_") + "."))
        .map((p) => ({ name: p.name, sideEffect: p.sideEffect }));
      return { output: { ok: true, proposed }, claimedCostCents: 0 };
    },
  };

  // ── 2) tools.admit_mcp_server — IDENTITY-MUTATION: widens what agents can reach
  const admit: CapabilityPlugin = {
    name: "tools.admit_mcp_server",
    // Admitting a tool server changes the agent's REACH into the world. That is an
    // identity-level change, so it parks for human approval under any autonomy but
    // "full" — the strongest gate the model offers. This is the MCPTox defense:
    // tool-poisoning cannot self-admit; a human signs off on the side-effect classes.
    sideEffect: "identity-mutation",
    estimateCents: () => 0,
    async invoke(call: EffectCall) {
      const { serverName, admit: toAdmit, sideEffectOverrides } =
        call.input as {
          serverName: string;
          admit: string[]; // explicit per-tool admission — NOTHING admitted implicitly
          sideEffectOverrides?: Record<string, string>;
        };

      // Re-discover so we admit against the server's CURRENT advertised surface,
      // not a stale snapshot (a poisoned server cannot swap tools post-approval).
      const plugins = deps.mcp.allPlugins().filter((p) =>
        toAdmit.includes(p.name),
      );
      if (plugins.length !== toAdmit.length) {
        const missing = toAdmit.filter((n) => !plugins.some((p) => p.name === n));
        return {
          output: { ok: false, error: `tools not offered by server: ${missing.join(", ")}` },
          claimedCostCents: 0,
        };
      }

      // DENY-BY-DEFAULT: add only the named tools to the allowlist, then swap them
      // into the live Supervisor snapshot. Anything not named stays inert.
      for (const p of plugins) deps.admittedAllowlist.add(p.name);
      await deps.lifecycle.adoptPlugins(plugins, deps.admittedAllowlist);

      return {
        output: { ok: true, admitted: plugins.map((p) => ({ name: p.name, sideEffect: p.sideEffect })) },
        claimedCostCents: 0,
      };
    },
  };

  // ── 3) http.wrap_api — WRITE-IRREVERSIBLE: mints a new YAML capability on disk/registry
  const wrap: CapabilityPlugin = {
    name: "http.wrap_api",
    // Producing a NEW callable capability is a durable change to the install's
    // tool surface → write-irreversible, gated for human review of the descriptor.
    sideEffect: "write-irreversible",
    estimateCents: () => 0,
    async invoke(call: EffectCall) {
      const { yamlText } = call.input as { yamlText: string };

      // Validate against the real EXT-01..EXT-04 guards (six side-effect classes,
      // integer cents, dot-path responseField, secret-ref extraction).
      const parsed = validateYamlCapability(yamlText);
      if (!parsed.ok) {
        return { output: { ok: false, errors: parsed.errors }, claimedCostCents: 0 };
      }

      // SECRET BROKER: every {{secret:name}} the wrapper references must already be
      // registered. The new capability never sees raw credentials — only scoped
      // tokens the broker resolves at invoke time. Refuse to mint a capability that
      // would dangle on unregistered secrets.
      const missing = parsed.secretRefs.filter((ref) => !deps.broker.validate([ref]));
      if (missing.length > 0) {
        return { output: { ok: false, error: `unregistered secrets: ${missing.join(", ")}` }, claimedCostCents: 0 };
      }

      const compiled = compileYamlCapability(parsed.spec, deps.broker);
      // Admit the freshly-wrapped capability the same way MCP tools are admitted:
      // explicit allowlist + live snapshot swap. No special path, no eval.
      deps.admittedAllowlist.add(compiled.name);
      await deps.lifecycle.adoptPlugins([compiled], deps.admittedAllowlist);

      return {
        output: { ok: true, capability: { name: compiled.name, sideEffect: compiled.sideEffect } },
        claimedCostCents: 0,
      };
    },
  };

  return [discover, admit, wrap];
}

// Register in genesis.config.ts and grant ONLY to a privileged "admin" node:
//   { "id": "tool-admin", "autonomy": "act-with-veto", "capabilities": [
//       { "name": "tools.discover",          "sideEffect": "read",               "budgetCents": 0 },
//       { "name": "tools.admit_mcp_server",  "sideEffect": "identity-mutation",  "budgetCents": 0 },
//       { "name": "http.wrap_api",           "sideEffect": "write-irreversible", "budgetCents": 0 } ] }
// Because admit/wrap are identity-mutation / write-irreversible, every admission
// PARKS for a human under act-with-veto — the deny-by-default gate is structural,
// not advisory. The thousands of MCP/HTTP tools you absorb still each pass through
// the same admission, supervisor co-sign, and audit trail afterward.
```

### 4.5 `orchestrate.delegate` — recursive sub-agent fan-out *(spend)*

**Showcase rationale.** The capability that showcases Krelvan's whole thesis in a single object: it takes the highest-leverage agentic pattern in the field (recursive sub-agent fan-out — Anthropic's +90.2% result, a16z's "coordination is the moat") and makes it safe-by-construction. Others copy it as the canonical template for "one agent that spawns many." It teaches the load-bearing idioms in context — declaring `sideEffect` at the honest *aggregate* class (`spend`, not `read`), returning the full budget from `estimateCents` so the kernel reserves before any child runs, integer-cents budget slicing with zero float drift, deterministic sub-run ids for crash-safe re-spawn, and per-child autonomy gating so one child can park for a human while siblings keep working.

```typescript
// capabilities/orchestrate-delegate.ts
//
// orchestrate.delegate — recursive sub-agent fan-out with per-child budget
// reservation and per-child autonomy gating.
//
// WHY THIS IS THE SHOWCASE:
//   Fan-out is the known-powerful pattern (Anthropic's multi-agent research
//   system reported +90.2% over single-agent on its eval; a16z calls
//   coordination-across-recursive-fan-out "the moat"). The pattern's danger is
//   ALSO its power: N children planning and acting in parallel = N ways to
//   overspend or take an unapproved irreversible action. Krelvan turns the
//   powerful pattern into a GOVERNED one, for free, via the ledger:
//     - the parent RESERVES the whole fan-out budget before any child starts
//       (reserve-then-settle, integer cents — no float, no runaway spend);
//     - each child draws from a deterministic slice of that reservation and
//       CANNOT exceed it (admission denies pre-dispatch inside the sub-run);
//     - each child re-gates through admission with ITS OWN autonomy level, so a
//       child that tries a "spend"/"write-irreversible" effect under
//       "act-with-veto" parks for a human while its siblings keep running;
//     - the supervisor co-signs the AGGREGATE result; the plugin's claimed cost
//       is recorded as explicitly-untrusted data next to the settled cost.
//
// HONESTY NOTE: this plugin makes NO external network calls itself. The only
// outward effects are whatever the *child* sub-agents are granted to do — and
// every one of those is independently admitted, budgeted, and audited. Children
// here are run via Krelvan's existing sub-agent executor; nothing is faked.

import type { CapabilityPlugin, EffectCall } from "../src/core/capability/capability.js";
import { deriveSubRunId } from "../src/core/kernel/sub-agent-executor.js";

/** What the calling node passes as call.input. */
interface DelegateInput {
  // The sub-agent template each child runs (a content-addressed manifest id the
  // owner has already admitted). Same id for homogeneous fan-out (map over data),
  // or one id per task for heterogeneous fan-out (specialist sub-agents).
  childManifestId: string;
  // One work item per child. Fan-out width = tasks.length.
  tasks: Array<{ id: string; state: Record<string, string | number | boolean | null> }>;
  // Total cents the parent is willing to commit to the WHOLE fan-out. The plugin
  // never exceeds this; estimateCents returns exactly this so the kernel reserves
  // it up front. Integer cents only (ledger invariant LED-02/EXT-04).
  fanoutBudgetCents: number;
  // Optional hard cap on concurrent children (default = run them all).
  maxConcurrent?: number;
}

interface ChildResult {
  taskId: string;
  subRunId: string;       // deterministic; survives crash-resume (no double spawn)
  status: "completed" | "failed" | "parked-for-approval";
  output?: Record<string, unknown>;
  costCents: number;      // settled, per child
  reason?: string;        // present when failed/parked
}

// Deterministic, integer-only budget split: floor-divide, hand the remainder to
// the first children. Sum of slices === fanoutBudgetCents exactly (no float drift).
function sliceBudget(totalCents: number, n: number): number[] {
  const base = Math.floor(totalCents / n);
  const rem = totalCents - base * n;
  return Array.from({ length: n }, (_, i) => base + (i < rem ? 1 : 0));
}

export const OrchestrateDelegatePlugin: CapabilityPlugin = {
  name: "orchestrate.delegate",

  // Declared at the AGGREGATE class. Children may do anything from "read" to
  // "spend"; the parent node's grant for orchestrate.delegate is "spend" so the
  // supervisor and approval gates treat the whole fan-out as money-moving. When
  // in doubt, declare higher — declaring "read" here would bypass the gates that
  // make fan-out safe.
  sideEffect: "spend",

  // Pre-flight estimate == the full fan-out budget. The kernel RESERVES this much
  // against the run ceiling and the node-cap ceiling BEFORE invoke() runs, so the
  // fan-out can never start if it wouldn't fit. This is the runaway-spend guard.
  estimateCents(call: EffectCall): number {
    const input = call.input as DelegateInput;
    if (!Number.isInteger(input.fanoutBudgetCents) || input.fanoutBudgetCents < 0) {
      throw new Error("orchestrate.delegate: fanoutBudgetCents must be a non-negative integer (cents)");
    }
    return input.fanoutBudgetCents;
  },

  async invoke(call: EffectCall): Promise<{ output: unknown; claimedCostCents: number }> {
    const input = call.input as DelegateInput;
    const tasks = input.tasks ?? [];
    if (tasks.length === 0) {
      return { output: { children: [], spentCents: 0 }, claimedCostCents: 0 };
    }

    const slices = sliceBudget(input.fanoutBudgetCents, tasks.length);
    const limit = Math.max(1, input.maxConcurrent ?? tasks.length);

    // One child = one sub-run, each with its OWN budget ceiling (its slice) and
    // its OWN admission/autonomy gates inside the sub-run. A child that exhausts
    // its slice is denied PRE-dispatch by the sub-run's kernel — it cannot reach
    // into a sibling's budget or the parent's headroom.
    //
    // runChild here delegates to Krelvan's existing sub-agent executor
    // (deriveSubRunId gives the deterministic, crash-safe id; executeSubRun spawns
    // / re-attaches the sub-run). Pseudocode for the two load-bearing calls:
    //
    //   const subRunId = deriveSubRunId(call.runId, call.nodeId, `${call.capability}#${task.id}`);
    //   const r = await executeSubRun({
    //     parentRunId: call.runId, tenantId: call.tenantId,
    //     nodeId: call.nodeId, capabilityName: `${call.capability}#${task.id}`,
    //     budgetCents: slices[i],                 // <- the child's hard ceiling
    //     binding: { manifestId: input.childManifestId, outputMapping: {...},
    //                onFailure: "return-error" }, // one child failing != fan-out aborting
    //     initialState: task.state,
    //     approve: call.approve,                  // parent HITL policy propagates down
    //   }, store, deps, parentProjection);
    //
    // The child re-gates EVERYTHING through admission inside the sub-run, so its
    // autonomy (suggest | act-with-veto | full) is enforced independently: a child
    // attempting a "spend"/"write-irreversible" under "act-with-veto" parks for a
    // human (status "parked-for-approval") while its siblings continue.
    const declareChild = (taskId: string, sliceCents: number): string =>
      deriveSubRunId(call.nodeId, call.capability, `${call.capability}#${taskId}`);

    const results: ChildResult[] = [];
    let cursor = 0;
    async function worker(): Promise<void> {
      while (cursor < tasks.length) {
        const i = cursor++;
        const task = tasks[i]!;
        const subRunId = declareChild(task.id, slices[i]!);
        // r = await runChild(task, slices[i], subRunId)  // -> executeSubRun(...)
        const r = await runChild(task, slices[i]!, subRunId, input.childManifestId, call);
        results.push(r);
      }
    }
    await Promise.all(Array.from({ length: Math.min(limit, tasks.length) }, worker));

    // Sum of SETTLED child costs. By construction this is <= fanoutBudgetCents
    // (each child capped at its slice; slices sum to the budget). The supervisor
    // settles this against the parent's up-front reservation; any unused cents are
    // released back to the run. claimedCostCents is the plugin's claim — the
    // supervisor records it as explicitly-untrusted next to its own observation.
    const spentCents = results.reduce((sum, r) => sum + r.costCents, 0);

    return {
      output: {
        children: results,
        completed: results.filter((r) => r.status === "completed").length,
        parked: results.filter((r) => r.status === "parked-for-approval").length,
        failed: results.filter((r) => r.status === "failed").length,
        spentCents,
      },
      claimedCostCents: spentCents,
    };
  },
};

export default OrchestrateDelegatePlugin;

// ── Grant it on a supervisor node (the "lead agent" that fans out) ─────────────
// manifest node:
//   {
//     "id": "lead",
//     "role": "split the research brief into subtopics and delegate one per subtopic",
//     "autonomy": "act-with-veto",          // lead's aggregate spend co-signed by a human
//     "capabilities": [
//       { "name": "orchestrate.delegate", "sideEffect": "spend", "budgetCents": 500 }
//     ]
//   }
// Child manifest (childManifestId) declares its own nodes/capabilities/autonomy.
// Recursion: a child can itself grant orchestrate.delegate -> recursive fan-out,
// each level reserving from its parent's slice. Depth/spend is bounded at every
// level by the same reserve-then-settle invariant. No level can overspend.
```

### 4.6 `code.execute` + `code.engineer` — the autonomous SWE loop walking the side-effect ladder *(read → write-reversible → write-irreversible)*

**Showcase rationale.** The single best template to copy because it teaches the whole Krelvan trust model in one familiar, high-value task — the autonomous SWE loop. It is the only sample that walks the full side-effect ladder in a single capability family: the *same job* is `read` (sandboxed execution), `write-reversible` (patch in a throwaway worktree), and `write-irreversible` (merge the PR), so the gating behavior differs across three steps the reader already understands. It makes "full autonomy is safe" concrete and honest (the comment ties the safety claim to the deny-all-egress sandbox mechanism) and demonstrates **CI-as-co-sign**: merge refuses unless a verifiable green-CI token is present, and that token rides into the `EffectResult` the supervisor signs.

```typescript
// capabilities/code-execute.ts
// SHOWCASE capability: run agent-authored code inside a disposable sandbox.
// sideEffect = "read": the sandbox has an ephemeral FS and NO network egress, so a run
// changes nothing in the real world and is safe to re-run -> safe at autonomy "full".
// HONESTY NOTE: the SANDBOX_RUN_URL call below is an EXTERNAL sandbox provider
// (e.g. Modal / E2B / Firecracker). It is clearly marked. Everything else is real.
import type { CapabilityPlugin, EffectCall } from "../src/core/capability/capability.js";

interface ExecInput {
  language: "python" | "node" | "bash";
  code: string;
  files?: Record<string, string>; // virtual FS seeded into the sandbox
  timeoutSec?: number;            // hard ceiling; sandbox is killed at expiry
}
interface ExecOutput {
  exitCode: number;
  stdout: string;
  stderr: string;
  durationMs: number;
  cpuSeconds: number;             // the supervisor will reconcile cost against this
}

const CENTS_PER_CPU_SEC = 1;      // integer cents only — ledger invariant (no floats)
const BASE_CENTS = 1;             // sandbox cold-start

export const CodeExecutePlugin: CapabilityPlugin = {
  name: "code.execute",
  sideEffect: "read", // isolated + no egress => reversible by construction

  estimateCents(call: EffectCall): number {
    const { timeoutSec = 30 } = call.input as ExecInput;
    // reserve worst case (full timeout). Supervisor settles the real cpuSeconds after.
    return BASE_CENTS + Math.ceil(timeoutSec) * CENTS_PER_CPU_SEC;
  },

  async invoke(call: EffectCall): Promise<{ output: unknown; claimedCostCents: number }> {
    const input = call.input as ExecInput;
    if (typeof input.code !== "string") throw new Error("code.execute: 'code' must be a string");
    const timeoutSec = Math.min(input.timeoutSec ?? 30, 120);

    // === EXTERNAL CALL (clearly marked) — a real sandbox provider runs the code ===
    // The container has: ephemeral FS, no inbound, NO network egress (enforced by the
    // provider's egress allowlist = empty). Credentials never enter the sandbox.
    const res = await fetch(process.env.SANDBOX_RUN_URL!, {
      method: "POST",
      headers: { "content-type": "application/json" },
      body: JSON.stringify({
        language: input.language, code: input.code,
        files: input.files ?? {}, timeoutSec, egress: "deny-all",
      }),
    });
    if (!res.ok) throw new Error(`code.execute: sandbox returned ${res.status}`);
    const r = (await res.json()) as ExecOutput;
    // ===========================================================================

    const claimedCostCents = Math.max(BASE_CENTS, BASE_CENTS + Math.ceil(r.cpuSeconds) * CENTS_PER_CPU_SEC);
```

### 4.7 `browser.act` — drive a sandboxed VM to operate any UI-only software *(write-irreversible)*

**Showcase rationale.** The single best teaching example in the catalog because it forces an author to use every pillar of the trust model at once and demonstrates two hard patterns honestly: **write-irreversible done right** (declaring the real class is what makes the kernel auto-park order submission on any act-with-veto/suggest node — the Operator/Mariner "a submitted order cannot be undone" problem solved natively) and **credential takeover via the secret broker** (it hands the VM only an opaque scoped token, never the raw secret — the canonical "plugins never see credentials" showcase). It also models the commit/dry-run boundary, an egress allowlist (deny-by-default destinations), integer-cent reserve-then-settle, and an untrusted `claimedCostCents`. The `BrowserVm` is an injected, clearly-marked external boundary, not a fake API.

```typescript
/**
 * browser.act — drive a sandboxed headless-browser VM to operate ANY software
 * that has a UI but no API. The defining post-agentic capability: it acts on the
 * open web/SaaS the way a person would, while the Krelvan kernel keeps it gated,
 * budgeted, and replay-safe.
 *
 * HONESTY NOTE: the BrowserVm calls below are an EXTERNAL service boundary
 * (Browserbase / Steel / a self-hosted Playwright pod). They are MARKED as such
 * and shown as an injected dependency — this file does NOT ship a real browser
 * driver. Wire your own `BrowserVm` and the trust model (admission, budget,
 * supervisor co-sign, broker, audit, replay) is provided by Krelvan for free.
 *
 * WHY write-irreversible: a submitted order, posted form, or confirmed booking
 * cannot be undone. Declaring this honestly is what makes the kernel PARK the
 * commit step for human approval on any node whose autonomy != "full".
 */
import type { CapabilityPlugin, EffectCall } from "../src/core/capability/capability.js";
import type { SecretBroker } from "../src/core/identity/identity.js";

// ---- EXTERNAL boundary (you provide this; not a real API in this file) --------
export interface BrowserVm {
  open(startUrl: string): Promise<string>;               // -> sessionId (sandboxed VM)
  // takeover: present a broker-minted token to the VM's login bridge. The VM
  // injects the credential server-side; the plan/LLM never sees raw secrets.
  authenticate(sessionId: string, brokerToken: string): Promise<void>;
  run(sessionId: string, steps: BrowserStep[], opts: { commit: boolean }): Promise<{
    transcript: { step: string; ok: boolean; note: string }[];
    finalUrl: string;
    confirmationText?: string;   // e.g. "Order #A1B2 placed" — present only when commit=true
    usageCents: number;          // VM bills compute/proxy minutes
  }>;
  close(sessionId: string): Promise<void>;
}
type BrowserStep =
  | { kind: "goto"; url: string }
  | { kind: "click"; selector: string }
  | { kind: "type"; selector: string; text: string }
  | { kind: "submit"; selector: string };   // the irreversible act

interface BrowserActInput {
  destination: string;          // e.g. "app.shopify.com" — what the takeover unlocks
  startUrl: string;
  steps: BrowserStep[];
  commit: boolean;              // false => dry-run (read-only preview, no submit)
}

// Egress allowlist: the ONLY destinations this capability may take over.
// Deny-by-default — anything else => broker.mint returns null => we refuse.
const ALLOWED_DESTINATIONS: ReadonlySet<string> = new Set([
  "app.shopify.com",
  "partner.flights.example",   // a Mariner/Operator-style booking surface
]);

export function makeBrowserActPlugin(vm: BrowserVm, broker: SecretBroker): CapabilityPlugin {
  return {
    name: "browser.act",
    sideEffect: "write-irreversible",   // honest: a submitted order cannot be undone

    // Reserved BEFORE invoke runs. VM minutes vary; reserve a sane ceiling so the
    // run's budget covers the worst case. Supervisor settles the real cost after.
    estimateCents(_call: EffectCall): number {
      return 50;   // integer cents, no floats (ledger invariant)
    },

    async invoke(call: EffectCall) {
      const input = call.input as BrowserActInput;
      const now = Date.now();

      // 1) CREDENTIAL TAKEOVER via the secret broker. We never touch the raw
      //    secret — we ask for a short-lived, single-destination scoped token.
      const scoped = broker.mint(input.destination, ALLOWED_DESTINATIONS, now, 120);
      if (!scoped) {
        // destination not allowlisted OR no secret registered -> structural refusal
        throw new Error(`browser.act: takeover denied for '${input.destination}'`);
      }

      // 2) Spin up the sandboxed VM and authenticate with the opaque token only.
      const sessionId = await vm.open(input.startUrl);          // EXTERNAL
      try {
        await vm.authenticate(sessionId, scoped.token);         // EXTERNAL (token, not secret)

        // 3) Run the plan. commit=false is a true dry-run: the VM stops before any
        //    submit step, so a counterfactual/preview never performs the act.
        const res = await vm.run(sessionId, input.steps, { commit: input.commit }); // EXTERNAL

        return {
          output: {
            committed: input.commit,
            finalUrl: res.finalUrl,
            confirmation: res.confirmationText ?? null,
            transcript: res.transcript,        // full step-by-step for the audit/ledger
          },
          // claimedCostCents is the plugin's UNTRUSTED claim; the supervisor
          // co-signs the EffectResult with what it observed. Always integer cents.
          claimedCostCents: Math.ceil(res.usageCents),
        };
      } finally {
        await vm.close(sessionId);             // EXTERNAL — always tear the VM down
      }
    },
  };
}
```

### 4.8 `skills.acquire` — self-improvement that emits a shareable signed manifest *(write-reversible)*

**Showcase rationale.** The rare capability impossible to build in the pre-agentic era and trivial to copy once it exists. It exercises all the load-bearing primitives in one file — it READS the agent's own ledger trace, calls `memory.persist` to write a provenance-tracked `SemanticFact`, and emits a signed manifest (Path C) as output, connecting the TS plugin path and the shareable-manifest path end to end. It models side-effect honesty (`write-reversible`, explicitly not spend/message/identity), demonstrates that admission stays **deny-by-default across sharing** (acquired skills land with `budgetCents:0` and `autonomy:"suggest"`, so first reuse parks for human approval and cannot bypass a gate), and is honest about the LLM (the one non-deterministic touchpoint is an injected `ModelPort`, its output captured so replay re-feeds by hash and never re-calls the model). The resulting skill manifests are the shareable units of the self-host extension gallery — the durable, self-hostable wedge.

```typescript
// capabilities/skills-acquire.ts
//
// SHOWCASE capability: skills.acquire
// Genuinely-agentic self-improvement. After a run succeeds, the agent REFLECTS on
// its own ledger-backed trace, distills the repeatable part into a reusable "skill",
// persists it to its semantic-memory plane (memory.persist), and emits a SignedManifest
// that any other Krelvan install can import as a template.
//
// Why write-reversible (honest classification): it durably writes a new skill record
// + a draft manifest the owner can delete/supersede. It does NOT spend, message a
// human, or mutate identity. A newly-acquired skill is INERT until an owner grants
// its capabilities in a manifest — admission stays deny-by-default on first reuse,
// so a shared skill can never bypass a gate just because it was imported.
//
// EXTERNAL CALL — clearly marked, NOT a fake "Krelvan API":
//   The distillation step calls an LLM via an injected ModelPort. Its non-deterministic
//   output is CAPTURED into the result (so replay re-feeds the captured skill by hash,
//   never re-calls the model). No invented endpoints; the port is supplied by the host.

import type { CapabilityPlugin, EffectCall } from "../core/capability/capability.js";
import type { SemanticFact } from "../core/memory/memory.js";
import { distilledProvenance } from "../core/memory/memory.js";          // reuse real fn
import { saveSemanticFact } from "../core/plugins/memory-plugins.js";     // memory.persist
import { contentAddress } from "../core/ledger/crypto.js";                // real addresser
import { canonicalize } from "../core/ledger/canonical.js";

// Injected, not imported from a fake SDK. Host wires the real model in genesis.config.ts.
export interface ModelPort { complete(prompt: string): Promise<string>; }

export function makeSkillsAcquire(model: ModelPort): CapabilityPlugin {
  return {
    name: "skills.acquire",
    sideEffect: "write-reversible",

    // Flat integer cents (ledger invariant: no floats). One reflective LLM pass.
    estimateCents: () => 3,

    async invoke(call: EffectCall) {
      const input = call.input as Record<string, unknown>;
      const agentId = String(input["_agentId"] ?? "default");
      const runId   = String(input["_runId"]   ?? call.nodeId);

      // 1) READ the agent's own captured trace from run state (already in the ledger).
      const trace = String(input["trace"] ?? input["recall.last_summary"] ?? "");
      const sourceProvenances = (Array.isArray(input["_provenances"])
        ? (input["_provenances"] as string[])
        : ["tool-observed"]) as ("owner"|"tool-observed"|"channel"|"agent"|"memory")[];

      // 2) REFLECT + DISTILL (EXTERNAL, non-deterministic — captured below).
      const raw = await model.complete(
        `From this successful run, extract a reusable skill as JSON ` +
        `{ "name": "...", "intent": "...", "steps": ["..."], "capabilities": ["..."] }.\n` +
        `Trace:\n${trace}`
      );
      const skill = JSON.parse(raw) as {
        name: string; intent: string; steps: string[]; capabilities: string[];
      };

      // 3) Untrustedness propagates — never washes out (reuse real rule).
      const prov = distilledProvenance(sourceProvenances);

      // 4) memory.persist — durable semantic fact, provenance back to the episode.
      const fact: SemanticFact = {
        key: `skill:${skill.name}`,
        value: skill.intent,
        derivedFrom: [runId],
        provenance: prov,
        distilledBy: "skills.acquire@1",
        version: 1,
        ts: Date.now(),
      };
      saveSemanticFact(agentId, fact);

      // 5) Emit a SHAREABLE template (Path C). UNSIGNED draft — the owner re-signs on
      //    deploy (their install, their authority). Capabilities land as deny-by-default
      //    grants with budgetCents:0 so reuse cannot act until the owner opens a gate.
      const manifestBody = {
        version: 1 as const,
        name: skill.name,
        intent: skill.intent,
        entry: "step-0",
        runBudgetCents: 0,                 // owner must fund before first reuse
        maxNodeVisits: 1,
        nodes: skill.steps.map((role, i) => ({
          id: `step-${i}`,
          role,
          autonomy: "suggest" as const,     // first reuse parks for human approval
          capabilities: skill.capabilities.map(name => ({
            name, sideEffect: "read" as const, budgetCents: 0,
          })),
        })),
        edges: skill.steps.slice(1).map((_, i) => ({ from: `step-${i}`, to: `step-${i+1}` })),
      };
      const draftId = "sha256:" + contentAddress(canonicalize(manifestBody));

      // 6) Return BOTH artifacts. claimedCostCents is an untrusted claim — the
      //    supervisor co-signs the EffectResult, not this plugin.
      return {
        output: {
          "skill.acquired": skill.name,
          "skill.provenance": prov,                 // surfaces quarantine if untrusted
          "skill.manifestDraftId": draftId,
          "skill.manifest": manifestBody,           // CAPTURED → replay re-feeds by hash
          "skill.sharable": true,
          "skill.inertUntilGranted": true,          // honest: no auto-acquired authority
        },
        claimedCostCents: 3,
      };
    },
  };
}
```

## 5. How these plug in

All capabilities — core, community-authored, MCP-admitted, or YAML-wrapped — implement the same **four-method `CapabilityPlugin` contract** and inherit the platform's budget, admission, audit, and idempotency guarantees for free. See [`EXTENSION_MODEL.md`](./EXTENSION_MODEL.md) for the authoritative reference; the summary:

### The `CapabilityPlugin` contract (4 members)
1. `name` — the capability id used in manifests and admission.
2. `sideEffect` — one of the six side-effect classes (below); the single most important authoring decision. *When in doubt, declare higher.*
3. `estimateCents(call)` — the worst-case integer-cent reservation; the kernel reserves this *before* `invoke()` runs, so an over-ceiling call is denied pre-dispatch.
4. `invoke(call)` — performs the effect and returns `{ output, claimedCostCents }`; `claimedCostCents` is **untrusted** — the supervisor co-signs the observed cost into the ledger.

### The six side-effect classes (the risk ladder)
`read` → `write-reversible` → `write-irreversible` → `spend` → `message-human` → `identity-mutation`. The class drives the automatic approval gate via `needsApproval(autonomy, effect)`: `read` never gates; under `act-with-veto`, `write-irreversible`/`spend`/`identity-mutation` always park; under `suggest`, any non-`read` effect parks; under `full`, side effects proceed (safe only when the effect is isolated/reversible).

### The three authoring paths
- **Path A — TypeScript plugin.** Custom logic, auth flows, multi-step sessions, control flow over other capabilities. (e.g. `verify.cosign`, `hitl.escalate`, `browser.act`, `orchestrate.delegate`.)
- **Path B — YAML capability.** No-code wrap of an HTTP API; declare endpoint, params, and side-effect class. The lowest-friction on-ramp; secrets resolved by the broker, never in the YAML. (e.g. `http.wrap_api`, `act.execute_effect` connectors.)
- **Path C — signed JSON manifest.** Wires admitted nodes/capabilities into an agent; the shareable, content-addressed unit of the extension gallery. (e.g. the output of `skills.acquire`.)

Across all paths, admission is **deny-by-default**: a discovered, imported, or freshly authored capability is inert until an owner grants it in a manifest, and on first side-effecting use it parks for approval.
