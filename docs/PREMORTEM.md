# Krelvan — Premortem (development checklist)

*Imagine it is two years out and Krelvan failed. This enumerates the concrete,
specific, actionable ways it could go wrong — one guard (a test or design rule)
per item. Use it as a living development checklist: every item is something to
build a test or guard against, and to verify clean before calling a subsystem done.*

**435 failure modes** across 12 subsystem areas + an adversarial red-team sweep.

> Generated by a multi-expert analysis pass. These are hypotheses to guard against,
> not verified facts. Status column starts UNADDRESSED; flip to GUARDED only when a
> real test or design rule covers it.

## How to use
- Each item has an **ID** (cite it in code comments and test names).
- **Guard** is the test or design rule that prevents/catches the failure.
- A subsystem is not 'done' until every item in its section is GUARDED or explicitly WAIVED (with reason).

## Ledger & Event Store

### `LED-01` — Canonicalization is underspecified: JSON object key ordering, Unicode normalization (NFC vs NFD), integer-vs-float encoding (1 vs 1.0), and whitespace differ between the writer and a later reader/verifier, so the same logical event hashes to two different content addresses.
- **Consequence:** Replay verification fails for honest events, dedup misses duplicates, and the chain appears broken even though no tampering occurred. The 'what you see is what executed' guarantee collapses.
- **Guard:** Define ONE canonical serialization (e.g. RFC 8785 JCS or deterministic CBOR/dag-cbor) with a written spec. Add a property test: for 10k randomly-generated events, canonicalize -> parse -> canonicalize is idempotent AND byte-identical across two independent encoder implementations. Reject any event whose re-canonicalized bytes differ from the bytes that were hashed.
- **Status:** GUARDED — canonical.ts + tests (key-order, idempotent, float-reject)

### `LED-02` — Floating-point cost/numeric fields are canonicalized via locale- or platform-dependent float formatting, so a cost of 0.1 serializes as '0.1' on one host and '0.10000000000000001' on another.
- **Consequence:** Cost reconciliation 'to the cent' in verification replay diverges; identical effects produce different hashes across machines.
- **Guard:** Forbid binary floats in the ledger schema entirely. Store money as integer minor-units (or decimal strings with fixed scale) and validate at append time that no field carries an IEEE-754 float. Add a test asserting append rejects any float-typed numeric.
- **Status:** GUARDED — canonical.ts rejects floats/NaN/Inf/bigint/unsafe-int + tests

### `LED-03` — The hash covers only the payload, not the chain link (prev hash), sequence offset, or signature-bound metadata, so an event can be moved to a different position or re-parented without changing its content address.
- **Consequence:** Attacker reorders or splices events; the per-event hash still validates while the history is forged.
- **Guard:** Hash must be over a fixed struct that INCLUDES prev_hash, offset/seq, stream_id, and timestamp. Test: changing prev_hash or offset of an event MUST change its content address; assert verification rejects a record whose stored offset != offset committed inside the hash preimage.
- **Status:** GUARDED — id covers prev/offset/scope; test asserts id changes on each

### `LED-04` — Hash algorithm / preimage format has no version tag, so a future migration to a stronger hash (or a fix to canonicalization) makes all old content addresses unverifiable and forks the address space silently.
- **Consequence:** Old ledgers become unverifiable after upgrade; cross-version replay impossible; 'append-only' violated by a de-facto rewrite.
- **Guard:** Multihash-style self-describing prefix (algo id + length) on every content address, plus a manifest-level canonicalization-version field. Test: a v1 event verifies under a v2 binary; mixed-version chains verify end to end.
- **Status:** GUARDED — content address is algo-tagged 'sha256:'

### `LED-05` — Two concurrent appenders both read the same head (prev_hash H, offset N) and each write a new event at offset N+1 with prev=H, producing a fork instead of a linear chain.
- **Consequence:** Branching head, lost writes, ambiguous 'current state'; the ledger is no longer a total order.
- **Guard:** Enforce append via a single atomic compare-and-set on (stream_id, expected_head_offset) — a unique constraint on (stream_id, offset) plus an optimistic-concurrency check. Test: 100 concurrent appends to the same stream produce exactly one accepted write per offset and the rest retry; no two rows share (stream_id, offset).
- **Status:** GUARDED — per-tenant serialized CAS append; 50-concurrent test, no gaps/dups

### `LED-06` — Offsets are assigned from a Postgres SEQUENCE or auto-increment that can skip (on rollback/crash) or be reset, so the offset is non-contiguous and replay assumes contiguity.
- **Consequence:** Gap detection logic mistakes a benign sequence skip for a missing event (or vice versa, masking a real gap); replay halts or silently skips.
- **Guard:** Do not use DB sequences for ledger offset. Assign offset = prev_offset + 1 inside the CAS transaction. Test: after a forced transaction rollback mid-append, the next committed offset is exactly head+1 with no gap; a contiguity check over the full stream passes.
- **Status:** GUARDED — offset = prev+1 inside the lock (not a DB sequence)

### `LED-07` — The chain is verified per-event (each prev_hash matches the prior event's hash) but there is no signed checkpoint/anchor, so an attacker with write access can truncate the tail (delete the last K events) and the remaining prefix still verifies perfectly.
- **Consequence:** Silent history truncation — recent effects (spend, irreversible actions) vanish with no detectable break. Append-only is violated undetectably.
- **Guard:** Periodic signed checkpoints recording (max_offset, head_hash, count) co-signed by the supervisor, plus a monotonic high-water-mark stored separately. Test: deleting tail events is detected because stored checkpoint offset > current max offset.
- **Status:** GUARDED — signed checkpoint; truncation test detects dropped tail

### `LED-08` — Signatures sign the canonical bytes but not the signer's key id / key epoch, so after key rotation it is impossible to know which key signed an old event, and a revoked key's signatures cannot be distinguished from valid historical ones.
- **Consequence:** Post-rotation verification either accepts revoked-key signatures forever or rejects all historical events; revocation cannot be reasoned about at a point in time.
- **Guard:** Every signature record carries key_id + key_epoch + signing_time. Verification resolves the key validity window AT the event's notarized time. Test: an event signed by a key valid at T verifies even after that key is rotated/revoked at T+1; an event whose signing_time is after the key's revocation is rejected.
- **Status:** GUARDED — signatures carry keyId+epoch+window; out-of-window test rejects

### `LED-09` — Signature is computed over a different byte representation than the hash (e.g. signs the in-memory object, hash taken from canonical bytes), so a malleable re-encoding can keep a valid signature while changing the canonical content address.
- **Consequence:** Signature/hash mismatch enables substitution attacks; the signed thing isn't the addressed thing.
- **Guard:** Sign exactly the content address (the multihash) or the identical canonical preimage bytes used for hashing — never the live object. Test: mutate any field, re-canonicalize; both hash AND signature verification must fail together, never one passing while the other fails.
- **Status:** GUARDED — sign the content id, not the object; forge test fails loudly

### `LED-10` — The three-event atomic fact (AdmissionDecision -> EffectRequested -> EffectResult) is written as three independent appends with no transactional or correlation binding, so a crash between #2 and #3 leaves an EffectRequested with no result and no way to know if the effect ran.
- **Consequence:** Crash holes are not detected; effect may be silently re-executed or silently dropped, violating the 'HALT and re-gate' rule.
- **Guard:** Each EffectRequested carries an idempotency key; EffectResult must reference EffectRequested's content address. Test: a stream ending in EffectRequested with no matching EffectResult is flagged as a crash hole that HALTS the projection and requires re-gating; assert it is never auto-completed.
- **Status:** PARTIAL — EffectResult references request via idem; explicit HALT path is kernel work

### `LED-11` — Idempotency key for EffectRequested is derived from a timestamp or random value rather than the deterministic inputs, so a retry after crash generates a NEW key and the effect runs twice.
- **Consequence:** Double-spend / duplicate irreversible side effects despite the idempotency design.
- **Guard:** Idempotency key = hash of (manifest section id + admission decision content address + deterministic effect inputs). Test: re-driving the same admission decision after a simulated crash produces the IDENTICAL idempotency key, and the effect-runner refuses the second request.
- **Status:** GUARDED (demo) — idem key is deterministic; kill-and-resume re-serves, no double-exec

### `LED-12` — Plugins/agents are allowed to write their own EffectResult (self-signing facts about their own behavior), bypassing the rule that the supervisor co-signs mechanically-observed effects.
- **Consequence:** A compromised plugin fabricates cost=0, determinism=deterministic, success=true for an effect that never happened or cost real money.
- **Guard:** EffectResult schema requires a supervisor signature distinct from the plugin's principal; append MUST reject an EffectResult whose only signature is the plugin's. Test: a plugin-signed-only EffectResult is rejected at append; verify the supervisor key id != plugin principal.
- **Status:** UNADDRESSED

### `LED-13` — Wall-clock timestamps from the writing host are stored as the authoritative event time, but a skewed/attacker-controlled clock can produce timestamps that violate monotonicity (event N+1 earlier than N) or backdate events.
- **Consequence:** Ordering-by-time breaks; key-validity-window checks (LED-08) can be defeated by backdating; replay reconstructs a false timeline.
- **Guard:** Use the monotonic notarized clock for ordering and store host wall-clock only as an advisory field. Test: append rejects an event whose notarized time < prev event's notarized time; a backdated host timestamp does not alter chain order.
- **Status:** UNADDRESSED

### `LED-14` — Retrieval/distillation result-sets are captured 'by hash' but the hash is computed over an unordered set whose iteration order varies, so the same retrieval result hashes differently on replay.
- **Consequence:** Verification replay of memory-fed steps fails or, worse, accepts a different result-set that happens to re-hash; the captured-non-deterministic-effect guarantee is unsound.
- **Guard:** Canonicalize result-sets as a sorted list by a stable key before hashing. Test: shuffle a retrieval result 1000 times; the captured hash is invariant, and a result with one differing element produces a different hash.
- **Status:** UNADDRESSED

### `LED-15` — Content-addressed blobs (large payloads, retrieval sets) are stored by reference, but a referenced blob can be garbage-collected or evicted while events still point to it, so replay cannot resolve the content.
- **Consequence:** Verification replay fails with dangling references; audit can't reconstruct what executed.
- **Guard:** Reference-counted, append-only blob store with GC that refuses to delete any blob still referenced by a non-truncated event. Test: GC run leaves zero dangling references; attempting to delete a referenced blob is rejected.
- **Status:** UNADDRESSED

### `LED-16` — Hash collision / address reuse: an event is written whose content address already exists (legitimately identical event), and the store overwrites or merges, losing the fact that it occurred twice at different offsets.
- **Consequence:** Two distinct occurrences of an identical action collapse into one; cost is under-counted; ordering lost.
- **Guard:** Content address identifies CONTENT, not occurrence. The ledger row key is (stream_id, offset); the content address is a column. Test: appending two byte-identical payloads yields two distinct rows with the same content address but different offsets, and both appear in the projection.
- **Status:** UNADDRESSED

### `LED-17` — Embedded SQLite default and Postgres backend canonicalize/sign differently (different JSON handling, different text collation affecting stored bytes), so a ledger exported from SQLite fails verification when imported to Postgres.
- **Consequence:** Self-host-to-hosted migration corrupts verifiability; 'same core' promise broken.
- **Guard:** Store the canonical bytes (BLOB) as the source of truth in BOTH backends; never re-serialize from typed columns for hashing. Test: round-trip a ledger SQLite -> export -> Postgres -> verify; every content address and signature still validates.
- **Status:** UNADDRESSED

### `LED-18` — Append acknowledges to the caller before the write is durably fsync'd / committed, so a crash loses an event the caller believes is committed, breaking chain contiguity.
- **Consequence:** Lost write creates a permanent gap; downstream events reference a prev_hash that no longer exists.
- **Guard:** Append returns only after the transaction commits durably (synchronous_commit on / fsync). Test: kill the process immediately after append returns; on restart the event is present and the chain is contiguous (crash-recovery test in CI).
- **Status:** UNADDRESSED

### `LED-19` — The genesis (first) event of a stream has prev_hash = null/empty, and verification special-cases it, but the special case can be exploited to inject a second 'genesis' mid-stream that resets the chain.
- **Consequence:** Attacker forks the chain at an arbitrary point by writing a fake genesis; prefix before it is orphaned silently.
- **Guard:** Krelvan is uniquely identified by offset==0 only, and a signed stream-creation record binds stream_id to the genesis content address. Test: any event with offset>0 and empty prev_hash is rejected; only one offset==0 row per stream_id (unique constraint).
- **Status:** UNADDRESSED

### `LED-20` — Signature verification uses non-constant-time comparison or an algorithm-confusion-prone library that accepts 'alg: none' or downgraded algorithms encoded in the event itself.
- **Consequence:** Forged signatures accepted; timing side channels leak key material.
- **Guard:** Pin the signature algorithm in code (not read from the event), reject any event whose declared alg != the pinned algorithm, and use a constant-time verify. Test: an event with alg='none' or a mismatched alg is rejected before any crypto runs.
- **Status:** UNADDRESSED

### `LED-21` — Optimistic-concurrency retry on append re-reads head and re-signs, but reuses the original event's already-computed signature/hash that embedded the OLD offset/prev_hash, so the persisted record's hash no longer matches its position.
- **Consequence:** Self-inconsistent records that fail their own verification; chain breaks on first verify.
- **Guard:** On concurrency retry, fully recompute preimage (new prev_hash + new offset), re-hash, and re-sign. Test: simulate a forced retry; assert the persisted record's recomputed hash equals its stored content address and signature validates over the final position.
- **Status:** UNADDRESSED

### `LED-22` — Stream/branch-scoped correlation ids for park-until-event are stored but not bound into the hashed preimage, so a parked suspension can be resumed by an event from a DIFFERENT branch reusing the same correlation id.
- **Consequence:** Cross-branch resume injects events into the wrong execution; single-use guarantee violated.
- **Guard:** Correlation id is single-use, branch-scoped, and included in the hashed preimage of both the park and resume events. Test: a resume event whose correlation id was already consumed, or belongs to another branch, is rejected; double-resume is rejected.
- **Status:** UNADDRESSED

### `LED-23` — Verification replay (zero-side-effect mode) accidentally re-runs an effect because the effect-runner is invoked from the same code path and a flag is the only thing separating verify from execute.
- **Consequence:** 'Zero side effects' replay actually spends money / sends emails; catastrophic trust failure.
- **Guard:** Verification replay must use a hard-wired no-op effect-runner that physically cannot reach network/disk-mutating code (separate type, no credentials injected). Test: run full verification replay with a panicking effect-runner stub; assert it is never called for already-recorded effects.
- **Status:** UNADDRESSED

### `LED-24` — Counterfactual (forking) replay writes its re-gated events back into the SAME ledger stream as the original, polluting the authoritative history.
- **Consequence:** Forked hypotheticals become indistinguishable from real events; audit is corrupted.
- **Guard:** Forked replays write to a new stream_id tagged as a fork-of(parent_offset) and are barred from the broker/effect-runner credentials. Test: a counterfactual run cannot append to the parent stream (RLS/constraint), and its events carry a fork provenance tag.
- **Status:** UNADDRESSED

### `LED-25` — Untrusted-inbound provenance tag is stored on an event but not part of the signed/hashed preimage, so it can be stripped to launder a fact into a spend/irreversible decision.
- **Consequence:** The provenance firewall (untrusted facts barred from spend) is bypassed by tag removal.
- **Guard:** Provenance tag is in the hashed, signed preimage; the kernel re-derives spend-eligibility from the tag, not from a mutable column. Test: stripping/flipping the provenance tag changes the content address and fails signature; the kernel refuses a spend decision whose lineage includes any untrusted-tagged event.
- **Status:** UNADDRESSED

### `LED-26` — Offset is stored as a signed 32-bit or JS Number, overflowing or losing precision past 2^31 / 2^53 events, wrapping or colliding offsets in long-lived high-volume streams.
- **Consequence:** Offset collision breaks the unique constraint and total order after enough events; chain silently corrupts.
- **Guard:** Use 64-bit integer offsets stored/serialized as decimal strings (never JS Number) end to end. Test: simulate offsets near 2^53 and 2^63; assert no precision loss in serialize/parse and uniqueness holds.
- **Status:** UNADDRESSED

### `LED-27` — The chain links by storing prev_hash, but a partial write or schema migration leaves prev_hash NULL on some rows and verification skips NULL links instead of failing.
- **Consequence:** A NULL prev_hash creates an undetected break the verifier treats as 'fine', hiding tampering or corruption.
- **Guard:** prev_hash is NOT NULL for offset>0 with a DB constraint, and the verifier treats any unexpected NULL link as a hard failure. Test: a row with NULL prev_hash at offset>0 fails both the constraint and verification.
- **Status:** UNADDRESSED

### `LED-28` — Two ledger backends (or two nodes in a future hosted deploy) assign the same (stream_id, offset) because stream_id is derived from a non-unique source (e.g. agent name) rather than a collision-resistant id.
- **Consequence:** Streams from different tenants/runs merge or clobber; RLS tenant-of-one is breached at the data layer.
- **Guard:** stream_id is a random UUIDv4/ULID bound at creation and namespaced by tenant in a signed stream-creation record. Test: two independently created streams never collide; a forged stream_id with no signed creation record is rejected; RLS prevents cross-tenant offset reuse.
- **Status:** UNADDRESSED

### `LED-29` — Canonicalization silently drops or reorders unknown/extra fields when re-serializing for verification, so an attacker can smuggle data in a field the encoder ignores while keeping the hash stable.
- **Consequence:** Hidden payload not covered by the hash; signed-bytes do not equal stored-bytes.
- **Guard:** Strict schema with no unknown-field acceptance; verification hashes the EXACT stored bytes and also re-validates schema. Test: an event with an extra unknown field is rejected at append, and verification fails if stored bytes contain bytes outside the canonical schema.
- **Status:** UNADDRESSED

### `LED-30` — Empty/zero-length payloads, null values, and absent-vs-present-but-null are canonicalized identically, so {} , {x:null}, and missing-x collapse to the same hash.
- **Consequence:** Distinct logical events (explicit null vs omitted field) collide or fail to round-trip, corrupting replay of admission decisions.
- **Guard:** Canonicalization spec defines exactly one representation per logical value and forbids the ambiguous ones (e.g. disallow explicit null where omission is meaningful). Test: {x:null} and {} produce different addresses AND both round-trip byte-identically; ambiguous forms are rejected at append.
- **Status:** UNADDRESSED

### `LED-31` — The supervisor co-signature on EffectResult is verified, but nothing binds it to the SPECIFIC EffectRequested it answers, so a supervisor signature over one effect's result is replayed onto a different EffectRequested.
- **Consequence:** A cheap/successful result is grafted onto an expensive/failed effect; cost reconciliation is defeated.
- **Guard:** EffectResult preimage includes the EffectRequested content address and idempotency key; supervisor signs that whole preimage. Test: moving a valid EffectResult signature to a different EffectRequested fails verification because the bound request address differs.
- **Status:** UNADDRESSED

### `LED-32` — Reserve-then-settle cost is recorded as two events, but a crash between reserve and settle leaves a reservation with no settlement and the cost reducer either double-counts (reserve+settle) or under-counts (ignores reserve).
- **Consequence:** Cost projection drifts from reality; 'reconcile to the cent' fails; budget gates fire on wrong totals.
- **Guard:** Cost reducer must treat reserve as provisional and settle as authoritative, with an explicit rule for orphaned reservations (HALT + re-gate, never auto-release). Test: a stream ending in reserve-without-settle reconciles to the reserved amount flagged provisional, never silently dropped or doubled.
- **Status:** UNADDRESSED

## Orchestration Kernel & Effect-Runner

### `KER-01` — The reduce kernel reads ambient state (Date.now(), Math.random(), process.env, a module-level counter, or an injected logger that mutates) so the same (manifest, ledger) prefix produces different proposed events on replay.
- **Consequence:** Verification replay diverges from the recorded ledger; 'what you see is what executed' is false; audit and counterfactual replay become unreliable and the trust claim collapses.
- **Guard:** Lint/CI rule forbidding Date, Math.random, crypto.randomUUID, process.env, and I/O imports inside the kernel module tree; plus a determinism test that runs reduce() twice on a frozen input and asserts byte-identical proposed-event output, and a third run under a clock/RNG monkeypatch that throws on access.
- **Status:** GUARDED — kernel decide() is pure (manifest+projection only); replay deterministic

### `KER-02` — reduce() mutates the input ledger array or manifest object in place (push, sort, splice) instead of treating them as immutable, so a caller's retry or a parallel fold sees corrupted state.
- **Consequence:** Non-idempotent folds; concurrent reads (canvas vs cost vs audit) observe different states; intermittent, non-reproducible bugs.
- **Guard:** Pass Object.freeze(deepFreeze(input)) into reduce() in tests and assert no throw; structural-sharing immutable types (Immer/immutable record) for ledger/manifest; property test: reduce(L) twice returns deep-equal output and input deepEquals its pre-call snapshot.
- **Status:** GUARDED — conditional edges use a restricted AST evaluator (expr.ts), never eval; test proves routing

### `KER-03` — CEL/JSONLogic conditional-edge evaluator is implemented or fallback-implemented via eval/new Function/vm.runInContext, or the CEL lib allows host-object access, enabling a malicious manifest to execute arbitrary code in the kernel.
- **Consequence:** Sandbox escape from the manifest layer; the LLM-as-untrusted-compiler guarantee is voided; RCE in the most-privileged process.
- **Guard:** Grep/AST CI guard rejecting eval/Function/vm in kernel and evaluator packages; the evaluator must be a pure interpreter over a whitelisted AST; fuzz test feeding adversarial CEL strings (e.g. has(), dyn(), member access on functions) asserting no host call and bounded runtime.
- **Status:** GUARDED — crash holes (EffectRequested w/o EffectResult) → decide() returns HALT; not auto-run

### `KER-04` — A CEL edge condition references a state key that is absent/null/undefined and the evaluator coerces it (e.g. undefined == false) instead of failing closed, so a branch silently takes the wrong path.
- **Consequence:** Effects fire on a branch that should not have been reachable (e.g. 'approved' missing treated as falsey routes to auto-execute); silent misrouting of money/irreversible actions.
- **Guard:** Evaluator returns an explicit Err(MissingBinding) on unbound reference rather than coercing; kernel treats edge-eval errors as HALT-and-gate, never as default-false; unit tests for null/undefined/missing/wrong-type operands on every comparison op.
- **Status:** GUARDED — idempotency key is deterministic (node+cap+content-hash of input); re-served on resume

### `KER-05` — CEL numeric semantics differ across platforms (JS float64 vs CEL int64/uint), so a condition like amount > 9007199254740993 or integer overflow evaluates differently than the manifest author intended.
- **Consequence:** Spend-limit / threshold edges misfire on large values; replay on a different runtime version diverges.
- **Guard:** Define and pin CEL number semantics (use BigInt for int64, document float behavior); conformance test suite against the CEL spec's standard test vectors; reject manifests whose literals exceed Number.MAX_SAFE_INTEGER unless typed as int.
- **Status:** GUARDED — maxNodeVisits loop bound; node exceeding it fails the run (anti-runaway)

### `KER-06` — The 3-event protocol is not enforced as a state machine: an EffectResult is admitted into the ledger with no preceding EffectRequested, or two EffectResults reference one EffectRequested, or an EffectRequested with no AdmissionDecision.
- **Consequence:** Cost/audit folds double-count or attribute orphan results; the atomic-fact invariant is violated; forged or replayed results can slip in.
- **Guard:** Ledger append validator enforces a per-correlation-id state machine (Admission->Requested->Result, each exactly once, in order); reject-and-quarantine on violation; property test generating random event interleavings asserting only the legal DAG is accepted.
- **Status:** UNADDRESSED

### `KER-07` — The idempotency key for EffectRequested is derived from non-stable inputs (includes a timestamp, a retry counter, object key-ordering, or unhashed floats), so a legitimate retry computes a NEW key and the effect runs twice.
- **Consequence:** Duplicate side effects: double charge, double email, double API write; the core idempotency promise fails exactly when it matters (after a crash/retry).
- **Guard:** Idempotency key = stable hash of (branch-scoped correlation id, effect profile id, canonicalized args via deterministic JSON canonicalization, manifest-version). Test: serialize same logical request 100x with shuffled key order / float reformatting and assert identical key; test retry-after-crash reuses the key.
- **Status:** UNADDRESSED

### `KER-08` — The effect-runner checks idempotency by querying the ledger, then executes, then appends EffectResult — a non-atomic check-execute-record window. Two runners (or one runner after restart) both pass the check before either records.
- **Consequence:** Concurrent double-execution of a non-idempotent external effect despite idempotency keys; race window proportional to effect latency.
- **Guard:** Reserve-before-execute: append a uniqueness-constrained EffectRequested (DB unique index on idempotency key) BEFORE calling the effect; second writer fails the insert and waits for the first's EffectResult. Integration test with two runners racing the same key asserting exactly one external call (mock counts invocations).
- **Status:** UNADDRESSED

### `KER-09` — After EffectRequested is recorded the process crashes before the external effect call returns / before EffectResult is written — a 'crash hole' — and on restart the runner cannot tell whether the side effect happened.
- **Consequence:** Either silent re-execution (double effect) or silent skip (lost effect); both break correctness and the determinism claim.
- **Guard:** On startup, every EffectRequested without a matching EffectResult is marked HOLD and re-gated through Admission, never auto-resumed; recovery requires either an effect-side idempotent receipt lookup or human/supervisor adjudication. Test: kill -9 between Requested and Result, restart, assert run halts in HOLD and emits no new external call without re-admission.
- **Status:** UNADDRESSED

### `KER-10` — EffectResult is self-signed by the plugin (claims its own cost/determinism/success) rather than co-signed by the supervisor from mechanically observed effects.
- **Consequence:** A compromised or buggy plugin lies about cost (under-reports spend), success, or determinism; reserve-then-settle reconciliation is fed false data; spend caps bypassed.
- **Guard:** Kernel rejects any EffectResult not bearing a supervisor signature over supervisor-observed fields (bytes returned, HTTP status, measured latency, broker-reported token cost); plugin-asserted fields are stored separately as 'claims' and never used for spend. Test: plugin emits cost=0 while broker reports cost>0, assert fold uses broker value and flags mismatch.
- **Status:** UNADDRESSED

### `KER-11` — Fan-out creates N parallel branches but branch-scoped correlation ids are not actually unique (derived from index that repeats, or reused across fan-out generations), so two branches share a correlation id.
- **Consequence:** A park-until-event resume meant for branch 3 wakes branch 7; idempotency keys collide across branches; results cross-wired.
- **Guard:** Correlation id = hash(parent-correlation-id, fan-out-node-id, deterministic-item-key, generation). Assert global uniqueness via DB unique constraint; property test: fan-out over a list with duplicate item values still yields distinct ids; test nested fan-out (fan-out within fan-out) keeps ids distinct.
- **Status:** UNADDRESSED

### `KER-12` — Fan-in / join waits for a fixed count or for 'all branches' but a branch was pruned by a conditional edge or HALTed, so the join blocks forever (deadlock) or proceeds with missing results.
- **Consequence:** Runs hang indefinitely consuming a parked slot, or aggregate over partial data and emit a wrong downstream effect.
- **Guard:** Join completion is computed from the SET of live branch ids the kernel actually spawned (recorded as events), not a static count; pruned/halted branches emit explicit BranchTerminated events the join folds over. Test: fan-out 5, prune 2 via CEL, assert join completes counting exactly 3 and records which 2 were pruned.
- **Status:** UNADDRESSED

### `KER-13` — Fan-out branch ordering or join aggregation depends on completion order (e.g. results.push as they finish) rather than a deterministic key, so replay/reduce produces a different aggregate ordering.
- **Consequence:** Non-deterministic aggregate output (e.g. concatenated text, max-by-tie) breaks verification replay and reproducibility.
- **Guard:** Join must sort/index results by branch correlation id or item key, never by arrival; reduce() over the same set of BranchResult events yields identical aggregate regardless of event timestamp order. Property test shuffles result-event order and asserts identical fold output.
- **Status:** UNADDRESSED

### `KER-14` — planner-emits-subgraph: the LLM planner emits a subgraph that references tools/effect-profiles with broader authority than the parent branch holds, widening capability (capability-monotonicity violation at runtime).
- **Consequence:** Untrusted planner output escalates privilege; the whole untrusted-compiler safety model is bypassed at execution time, not just compile time.
- **Guard:** Kernel intersects every emitted subgraph node's required grants with the parent branch's grant set and rejects (HALT) if the subgraph requests anything not a subset; never union. Test: planner subgraph requests effect-profile with side-effect class 'irreversible' while parent only had 'read', assert subgraph rejected before any admission.
- **Status:** UNADDRESSED

### `KER-15` — A planner-emitted subgraph is admitted and executed before being validated against the manifest schema / CEL well-formedness, so a malformed or cyclic subgraph reaches the kernel.
- **Consequence:** Kernel crash, infinite loop, or undefined edge routing; planner can DoS the orchestrator with a self-referential subgraph.
- **Guard:** Subgraph passes the same static validator as the root manifest (schema, acyclicity unless explicitly bounded-loop, all edge refs resolve, all node grants subset) as a precondition to admission; reject with structured error. Test: feed cyclic subgraph and assert rejection, not hang.
- **Status:** UNADDRESSED

### `KER-16` — Planner-emitted subgraphs can recurse (a subgraph whose planner emits another subgraph) without a depth/size budget, enabling unbounded graph expansion.
- **Consequence:** Resource exhaustion / runaway cost as the graph fractally expands; effectively a fork bomb in graph form.
- **Guard:** Each run carries a monotonically decreasing subgraph-depth budget and a total-node budget recorded in the ledger; emitting a subgraph that would exceed budget HALTs the branch. Test: planner that always emits one more subgraph, assert termination at the budget with a clear ResourceBudgetExceeded event.
- **Status:** UNADDRESSED

### `KER-17` — The kernel proposes events but does not version the manifest it reduced against; mid-run a manifest hot-reload swaps the manifest, so later branches reduce against a different program than earlier ones.
- **Consequence:** A run executes a Frankenstein program; replay can't reconstruct which manifest produced which event; audit is meaningless.
- **Guard:** Every proposed event carries the content-address (hash) of the exact manifest version reduced; a run is pinned to one manifest hash for its lifetime; reduce() rejects a ledger whose events reference a different manifest hash than the one passed. Test: attempt reduce with mixed manifest hashes, assert rejection.
- **Status:** UNADDRESSED

### `KER-18` — reduce() is O(ledger length) per call and is invoked on every event append (full re-fold), so for long-running runs each step scans the entire history.
- **Consequence:** Quadratic blowup; long agent runs grind to a halt or time out; the 'ledger is the runtime' becomes unusably slow at scale.
- **Guard:** Kernel supports incremental fold with verified snapshots: reduce(snapshot, newEvents) must equal reduce(genesis, allEvents); snapshot is content-addressed and re-derivable. Benchmark/regression test asserting per-step cost is O(new events) and that snapshot fold == full fold on a 100k-event ledger.
- **Status:** UNADDRESSED

### `KER-19` — Two effect-runner instances both poll the ledger for admitted-but-unexecuted effects and both pick up the same EffectRequested because there is no lease/claim, only a status read.
- **Consequence:** Horizontal scaling of the runner causes duplicate execution; idempotency at the key level may catch it but only after both have side-effected if KER-08 is also weak.
- **Guard:** Claiming an effect requires an atomic lease (SELECT ... FOR UPDATE SKIP LOCKED or a lease event with a unique constraint and TTL); lease expiry re-gates rather than auto-reassigns. Test: N runners against M effects, assert each effect claimed by exactly one, and a runner that dies mid-lease causes a HOLD, not a silent reassign.
- **Status:** UNADDRESSED

### `KER-20` — Park-until-event correlation ids are reused after the parked branch resumes (not marked single-use), so a duplicate or replayed inbound event wakes an already-resumed branch.
- **Consequence:** Double-resume executes downstream effects twice from one logical wakeup; replay attacks via re-delivered webhooks/messages.
- **Guard:** Correlation id consumption is recorded as a BranchResumed event with a unique constraint; a second matching inbound event finds the id consumed and is dropped/quarantined, not re-applied. Test: deliver the same wakeup event twice, assert exactly one resume and the duplicate is logged as already-consumed.
- **Status:** UNADDRESSED

### `KER-21` — A parked branch never receives its event (upstream source dies, correlation id typo, channel down) and there is no timeout/dead-letter, so it parks forever holding resources.
- **Consequence:** Silent stuck runs accumulate; users see 'running' indefinitely; eventual resource exhaustion of parked slots.
- **Guard:** park-until-event takes a mandatory deadline; on expiry the kernel emits ParkTimeout and re-gates (resume on timeout branch or HALT), never infinite. Test: park with 1s deadline, deliver nothing, assert ParkTimeout event and the run leaves the parked state deterministically.
- **Status:** UNADDRESSED

### `KER-22` — The kernel emits 'proposed events' but the boundary between proposal and commit is fuzzy: the effect-runner reads proposals and executes before the proposed events are durably committed and signed in the ledger.
- **Consequence:** An effect executes for a proposal that was never committed (e.g. committer rejected it); ledger no longer contains the cause of an observed effect; 'what you see is what executed' breaks in reverse (effect with no recorded cause).
- **Guard:** Strict ordering: propose -> durably append signed AdmissionDecision -> only then is the effect eligible for claim. The runner queries committed admissions only, never the kernel's in-memory proposals. Test: kill between propose and commit, assert no effect is claimable for the uncommitted proposal.
- **Status:** UNADDRESSED

### `KER-23` — Event signatures cover the payload but not the causal links (parent event hashes / sequence), so an attacker can re-parent a validly-signed EffectResult onto a different branch or reorder events.
- **Consequence:** Forged causal history; an expensive effect's result is re-attributed to a cheaper branch; replay reconstructs a false DAG.
- **Guard:** Each event's signed body MUST include the hash(es) of its causal parents and its branch-scoped position; the fold verifies the hash chain, rejecting any event whose declared parent isn't already in the verified prefix. Test: take two valid events, swap their parent pointers, assert signature/chain verification fails.
- **Status:** UNADDRESSED

### `KER-24` — Reserve-then-settle: the kernel reserves an estimated cost before EffectRequested but never reconciles to the supervisor-reported actual cost, or reconciles in the wrong direction (releases reserve without settling).
- **Consequence:** Spend caps drift from reality; either runs are throttled on phantom reserves or real spend silently exceeds the cap.
- **Guard:** Every EffectRequested with a reserve MUST be followed by either an EffectResult that settles (reserve replaced by actual) or a release event; a fold invariant asserts sum(reserves outstanding) + sum(settled) reconciles, and any reserve older than its effect's terminal state is flagged. Test: reserve 10, actual 7, assert ledger shows settle=7 release=3 and cap freed by 3; reserve with no terminal event flagged by reconciler.
- **Status:** UNADDRESSED

### `KER-25` — The canonical serialization used for hashing/signing events is not stable: JSON.stringify with default key order, NaN/Infinity, -0 vs 0, or unicode normalization differences produce different bytes for equal events.
- **Consequence:** Content addresses differ across nodes/versions; replay verification fails on equal data; idempotency keys (KER-07) and chain verification (KER-23) all inherit the instability.
- **Guard:** Mandate one canonical encoding (e.g. RFC 8785 JCS or deterministic CBOR) used everywhere a hash is taken; ban raw JSON.stringify in the ledger/kernel packages via lint; conformance vectors test (NaN rejected, key order normalized, -0 normalized, NFC unicode). Cross-runtime test hashes the same struct on two Node versions and asserts equality.
- **Status:** UNADDRESSED

### `KER-26` — A bounded loop edge (CEL back-edge) has its termination condition depend on state that an effect inside the loop fails to update on the HALT/retry path, so the loop condition can never flip.
- **Consequence:** Infinite loop emitting effects each iteration (cost runaway) or a hung run.
- **Guard:** Every loop construct carries a kernel-enforced max-iteration budget recorded in the ledger independent of the CEL condition; exceeding it HALTs with LoopBudgetExceeded. Test: loop whose CEL condition is constantly true, assert it terminates at the budget, and assert the iteration counter is incremented purely by the kernel, not by effect output.
- **Status:** UNADDRESSED

### `KER-27` — The kernel admits an effect whose inputs were tainted by untrusted-inbound provenance (per memory plane) into a spend/irreversible decision, because provenance tags are dropped when state flows through reduce()/fan-out.
- **Consequence:** Prompt-injection-laundered data triggers money movement or irreversible actions; the untrusted-provenance bar is defeated by the kernel itself.
- **Guard:** Provenance taint is a monotone label that propagates through every reduce transform and join (output taint = union of input taints); Admission for any irreversible/spend effect rejects if any input carries untrusted-inbound taint without an intervening trusted-approval event. Test: feed untrusted-tagged value into a transfer effect's amount, assert admission denied; test taint survives a fan-out/fan-in round trip.
- **Status:** UNADDRESSED

### `KER-28` — determinism tag on EffectResult is informational only — the kernel doesn't use it, so a non-deterministic effect (LLM call, time-dependent API) is replayed in verification mode by re-execution instead of by replaying the captured result.
- **Consequence:** Verification replay produces different output and is falsely reported as 'divergence'/'tamper'; or counterfactual replay silently re-incurs cost it claimed it wouldn't.
- **Guard:** reduce()/replay branches on the determinism tag: non-deterministic effects MUST replay from the captured EffectResult (result-set by hash, never re-called) in verification mode, and re-gate (not silently re-run) in counterfactual mode. Test: verification replay of a run containing an LLM call makes zero outbound calls and reproduces the recorded output to the byte; assert cost reconciles to the cent.
- **Status:** UNADDRESSED

### `KER-29` — AdmissionDecision is computed by the kernel but the policy inputs (grants, caps, provenance, assurance level) are read at execution time from mutable current state rather than snapshotted into the signed decision.
- **Consequence:** Replay can't reproduce why an effect was admitted (policy changed since); audit can't prove the decision was correct at the time; counterfactual replay uses today's policy on yesterday's run.
- **Guard:** AdmissionDecision event embeds the full content-addressed snapshot of every policy input it consulted (or their hashes) so the decision is self-contained and re-checkable. Test: change a grant after a run, replay-verify the run, assert admission re-derives identically from the embedded snapshot, not from current grants.
- **Status:** UNADDRESSED

### `KER-30` — The kernel allows an EffectRequested to be retried with mutated args under the SAME idempotency key (e.g. a corrected amount), so the dedup layer returns the stale first result or executes with ambiguity.
- **Consequence:** Either a corrected effect silently doesn't happen (stale dedup) or the key/args mismatch is undetected; correctness of retries is undefined.
- **Guard:** Idempotency key is bound to a hash of its args (KER-07); a second EffectRequested with the same key but different arg-hash is a hard error (IdempotencyKeyConflict), forcing a new key for changed args. Test: same key + different args asserts conflict raised, never silent reuse.
- **Status:** UNADDRESSED

### `KER-31` — Kernel error handling treats any thrown exception inside reduce()/edge-eval as a benign skip or default route instead of failing closed, so a bug in one edge silently disables a guard.
- **Consequence:** Security/spend guards become no-ops on internal error; failures hide instead of halting.
- **Guard:** reduce() and the evaluator are total functions returning Result types; any internal exception is converted to a KernelFault event that HALTs the branch and re-gates; CI test injects throws into each evaluator op and asserts HALT, never default-allow. No try/catch that swallows to a permissive default in kernel code (lint guard).
- **Status:** UNADDRESSED

### `KER-32` — Branch-scoped state is shared by reference across fan-out branches (shallow copy), so one branch's mutation of a nested object is visible to siblings, breaking branch isolation.
- **Consequence:** Cross-branch contamination; one branch's effect inputs corrupted by a sibling; non-reproducible, order-dependent results.
- **Guard:** Fan-out gives each branch a structurally-isolated (deep-cloned or persistent-immutable) view of state; a property test mutates a deeply nested field in one branch's view and asserts siblings and parent are unchanged; the kernel never hands out a mutable shared reference.
- **Status:** UNADDRESSED

<a name="durability"></a>
## Durability, Suspension & Resume

### `DUR-01` — A node crashes AFTER EffectRequested is appended (idempotency key minted) but BEFORE EffectResult is appended. On restart the reducer cannot tell whether the side effect actually fired with the external system.
- **Consequence:** Either the effect is silently re-executed (double charge / double email) or silently skipped (lost work), violating 'never silently re-executed'.
- **Guard:** Test: kill the effect-runner process between EffectRequested and EffectResult for an effect with a non-deterministic side-effect class. Assert the reducer transitions that fact into a HALT/re-gate state and refuses to auto-replay; require an explicit reconciliation event before resume.
- **Status:** GUARDED — kill-and-resume through the real engine: no double-execution (test + demo:resume)

### `DUR-02` — The idempotency key for EffectRequested is derived from wall-clock time or a random UUID generated at request time rather than from content (manifest node id + branch-scoped correlation id + input hash).
- **Consequence:** On crash-resume the regenerated request gets a NEW key, so the downstream/external dedup layer treats it as a fresh effect and executes twice.
- **Guard:** Design rule + test: idempotency key MUST be a pure function of (content-addressed input). Property test: derive the key twice from identical reducer state in two separate processes; assert byte-equality. Reject any key derivation that reads Date.now() or crypto.randomUUID().
- **Status:** GUARDED — resume folds the log; effect with existing result is re-served, not re-run

### `DUR-03` — A park-until-event correlation id is reused across two different parked branches (or recycled after completion), so a single inbound event resumes the wrong branch or fans out to multiple branches.
- **Consequence:** Cross-branch resume corruption; a webhook meant for run A resumes run B; or one event double-fires two waiters.
- **Guard:** Design rule: correlation ids are single-use and branch-scoped (run id + branch path + monotonic counter). Test: inject the same external event twice and after a branch already consumed its correlation id; assert second delivery is rejected as already-consumed and never resumes any branch.
- **Status:** GUARDED — on-disk EffectResult survives restart; re-served on resume (durability test)

### `DUR-04` — park-until-event has no upper time bound; a branch waits for an event that never arrives (counterparty died, webhook misconfigured).
- **Consequence:** Run hangs forever, holding reserved budget and tenant resources; appears 'running' indefinitely in monitor.
- **Guard:** Design rule: every park MUST carry a deadline/TTL that itself is an event source (timeout fires a TimedOut event through the same correlation machinery). Test: park with 100ms TTL, deliver nothing; assert a deterministic TimedOut event is appended and the branch resumes the timeout edge.
- **Status:** UNADDRESSED

### `DUR-05` — The wake-up timer for a multi-day wait is held only in process memory (setTimeout) and is lost when the process restarts during the wait window.
- **Consequence:** A 3-day park silently never wakes; the run stalls permanently with no error.
- **Guard:** Design rule: parks are durable rows/events in Postgres with a next-wake-at column; a recovery scan on startup re-arms all due/overdue timers. Test: park for T, kill process, restart after T elapsed; assert the timeout event fires immediately on startup (catch-up), not never.
- **Status:** UNADDRESSED

### `DUR-06` — On restart the recovery scan re-arms timers but does not de-duplicate against parks that already fired their event just before the crash (event appended, branch not yet advanced).
- **Consequence:** The branch's resume edge executes twice (timeout + already-delivered event), double-running downstream effects.
- **Guard:** Test: append the resume event, crash before advancing the reducer, restart. Assert the reducer treats the park as already-satisfied (consumes the existing event) and does NOT also fire the timeout, and resumes the branch exactly once.
- **Status:** UNADDRESSED

### `DUR-07` — Replay (verification mode) of a run that parked and resumed re-evaluates the wait condition against CURRENT clock instead of the notarized clock value captured at original park/resume time.
- **Consequence:** Replay diverges from the original execution (a 'wait until 9am' edge takes a different branch), breaking 'what you see is exactly what executed' and cost reconciliation.
- **Guard:** Design rule: the clock reading that satisfied a park is a captured event with a notarized timestamp; replay re-feeds the captured value. Test: replay a time-gated run hours later; assert identical branch selection and zero divergence vs original ledger fold.
- **Status:** UNADDRESSED

### `DUR-08` — Resume after kill re-runs a deterministic pure kernel step but re-reads a non-deterministic input (LLM call, retrieval result-set, random) that was supposed to be captured, because the capture write was not atomic with the step that consumed it.
- **Consequence:** Replay/resume produces a different manifest projection; the ledger no longer reproduces; determinism tag is a lie.
- **Guard:** Design rule: every non-deterministic effect result is captured-by-hash and committed in the SAME transaction boundary as the fact that consumes it (or fact is HALTED). Test: kill between LLM response and its capture-append; assert resume does NOT re-call the LLM but HALTS for reconciliation.
- **Status:** UNADDRESSED

### `DUR-09` — EffectResult supervisor co-signature is computed over the effect output but NOT over the correlation id / idempotency key, so a replayed or duplicated request with the same output but different intended target validates.
- **Consequence:** A captured result can be replayed to satisfy a different branch's effect, laundering a stale result into a new decision.
- **Guard:** Design rule: supervisor signature payload MUST bind {idempotency key, correlation id, run id, manifest node, output hash, real cost, determinism tag}. Test: take a valid EffectResult, swap its correlation id, assert signature verification fails.
- **Status:** UNADDRESSED

### `DUR-10` — Two effect-runner workers (HA / horizontal scale) both pick up the same EffectRequested because there is no lease/lock on the request.
- **Consequence:** Concurrent double execution of the same side effect across workers.
- **Guard:** Design rule: EffectRequested processing requires a single-holder lease keyed on the idempotency key (SELECT ... FOR UPDATE SKIP LOCKED or advisory lock). Test: start 2 workers, enqueue one request, assert exactly one EffectResult and exactly one external call (mock counts calls).
- **Status:** UNADDRESSED

### `DUR-11` — Idempotency dedup is enforced only inside Krelvan's ledger, but the external system (Stripe, email provider) is not passed the idempotency key, so a Krelvan-internal retry after a network timeout double-charges externally.
- **Consequence:** Real-money double effects despite a clean internal ledger.
- **Guard:** Design rule: the broker/plugin contract requires forwarding the Krelvan idempotency key as the external Idempotency-Key header where supported; effect profile marks providers that cannot dedup as 'at-most-once-must-halt'. Test: simulate provider 200-after-timeout; assert no second call for dedup-capable providers and HALT for non-capable.
- **Status:** UNADDRESSED

### `DUR-12` — A parked branch's reserved budget is released on park (to free quota) but on resume the reserve is not re-acquired, or is re-acquired at a now-exhausted budget.
- **Consequence:** Resume proceeds without budget guarantee and overspends, or fails non-deterministically depending on concurrent runs.
- **Guard:** Design rule: reserve persists across park (reserve-then-settle spans the suspension) OR resume re-gates the reserve as an explicit admission decision. Test: park with reservation, exhaust budget elsewhere, resume; assert resume re-gates and is denied/parked, never silently overspends.
- **Status:** UNADDRESSED

### `DUR-13` — The ledger is append-only but the SUSPENSION INDEX (which branch waits on which correlation id) is a mutable derived table that drifts from the ledger after a partial write.
- **Consequence:** Inbound event arrives, index says no waiter, event dropped; or index says waiter that ledger already resumed.
- **Guard:** Design rule: suspension index is a pure fold of the ledger and is rebuildable; on startup it is reconstructed and checksummed against ledger head. Test: corrupt/delete the index table, restart, assert it is fully rebuilt and resume routing is identical.
- **Status:** UNADDRESSED

### `DUR-14` — Event delivery to a parked branch is at-most-once (fire-and-forget pub/sub) with no durable inbox, so an event arriving while the runtime is down is lost.
- **Consequence:** Branch never resumes; permanent stall for events that occurred during a deploy/restart window.
- **Guard:** Design rule: inbound resume events are durably persisted (an EventReceived ledger entry) before ack to the source; matching to a park is a separate fold step. Test: deliver event while runtime down (write to durable inbox), start runtime, assert branch resumes from the buffered event.
- **Status:** UNADDRESSED

### `DUR-15` — Same inbound event delivered twice by an at-least-once source (webhook retry) both get matched and both resume the branch because consumption isn't transactional with matching.
- **Consequence:** Double resume -> double downstream effects.
- **Guard:** Design rule: matching+consume of a correlation id is a single atomic transaction that marks the correlation id consumed; duplicates hit a unique constraint. Test: deliver identical webhook twice concurrently; assert exactly one resume, one unique-violation swallowed as idempotent.
- **Status:** UNADDRESSED

### `DUR-16` — Crash during the multi-step resume (event consumed -> several kernel reductions -> new EffectRequested) leaves the run in a state where the event is consumed but no progress event is durable.
- **Consequence:** On restart the event is gone (consumed) but the branch hasn't advanced, so it parks again waiting for an event that will never re-arrive: deadlock.
- **Guard:** Design rule: consuming a correlation id and appending the first resulting reduction fact are in one transaction; if it can't complete, the consume rolls back. Test: kill mid-resume-transaction; assert either fully resumed or fully un-consumed (still parked), never the consumed-but-stalled state.
- **Status:** UNADDRESSED

### `DUR-17` — Correlation ids are guessable/sequential and a park accepts any event presenting a valid-looking id without authenticating the sender.
- **Consequence:** An attacker forges a resume event to wake a branch early or inject a chosen payload into a spend/irreversible decision.
- **Guard:** Design rule: correlation ids are high-entropy single-use capabilities AND the resume event carries the source principal's assurance; untrusted-inbound provenance tag bars laundered facts from spend. Test: forge a resume with a guessed id and untrusted source; assert it is rejected (or tagged untrusted and barred from irreversible edges).
- **Status:** UNADDRESSED

### `DUR-18` — Kill-and-resume restores reducer state but not the effect-runner's in-flight external connection state (e.g., a long-running HTTP request / streaming tool), so resume assumes the call never started.
- **Consequence:** The long effect is re-issued while the original may still be completing externally: double execution with a race.
- **Guard:** Design rule: any effect that can outlive the process must register EffectRequested durably with a 'check-status' reconciliation procedure in its effect profile before issuing. Test: kill during a long effect, restart; assert resume calls the provider's status/get endpoint keyed by idempotency key BEFORE re-issuing.
- **Status:** UNADDRESSED

### `DUR-19` — Replay determinism breaks because the reducer iterates over a Map/Set/object-key order that is not stable across Node versions or insertion histories.
- **Consequence:** Two replays produce different event ordering -> different content hashes -> ledger fails to verify against itself.
- **Guard:** Design rule: all reducer collection iteration uses explicitly sorted keys; no reliance on insertion order for hashing. Property test: shuffle insertion order of equivalent inputs; assert identical output content hash.
- **Status:** UNADDRESSED

### `DUR-20` — Content-addressed event hashing includes a serialization that is not canonical (floating-point formatting, unicode normalization, key ordering, undefined-vs-missing), so the same logical event hashes differently across platforms.
- **Consequence:** Resume on a different host or after a dependency bump cannot match prior hashes; replay verification fails spuriously and HALTs healthy runs.
- **Guard:** Design rule: one canonical serialization (e.g., deterministic CBOR/JCS) with pinned number handling. Test: hash the same event on macOS/Linux and on two Node majors; assert identical bytes. Round-trip property test for floats, big ints, unicode.
- **Status:** UNADDRESSED

### `DUR-21` — A wait condition (CEL/JSONLogic) referenced in a park predicate is re-compiled at resume from a manifest version that has since been edited, so the resume edge differs from what was parked under.
- **Consequence:** Branch resumes into a different logical path than it suspended from; silent behavior change across a multi-day wait spanning a deploy.
- **Guard:** Design rule: a park captures the content-addressed manifest version it suspended under; resume evaluates against THAT pinned version. Test: park, edit+redeploy manifest, deliver resume; assert evaluation uses pinned manifest hash and divergence is flagged, not silently applied.
- **Status:** UNADDRESSED

### `DUR-22` — Catch-up after long downtime fires thousands of overdue timeouts/parks simultaneously (thundering herd) and the recovery scan re-gates them all at once.
- **Consequence:** Restart storm overwhelms effect-runner/providers, triggers rate limits, and some reconciliations fail/half-run.
- **Guard:** Design rule: recovery dispatch is rate-limited and ordered by deadline with bounded concurrency; idempotency keys make retries safe. Test: seed 5000 overdue parks, restart; assert bounded concurrent dispatch and zero double-execution under the storm.
- **Status:** UNADDRESSED

### `DUR-23` — The 'crash hole HALT and re-gate' rule is implemented, but re-gating uses the ledger head's CURRENT authority/grants rather than the authority captured at the original AdmissionDecision.
- **Consequence:** A resumed effect is admitted under widened (or narrowed) grants that did not apply when it was decided, violating capability monotonicity across suspension.
- **Guard:** Design rule: AdmissionDecision binds the effective grant set by hash; re-gate after crash must re-prove against the captured grant or explicitly require fresh human/eval approval. Test: widen grants during a park, resume; assert the resumed effect does NOT inherit the widened grant automatically.
- **Status:** UNADDRESSED

### `DUR-24` — Determinism tag on a captured effect says 'deterministic' but resume re-executes it (treating it as cheap to redo) and the underlying tool was actually nondeterministic/time-dependent (e.g., 'current FX rate').
- **Consequence:** Resume value differs from captured value; downstream decisions diverge from the ledger; cost reconciliation is off.
- **Guard:** Design rule: determinism tag is asserted by the effect profile (single source of truth), and supervisor-observed; any tag of 'deterministic' is spot-checked by re-running and diffing in test. Test: a tool mislabeled deterministic but returning varying output -> assert profile validation/CI catches the mismatch and bars re-execution-on-resume.
- **Status:** UNADDRESSED

### `DUR-25` — A branch parks, and concurrently a counterfactual replay forks that run and RE-GATES its effects; the fork and the live parked branch share the same correlation id namespace.
- **Consequence:** A real inbound event resumes the counterfactual fork (or vice versa); replay leaks side effects into production despite 'verification = zero side effects'.
- **Guard:** Design rule: every replay/fork gets an isolated run id and correlation namespace; verification mode physically cannot subscribe to live inbound channels. Test: run a counterfactual fork of a parked run, deliver the real event; assert only the live branch resumes and the fork performs zero external effects.
- **Status:** UNADDRESSED

### `DUR-26` — Postgres connection/transaction for the resume commit succeeds but the subsequent ledger-signature/notarization step is in a separate transaction and crashes between them.
- **Consequence:** An unsigned/un-notarized fact exists in the ledger; verification later rejects it and HALTs the whole run, or worse, it's accepted unsigned.
- **Guard:** Design rule: a fact's append and its signature are committed atomically (signature is part of the row, written in the same tx); no fact row exists without a valid signature. Test: kill between append and sign; assert no half-signed rows are visible (constraint: signature NOT NULL + verify on read).
- **Status:** UNADDRESSED

### `DUR-27` — Multi-day wait spans a key rotation; the resume event / re-gate is signed under a new key while the park was bound to the old key, and verification of the whole run chain fails at the rotation boundary.
- **Consequence:** Legitimately resumed runs fail replay verification after every key rotation; operators learn to ignore verification failures.
- **Guard:** Design rule: rotation appends a notarized key-transition event; verifiers accept signatures valid under the key active at each fact's timestamp. Test: park, rotate signing key, resume+sign with new key; assert full-run verification passes across the rotation boundary.
- **Status:** UNADDRESSED

### `DUR-28` — The single-use correlation id is marked consumed in Redis (fast) but the durable ledger consume lags; a crash loses the Redis flag and the durable consume, but the resume side-effect already fired.
- **Consequence:** On restart the id appears un-consumed, branch resumes again, effect re-executes — exactly the double-execution the model forbids.
- **Guard:** Design rule: consumed-state lives in the durable ledger as the source of truth (Redis is only a cache derived from it); the side effect is never issued before the durable consume+EffectRequested commit. Test: kill after side-effect-issue but before durable consume with Redis-only flag; assert restart HALTs (crash hole) rather than re-resuming.
- **Status:** UNADDRESSED

### `DUR-29` — A node's park predicate depends on data that is mutated in working memory after parking (working-plane state is not snapshotted at suspension).
- **Consequence:** Resume reads a mutated working-memory value, evaluating the resume edge against state that never existed at suspend time; non-reproducible.
- **Guard:** Design rule: park snapshots (by hash) all working-plane inputs its resume depends on; resume re-feeds the snapshot. Test: park, mutate working memory, resume; assert resume uses the snapshot hash and result matches a fresh replay.
- **Status:** UNADDRESSED

### `DUR-30` — Resume ordering is wrong when one inbound event satisfies multiple parked branches that have a defined causal order (e.g., must resume parent before child) — the runtime resumes them in arrival/parallel order.
- **Consequence:** Out-of-order resume produces a different fold than the manifest's intended dependency order; race-dependent results.
- **Guard:** Design rule: resumes are appended as events and the reducer applies them in deterministic ledger order respecting branch dependency edges, not dispatch order. Test: one event waking parent+child branches; assert deterministic resume order across repeated runs and that replay reproduces it.
- **Status:** UNADDRESSED

### `DUR-31` — Idempotency keys are unique per (run, node) but a manifest loop visits the same node many times; keys collide across iterations.
- **Consequence:** The 2nd loop iteration's effect is deduped against the 1st and silently skipped (or the 1st result is replayed), corrupting loop semantics.
- **Guard:** Design rule: idempotency key includes the loop/iteration counter and full branch path, not just node id. Test: a loop calling the same side-effecting tool N times; assert N distinct keys and N external calls, and that replay reproduces exactly N.
- **Status:** UNADDRESSED

### `DUR-32` — On resume after a very long wait, the captured retrieval/LLM result-sets have been garbage-collected/evicted from the content-addressed store (retention shorter than max park duration).
- **Consequence:** Replay/verification cannot re-feed captured non-deterministic results; the run becomes unverifiable or HALTs incorrectly.
- **Guard:** Design rule: content-store retention for captured effects is >= max allowed park TTL and is pinned by any live park referencing the hash (refcount). Test: park beyond default GC window with a referenced capture; assert the capture is not collected and verification still passes.
- **Status:** UNADDRESSED

<a name="capability"></a>
## Capability Plane / Tools / MCP / Sandbox / Secrets

### `CAP-01` — A plugin's effect profile (authority/side-effect-class/determinism) is declared in the plugin manifest AND separately in the kernel registry, and the two drift apart over versions, so the kernel gates against a stale profile while the plugin actually performs a higher-authority effect.
- **Consequence:** A tool the operator believes is read-only silently performs writes/spends; deny-by-default is bypassed because the gate consulted the wrong profile.
- **Guard:** Enforce SINGLE source of truth: kernel computes the effect profile from a content-hash of the plugin's signed descriptor and REFUSES to load if any second copy disagrees. Test: load a plugin whose embedded profile != registry profile and assert load is rejected with a profile-mismatch error.
- **Status:** GUARDED — deny-by-default admission; ungranted capability denied (test)

### `CAP-02` — The secrets broker mints a scoped short-lived token but the plugin caches/logs it (writes to its stdout, temp file, or returns it in EffectResult), so the token outlives its scope.
- **Consequence:** Token exfiltration; an attacker reading ledger or logs reuses a still-valid credential.
- **Guard:** Broker tokens are single-use, bound to one EffectRequested idempotency key, and expire in <=60s. Test: capture all plugin stdout/stderr/return payloads and ledger events for a run and assert no substring matches the minted token; assert reuse of a consumed token is rejected.
- **Status:** GUARDED — reserve-then-settle budget; run+per-cap ceilings deny before overspend (test)

### `CAP-03` — SECRETS NEVER TOUCH PLUGINS is violated because a plugin needs to call a downstream API and the broker hands it the raw API key instead of proxying the call, on the assumption the plugin is trusted.
- **Consequence:** Plugin compromise = secret compromise; the core invariant fails.
- **Guard:** Broker NEVER returns raw long-lived secrets to plugins. Either it proxies the egress call (secret injected at the egress boundary outside the sandbox) or mints a downstream-scoped ephemeral token. Test: grep the broker API surface for any path returning a secret value to a plugin; assert none exists. Integration test: plugin requests 'the key' and receives only a proxy handle.
- **Status:** GUARDED — supervisor signs EffectResult; plugin self-report kept as untrusted claim (e2e shows author=supervisor)

### `CAP-04` — A plugin self-signs an EffectResult asserting 'no side effect occurred / cost was $0 / output is deterministic' and the supervisor accepts it without independently observing the effect.
- **Consequence:** A malicious or buggy plugin lies about its own behavior; cost reconciliation and determinism tags are forged; replay verification passes on false data.
- **Guard:** Plugins MUST NOT sign facts about their own behavior. The supervisor co-signs EffectResult based on MECHANICALLY OBSERVED effects (bytes egressed, syscalls, metered cost from the egress proxy), not plugin assertions. Test: a plugin returns 'cost=0, deterministic=true' while the proxy observed 3 network calls and metered cost; assert supervisor overrides plugin claim and the signed EffectResult carries observed values.
- **Status:** GUARDED — autonomy gradient (suggest/act-with-veto/full) gates side effects; parks for approval (test)

### `CAP-05` — Deny-by-default sandbox allows outbound DNS (or a metadata-service IP like 169.254.169.254) because only HTTP egress was gated, letting a plugin exfiltrate via DNS tunneling or steal cloud instance credentials.
- **Consequence:** Data exfiltration / cloud credential theft despite 'deny-by-default'.
- **Guard:** Egress allowlist is L3/L4 (IP+port) AND L7 (host), DNS resolution goes through a controlled resolver, and link-local/metadata ranges are hard-blocked. Test: from inside the sandbox attempt DNS to an unlisted name, a raw connect to 169.254.169.254, and a connect to an unlisted IP; assert all three are blocked and logged as denied EffectRequested.
- **Status:** UNADDRESSED

### `CAP-06` — An MCP plugin is updated upstream (npm/registry) and the new version is auto-pulled at runtime, changing behavior/effect profile without re-signing or operator approval — a classic supply-chain rug-pull.
- **Consequence:** Trusted tool becomes malicious between runs; the manifest the user 'owns' silently changes meaning.
- **Guard:** Plugins are pinned by content-address (hash of the exact artifact), not by version tag/range. Any new hash requires a fresh signature + explicit operator admission. Test: change one byte of a pinned plugin artifact and assert the kernel refuses to load it (hash mismatch) until re-admitted.
- **Status:** UNADDRESSED

### `CAP-07` — A plugin's signing key is valid but was revoked after a compromise; the kernel checks signature validity but not revocation status, so revoked plugins still load.
- **Consequence:** Known-compromised plugins keep executing.
- **Guard:** Plugin load checks a revocation list / short-lived signature freshness, not just cryptographic validity. Test: revoke a plugin key, then attempt to load a plugin signed by it; assert load fails with 'key revoked' even though the signature math verifies.
- **Status:** UNADDRESSED

### `CAP-08` — Capability monotonicity is enforced at compile time but a plugin requests ADDITIONAL scopes at RUNTIME (dynamic scope escalation via the broker), and the broker grants them because it trusts the running effect-runner.
- **Consequence:** Untrusted-influenced execution widens its own grants; the monotonicity invariant is bypassed post-compile.
- **Guard:** Broker grants are bounded by the admitted manifest section's authority; runtime requests can only NARROW, never widen. Test: a plugin at runtime requests a scope not present in its compiled grant; assert broker denies and emits a denied AdmissionDecision.
- **Status:** UNADDRESSED

### `CAP-09` — The idempotency key on EffectRequested is derived from inputs that include a timestamp or random nonce, so retried identical effects get different keys and execute twice.
- **Consequence:** Double-spend / duplicate irreversible side effects (e.g., two payments, two emails).
- **Guard:** Idempotency key = deterministic hash of (manifest node id, branch-scoped correlation id, semantic effect args) with NO clock/nonce. Test: issue the same logical effect twice; assert identical idempotency key and that the second EffectRequested is short-circuited to the prior EffectResult.
- **Status:** UNADDRESSED

### `CAP-10` — Crash between EffectRequested (idempotency key written) and EffectResult: on restart the kernel cannot tell if the side effect actually happened, and either re-executes (double effect) or skips (lost effect) instead of halting.
- **Consequence:** Silent double-execution or silent loss of an irreversible effect.
- **Guard:** A request without a matching result is a CRASH HOLE: HALT and re-gate; never auto-resume. Test: kill the effect-runner mid-effect, restart, and assert the run enters HALTED state requiring explicit operator/supervisor re-admission for that specific idempotency key — not auto-retry, not auto-skip.
- **Status:** UNADDRESSED

### `CAP-11` — The egress proxy meters cost but a plugin makes a side-channel effect the proxy can't see (writes to a mounted volume, spawns a subprocess that calls home, uses an already-open fd inherited from the parent).
- **Consequence:** Unmetered, unobserved side effects; supervisor co-sign reflects only proxied traffic, audit is incomplete.
- **Guard:** Sandbox grants NO ambient filesystem/network/process inheritance: read-only rootfs, no shared mounts, closed inherited fds, no subprocess spawn unless declared. Test: plugin attempts to write a file, open an inherited fd, and fork+exec; assert each is blocked and that the only observable effect path is the proxy.
- **Status:** UNADDRESSED

### `CAP-12` — Two effect profiles disagree on whether an effect is 'reversible': the manifest treats a tool as reversible (so it skips approval) but the side-effect class is actually irreversible (sends money/email).
- **Consequence:** Irreversible action taken without the required step-up/approval.
- **Guard:** Side-effect class is taken ONLY from the single signed effect-profile source; the manifest cannot override it downward. Compiled eval gate: any node invoking an irreversible-class effect MUST have an approval/authorization edge or fail compilation. Test: author a manifest sending email with no approval edge; assert compile fails.
- **Status:** UNADDRESSED

### `CAP-13` — Untrusted-inbound provenance tag is set on data but lost when the data passes through a plugin (the plugin returns a new value with no provenance), so laundered data then flows into a spend decision.
- **Consequence:** Prompt-injection / untrusted content drives an irreversible/spend effect — the laundering bar fails.
- **Guard:** Provenance is a non-droppable taint that propagates through every plugin transform (outputs derived from tainted inputs inherit the taint). Test: feed untrusted-tagged text through a transform plugin, then route output to a spend effect; assert the spend is barred and a provenance-violation event is emitted.
- **Status:** UNADDRESSED

### `CAP-14` — A plugin declares determinism='deterministic' but actually calls a clock/RNG/network, so replay-verification mode (which expects to reproduce output) diverges or worse, re-hits the network.
- **Consequence:** Replay verification falsely fails or causes real side effects during a 'zero side effect' replay.
- **Guard:** Determinism tag is supervisor-observed, not plugin-declared: if the proxy observed any network/clock read, the effect is tagged non-deterministic and its OUTPUT is captured-by-hash and re-fed on replay. Test: a plugin claiming determinism reads the clock; assert it is re-tagged non-deterministic and that verification replay re-feeds the captured output with zero outbound calls.
- **Status:** UNADDRESSED

### `CAP-15` — The sandbox time source is the host wall clock, so a plugin (or attacker) can read/relies-on real time, and counterfactual replay can't pin time, producing non-reproducible results.
- **Consequence:** Non-reproducible runs; attacker clock manipulation; replay drift.
- **Guard:** Sandbox sees only the monotonic notarized clock value injected per-effect and captured in the ledger. Test: plugin reads 'now' twice in one effect; assert both reads equal the single notarized value from the ledger and that replay yields the identical value.
- **Status:** UNADDRESSED

### `CAP-16` — MCP transport (stdio/SSE) lets a plugin send unsolicited messages or oversized responses that the kernel parses into trusted state, enabling injection or resource exhaustion.
- **Consequence:** Plugin-controlled data is treated as kernel facts; DoS via huge payloads.
- **Guard:** All plugin->kernel messages are untrusted: schema-validated, size-capped, and correlated to an outstanding request id; unsolicited messages dropped. Test: plugin emits a 1GB response and an uncorrelated message; assert both are rejected and the effect fails cleanly without OOM.
- **Status:** UNADDRESSED

### `CAP-17` — Two concurrent runs share the same plugin process instance (for performance) and one run's secrets/state leak into another via plugin-global memory.
- **Consequence:** Cross-tenant / cross-run secret and data leakage.
- **Guard:** One sandbox instance per effect invocation (or per branch-scoped correlation id) with no shared mutable state; RLS tenant boundary holds at the sandbox layer too. Test: run two effects in parallel with distinct tokens; assert plugin in run B cannot read any artifact/token belonging to run A even if processes overlap in time.
- **Status:** UNADDRESSED

### `CAP-18` — The supervisor co-signs the EffectResult but uses the SAME key as the plugin's invocation context or a key the effect-runner can access, so a compromised runner forges supervisor signatures.
- **Consequence:** Forged 'mechanically observed' effects; the co-sign trust anchor collapses.
- **Guard:** Supervisor signing key lives in a separate trust domain (separate process/enclave) the sandbox and effect-runner cannot read. Test: attempt to access the supervisor key from the effect-runner process; assert it is unreachable, and that EffectResult signatures verify only against the supervisor's distinct public key.
- **Status:** UNADDRESSED

### `CAP-19` — Effect profile has only 3 axes but a tool combines low-authority + observable-side-effect that together are dangerous (e.g., 'read your contacts' + 'send to allowlisted host' where the host is attacker-controlled-but-allowlisted), and gating treats axes independently.
- **Consequence:** Composition of individually-permitted capabilities yields an impermissible exfiltration path.
- **Guard:** Gate on the COMBINATION (read-of-sensitive + egress) not each axis alone; egress of data derived from sensitive reads requires explicit authority. Test: a node reads sensitive memory plane and egresses to an allowlisted host; assert it requires a distinct combined-capability grant, not just the two separate grants.
- **Status:** UNADDRESSED

### `CAP-20` — Plugin admission UI shows the operator a friendly name/description controlled by the plugin author, which lies about what scopes it requests (UI spoofing / consent phishing).
- **Consequence:** Operator approves a payment-capable plugin believing it only reads weather.
- **Guard:** Admission prompt renders the COMPILED effect profile and requested scopes from the signed descriptor, never author-supplied prose, and highlights irreversible/spend authority distinctly. Test: a plugin with description 'read-only' that requests spend authority; assert the admission screen surfaces 'SPEND' from the profile regardless of the description string.
- **Status:** UNADDRESSED

### `CAP-21` — A plugin times out / hangs holding a minted token and an open EffectRequested; the kernel reclaims the slot but the token isn't revoked, so the hung plugin later completes its side effect.
- **Consequence:** Late 'zombie' effect executes after the kernel believed the effect abandoned.
- **Guard:** On timeout the broker REVOKES the token and the effect transitions to a halted/abandoned state requiring re-gate; any late result with a revoked token is rejected. Test: stall a plugin past timeout, then have it attempt the downstream call; assert the call fails (token revoked) and any late EffectResult is rejected as orphaned.
- **Status:** UNADDRESSED

### `CAP-22` — Egress allowlist entries are by hostname and resolved at admission time; an attacker DNS-rebinds the host to an internal IP at execution time (TOCTOU).
- **Consequence:** SSRF into internal services despite an allowlist.
- **Guard:** Resolve-and-pin: the IP used at admission is pinned for the effect, or re-validate the resolved IP against the L3 allow-range at connect time and block private/link-local. Test: point an allowlisted host's DNS at 127.0.0.1/10.0.0.0/8 between admission and execution; assert the connect is blocked.
- **Status:** UNADDRESSED

### `CAP-23` — Plugin code path executes arbitrary network/file ops during MODULE IMPORT / installation (before any effect is gated), so deny-by-default never applied to install-time code.
- **Consequence:** Malicious side effects at install/load time, outside the gating model.
- **Guard:** Plugins are loaded inside the same deny-by-default sandbox; install/import has no ambient capabilities; any effect at load time must itself be an EffectRequested. Test: a plugin that opens a socket at import time; assert the import runs sandboxed and the socket attempt is denied/logged.
- **Status:** UNADDRESSED

### `CAP-24` — The 3-axis profile is assigned by the plugin author and the kernel trusts it, so an author labels a money-transfer tool as authority='low, read-only' to skip approval gates.
- **Consequence:** Mislabeled dangerous tool bypasses approval; deny-by-default defeated by author dishonesty.
- **Guard:** Effect profile is DERIVED/verified from observed behavior + declared egress allowlist, not solely author-asserted; mismatch between declared profile and first-run observed effects HALTS and demands re-admission. Test: a tool declared read-only performs a write on first run; assert run halts and flags profile escalation requiring operator re-approval.
- **Status:** UNADDRESSED

### `CAP-25` — Branch-scoped correlation ids for park-until-event are guessable/sequential, so one branch can resume another branch's parked effect (single-use id reuse across branches).
- **Consequence:** Cross-branch effect hijack; an attacker resumes a privileged parked effect.
- **Guard:** Correlation ids are unguessable (>=128-bit random), single-use, and bound to one branch + one expected event type; reuse or cross-branch use is rejected. Test: capture branch A's correlation id and attempt to resume from branch B or a second time; assert both rejected.
- **Status:** UNADDRESSED

### `CAP-26` — Cost reserve-then-settle: the broker reserves a budget, the proxy meters real spend, but a plugin makes a call whose cost is only known LATER (async billing), and the settle step trusts a plugin-reported amount.
- **Consequence:** Cost reconciliation to the cent fails; plugin under-reports spend.
- **Guard:** Settlement uses the proxy/provider-observed metered cost, never plugin-reported; if an effect's true cost is unknowable at settle time the effect is barred or runs under a hard reservation cap enforced by the proxy. Test: plugin reports cost=$0 for a metered call; assert settled cost equals proxy-observed amount and that exceeding the reservation cap aborts the call mid-flight.
- **Status:** UNADDRESSED

### `CAP-27` — A signed plugin is genuine but its dependency tree (transitive npm/pip deps) is not content-pinned, so a transitive dep is swapped (dependency confusion / typosquat) without changing the plugin's own signature.
- **Consequence:** Rug-pull via the dependency graph; signature verifies but behavior changes.
- **Guard:** The signed content-address covers the FULL resolved dependency closure (lockfile + vendored artifacts hashed), not just the plugin's top-level files. Test: alter a transitive dependency artifact and assert the plugin's content hash changes and load is rejected until re-admitted.
- **Status:** UNADDRESSED

### `CAP-28` — Supervisor observes egress bytes but cannot attribute them to a specific manifest node when multiple effects run concurrently in one sandbox, so cost/audit lines are mis-attributed.
- **Consequence:** Wrong node blamed for spend/egress; audit and cost-per-node are unreliable.
- **Guard:** Each effect runs with a distinct proxy identity/credential per idempotency key so all observed egress is unambiguously attributable; no shared egress channel across effects. Test: run two concurrent effects through the proxy and assert every metered byte/cost line maps to exactly one idempotency key.
- **Status:** UNADDRESSED

### `CAP-29` — The deny-by-default policy is enforced by the kernel but the sandbox runtime (container/VM) has a known escape or shared kernel namespace, so a plugin breaks out and touches the host directly.
- **Consequence:** Full host compromise bypassing the entire capability plane.
- **Guard:** Use a strong isolation boundary (microVM/gVisor-class), drop all caps, seccomp-filter syscalls, and treat the sandbox as adversarial; pin and scan the runtime for known escapes. Test: run a known container-escape PoC inside the sandbox in CI and assert it fails; assert seccomp denies disallowed syscalls.
- **Status:** UNADDRESSED

### `CAP-30` — A plugin returns output that itself contains forged ledger-event JSON or a fake supervisor signature, hoping the kernel ingests it as a trusted fact.
- **Consequence:** Plugin injects facts about its own behavior into the ledger, defeating the co-sign rule.
- **Guard:** Plugin output is always wrapped as untrusted opaque payload; only the supervisor constructs and signs EffectResult events. The kernel never parses plugin output as an event. Test: a plugin emits a well-formed signed-looking EffectResult; assert it is stored only as opaque payload data and the authoritative EffectResult is the supervisor-signed one.
- **Status:** UNADDRESSED

### `CAP-31` — Token scope is correct but its AUDIENCE/resource binding is too broad (e.g., scoped to 'send email' but to any recipient), so a compromised plugin uses the legitimate token for an unintended target.
- **Consequence:** In-scope-but-off-target abuse (email to attacker, transfer to attacker account).
- **Guard:** Broker tokens bind the concrete RESOURCE/target from the admitted effect args (recipient, account, path), not just the action class. Test: mint a token for 'email to alice@x', then attempt to send to mallory@y with it; assert the egress proxy rejects the mismatched target.
- **Status:** UNADDRESSED

### `CAP-32` — Plugin trust is established once at admission, but the plugin is later granted access to the SOUL/semantic memory plane and can quietly read long-term secrets/identity stored there.
- **Consequence:** Sensitive memory-plane data leaks to a tool that only needed working-plane scope.
- **Guard:** Memory-plane access is a distinct axis of the grant; tools default to working-plane only and require explicit, separately-approved scope for episodic/semantic/SOUL. Test: a plugin granted only working-plane attempts to read SOUL; assert denied and logged, and that admission of SOUL access is a separate operator decision.
- **Status:** UNADDRESSED

<a name="manifest"></a>
## Manifest & Meta-Agent Compiler

### `CM-01` — Capability-monotonicity check compares granted scopes by string equality, so an untrusted intent that requests 'files:read:/home/*' is rejected as wider than 'files:read:/home/' but 'files:read:/home/user/../../etc' normalizes to a path OUTSIDE the grant yet passes the naive subset check.
- **Consequence:** LLM-driven intent silently widens filesystem authority via path traversal; compiler emits a manifest granting access the principal never authorized.
- **Guard:** Property test: for every (granted, requested) scope pair, canonicalize (resolve .., symlinks, case-fold, percent-decode, unicode-normalize) BEFORE the subset test; assert no requested resource resolves outside the canonical granted set. Fuzz with traversal/encoding payloads.
- **Status:** UNADDRESSED

### `CM-02` — Monotonicity is enforced on the diff between old and new manifest, but a compile from scratch (no prior manifest) has no baseline, so the very first compilation from an untrusted NL prompt can request arbitrary capabilities with nothing to be monotone against.
- **Consequence:** Initial manifest gets full requested authority because 'widening' is undefined when there is no predecessor.
- **Guard:** Design rule + test: the monotonicity baseline for a fresh compile is the EXPLICIT human-granted capability budget for that principal/section, never the empty set. Test that a first-compile requesting any capability not in the principal's authorization budget is rejected.
- **Status:** GUARDED — untrusted principal carries an explicit allowedCapabilities baseline; first compile from untrusted intent can't exceed it (prompt-injection test). Owner principal is the only widening authority.

### `CM-03` — Per-section authority is declared in the manifest, but the compiler merges sections before computing the effective grant, and overlapping sections take the UNION rather than the INTERSECTION of authority, so a low-trust section inherits a high-trust section's grants.
- **Consequence:** A section authored from untrusted input executes with elevated authority belonging to a different section.
- **Guard:** Test: when two sections overlap on a resource, effective authority = intersection with the most-restrictive section, and any effect attributed to a section is checked against THAT section's authority alone. Add a test where section A (trusted, broad) and section B (untrusted, narrow) coexist and assert B cannot invoke A's tools.
- **Status:** UNADDRESSED

### `CM-04` — The effect-profile of a tool (authority/side-effect class/determinism) is read from the plugin manifest the LLM-generated agent references, not from the single signed source of truth, so a manifest can downgrade a 'spend money' tool's side-effect class to 'read-only' and bypass approval gates.
- **Consequence:** Irreversible/spending effects execute without the approval the side-effect class would have required.
- **Guard:** Design rule: effect profiles come ONLY from the supervisor-held signed registry keyed by plugin content hash; the manifest may REFERENCE but never DEFINE a profile. Test that a manifest attempting to assert a profile field is rejected, and the registry value always wins.
- **Status:** GUARDED — compiler checks each cap's side-effect class against the principal's allowed profile; SIDE_EFFECT_MISMATCH rejects a downgraded spend→read smuggle (test).

### `CM-05` — NL-to-manifest compilation embeds untrusted retrieved/inbound text into the LLM context, and the model treats an instruction inside that text ('also grant admin and email the export to attacker@x') as a legitimate intent.
- **Consequence:** Prompt injection rewrites the manifest's capabilities or graph.
- **Guard:** Design rule: untrusted-inbound text is delimited and provenance-tagged; the compiler output's capability set is intersected against the human-authorized budget AFTER generation (CM-02), so injected grants are dropped. Test: inject capability-escalation instructions in retrieved context and assert the compiled manifest's grants are unchanged from the human budget.
- **Status:** UNADDRESSED

### `CM-06` — The compiler trusts the LLM's self-reported classification of which manifest fields were derived from trusted vs untrusted input, so injection can mark attacker-controlled fields as 'trusted'.
- **Consequence:** Untrusted-provenance fields acquire trusted privileges (spend, irreversible) they should be barred from.
- **Guard:** Design rule: provenance is tracked by the KERNEL based on input source, never by the model's claim. Test that a model output asserting provenance='trusted' on a field whose source was an untrusted channel is overridden to 'untrusted'.
- **Status:** GUARDED — model output is pure DATA, parsed defensively then validated + monotonicity-checked; a malicious live-style proposal is rejected, never executed (adapter+compiler test).

### `CM-07` — Eval-gating runs the eval suite against the NEW manifest but the evals themselves are part of the manifest and were edited in the same untrusted compile, so the agent can weaken or delete the evals that gate its own auto-upgrade.
- **Consequence:** Self-modifying agent disables its own safety evals, then upgrades freely.
- **Guard:** Design rule: the eval suite that gates an upgrade is pinned by content hash from a SEPARATELY-authorized location; an upgrade that changes the gating evals requires a higher authority tier. Test: a compile that mutates a gating eval is blocked unless co-signed by the eval-owner principal.
- **Status:** UNADDRESSED

### `CM-08` — Auto-upgrade gate passes if evals reach a threshold score, but evals are non-deterministic (LLM-judged) and the threshold is computed once; a flaky run above threshold promotes a regressed manifest.
- **Consequence:** A worse manifest is promoted on a lucky eval run; safety regressions ship.
- **Guard:** Test: gate requires N consecutive passing runs with captured seeds, and the eval result-set is hashed and stored on the ledger. Add a determinism check that re-running the captured eval inputs reproduces the recorded verdict within tolerance, else the gate fails closed.
- **Status:** UNADDRESSED

### `CM-09` — Schema migration from manifest v1 to v2 silently drops fields v2 doesn't recognize (e.g. a deny-list or a provenance tag), defaulting the missing field to a permissive value.
- **Consequence:** Migration loosens security posture; a deny rule vanishes and the default-allow takes over.
- **Guard:** Design rule: migrations fail-closed — unknown/removed fields that carry authority must be explicitly mapped or the migration aborts. Test a v1 manifest with a deny-list and assert the v2 result preserves or tightens it, never widens; assert no field defaults to a more-permissive value than its predecessor.
- **Status:** UNADDRESSED

### `CM-10` — Manifest schema version is read from a field INSIDE the manifest, and a malformed/old manifest omits or lies about its version, so the wrong parser/migrator runs and misinterprets authority fields.
- **Consequence:** Authority fields parsed under the wrong schema grant unintended capabilities.
- **Guard:** Test: version must be present, in a signed envelope, and within the supported range; a missing/unknown/forged version fails closed. Round-trip test every supported version through detection -> parse -> re-serialize and assert byte-stable semantics.
- **Status:** UNADDRESSED

### `CM-11` — Manifest signing covers the manifest body but NOT the schema version or the capability budget reference, so an attacker swaps the version/budget while keeping a valid body signature.
- **Consequence:** Signed manifest is reinterpreted under a different schema or budget; signature verification gives false assurance.
- **Guard:** Design rule: the signed payload is the FULL canonicalized envelope (version + body + budget ref + section authority + eval hash). Test that mutating any of those fields invalidates the signature.
- **Status:** GUARDED — signature is over hash(manifest-id + provenance); swapping provenance/budget breaks it (tamper test).

### `CM-12` — Canonicalization before signing is not deterministic (map key ordering, whitespace, float formatting, unicode normalization), so two semantically identical manifests produce different signatures, or two different manifests produce the same canonical form.
- **Consequence:** Either valid manifests fail verification (DoS) or distinct manifests collide (signature reuse / confusion).
- **Guard:** Test: golden-vector canonicalization tests (RFC 8785 JCS or equivalent) — assert canonical(parse(serialize(m))) is byte-identical across platforms; fuzz for two distinct manifests mapping to one canonical form and one manifest mapping to two.
- **Status:** GUARDED — canonicalize() is deterministic (sorted keys, no floats, no NaN); LED-01/02 tests cover it.

### `CM-13` — Signature algorithm/key-id is taken from the manifest itself ('alg' header), enabling algorithm-confusion or 'alg:none' downgrade.
- **Consequence:** Attacker sets alg=none or swaps to a weaker/asymmetric-vs-symmetric confusion and forges a manifest.
- **Guard:** Design rule: verifier pins the allowed algorithm set and resolves the key by trusted key-id from custody, never from the token. Test that alg:none, an unexpected alg, or an unknown key-id is rejected.
- **Status:** UNADDRESSED

### `CM-14` — Compiler accepts a manifest referencing a plugin by name/version rather than content hash, so a later-published malicious plugin with the same name/version is loaded.
- **Consequence:** Supply-chain swap: the effect-runner executes attacker code under the manifest's grants.
- **Guard:** Design rule: every plugin reference is pinned by content hash; the registry lookup is by hash. Test that a name/version-only reference is rejected at compile time and that a hash mismatch at load time halts.
- **Status:** UNADDRESSED

### `CM-15` — Malformed manifest with deeply nested or recursive section references (a section that includes itself) causes the compiler's grant-resolution to recurse/loop unboundedly.
- **Consequence:** Compiler hangs or OOMs (DoS); a partially-resolved manifest may be used.
- **Guard:** Test: feed cyclic and deeply-nested section graphs; assert the resolver detects cycles, bounds depth, and rejects rather than partially compiling. Add fuzz corpus of pathological nestings.
- **Status:** UNADDRESSED

### `CM-16` — Conditional edges in the manifest are compiled to CEL/JSONLogic, but the compiler falls back to eval() or string interpolation for an expression it can't translate.
- **Consequence:** Arbitrary code execution in the kernel via a crafted edge condition; violates the 'NEVER eval' invariant.
- **Guard:** Design rule + test: the only edge-evaluation path is the sandboxed CEL/JSONLogic interpreter; any expression that does not parse under that grammar fails compilation. Grep/lint gate forbidding eval/Function/exec in kernel; test that an untranslatable condition is rejected, never executed.
- **Status:** UNADDRESSED

### `CM-17` — CEL/JSONLogic expressions can reference functions or variables with side effects (e.g. a custom function that performs I/O or reads secrets), turning a 'pure' edge into an effect.
- **Consequence:** Conditional edges leak data or cause effects outside the effect-runner; purity invariant broken.
- **Guard:** Test: the CEL environment exposes ONLY a whitelisted, pure, deterministic function/variable set; assert calling any non-whitelisted symbol fails. Verify the evaluator has no network/fs/clock access (clock comes from notarized time as an injected value).
- **Status:** UNADDRESSED

### `CM-18` — Eval-gating measures task success but has no test for capability MINIMALITY, so the compiler over-grants (requests every capability that might help) and evals still pass.
- **Consequence:** Manifests accumulate excessive standing authority that passes gates; blast radius grows silently.
- **Guard:** Test: a least-privilege eval that runs the agent and asserts every GRANTED capability was actually EXERCISED on the eval suite; unused grants fail the gate or trigger pruning. Track granted-vs-used capability ratio as a regression metric.
- **Status:** UNADDRESSED

### `CM-19` — The compiler is invoked by an untrusted channel and the requesting principal's identity/assurance level is not bound to the compile, so a low-assurance principal compiles a manifest requesting high-authority capabilities.
- **Consequence:** Authority granted exceeds the requester's identity assurance; step-up bypass.
- **Guard:** Test: compile requests carry a bound, verified principal + assurance level; assert that requesting a capability whose required assurance exceeds the principal's level forces step-up or fails. Include a test for a forged/absent principal binding.
- **Status:** UNADDRESSED

### `CM-20` — Manifest validation passes structurally-valid but semantically-contradictory rules (e.g. a section both grants and denies the same scope, or an edge references a node that doesn't exist).
- **Consequence:** Undefined runtime behavior; deny-vs-grant resolved inconsistently, or a dangling edge routes nowhere/everywhere.
- **Guard:** Test: a semantic validator runs after schema validation — assert deny always overrides grant, every edge target exists, every node is reachable or explicitly terminal, and no node has unsatisfiable guards. Reject on any violation.
- **Status:** PARTIAL — validateManifest catches dangling-edge/bad-entry/dup; deeper semantic contradiction checks still to add.

### `CM-21` — Two concurrent compiles for the same manifest (e.g. an auto-upgrade and a human edit) race; both pass evals against the old baseline and the last write wins, silently discarding one set of authority changes.
- **Consequence:** Lost-update on the security-critical manifest; a human's tightening is overwritten by an auto-upgrade.
- **Guard:** Test: compiles use optimistic concurrency keyed by the predecessor manifest's content hash; a compile whose baseline is stale must rebase and re-gate. Concurrency test asserting one of two racing compiles is rejected, never silently merged.
- **Status:** UNADDRESSED

### `CM-22` — The LLM compiler is asked to emit JSON manifest but returns prose-wrapped or truncated JSON; a lenient parser 'repairs' it (e.g. closes brackets, picks the first JSON-ish block), inferring fields.
- **Consequence:** Repaired manifest does not reflect intent and may include partially-formed authority fields with permissive defaults.
- **Guard:** Design rule: strict parse only, no repair; truncated/extra-content output is a hard failure that retries the compile. Test that prose-wrapped, double-encoded, or truncated outputs are rejected, never repaired.
- **Status:** UNADDRESSED

### `CM-23` — Capability scopes use a wildcard grammar (e.g. 'mcp:*' or 'mail:send:*') and the subset/monotonicity comparator treats '*' inconsistently (a child wildcard considered narrower than a specific grant, or vice versa).
- **Consequence:** A wildcard grant is mis-ranked, allowing widening to pass the monotonicity check.
- **Guard:** Property test: define a partial order over scope patterns and assert it is sound (a<=b implies every concrete resource matching a matches b). Fuzz wildcard pairs and cross-check the comparator against brute-force concrete-resource membership.
- **Status:** UNADDRESSED

### `CM-24` — Migration code path for an OLD signed manifest re-signs with the CURRENT key but does not record/verify the original signer, so a migration laundering an unauthorized old manifest produces a freshly-valid signature.
- **Consequence:** Migration becomes a signature-laundering oracle; provenance of authority is lost.
- **Guard:** Design rule: migration preserves a signed chain (original signer + migrator) on the ledger; re-signing requires the original signature to verify first. Test that migrating a manifest whose original signature is invalid/absent fails, and the signer chain is queryable post-migration.
- **Status:** UNADDRESSED

### `CM-25` — Eval suite uses fixtures that overlap with the compiler's training/few-shot examples, so the agent passes evals by memorization rather than capability, and the gate is not predictive of production behavior.
- **Consequence:** Eval-gated autonomy promotes manifests that fail on novel inputs; false confidence.
- **Guard:** Test: maintain a held-out eval set never shown to the compiler; assert held-out pass rate is within tolerance of the visible set, and alert on divergence (memorization signal). Rotate held-out fixtures and track score deltas.
- **Status:** UNADDRESSED

### `CM-26` — A manifest section marked 'untrusted' can still define or override the GRAPH topology (add an edge to a high-authority node), even though it cannot widen capabilities directly.
- **Consequence:** Untrusted input reroutes control flow into trusted high-authority nodes, achieving escalation via topology rather than scopes.
- **Guard:** Test: topology edges into/out of a node inherit the authority constraints of both endpoints; an untrusted section cannot create an edge whose target authority exceeds the section's. Assert an untrusted section adding an edge to a spend-class node is rejected.
- **Status:** UNADDRESSED

### `CM-27` — Determinism tag of a compiled manifest node is set by the LLM/author rather than derived from the referenced tool's profile, so a non-deterministic effect is tagged 'deterministic' and skipped on verification replay.
- **Consequence:** Verification replay diverges silently (skips re-capture of a non-deterministic effect); cost/audit reconciliation breaks.
- **Guard:** Design rule: determinism tag is derived solely from the signed tool effect-profile (CM-04), never author-set. Test that a manifest asserting a determinism tag inconsistent with the registry is rejected, and replay re-captures every non-deterministic node.
- **Status:** UNADDRESSED

### `CM-28` — Compiler enforces capability limits but not RESOURCE/RATE limits (e.g. no cap on number of nodes, fan-out, loop iterations, or scheduled triggers), so a compiled manifest can request an unbounded effect budget.
- **Consequence:** A compiled manifest exhausts cost/rate budget or creates a runaway loop; denial of wallet.
- **Guard:** Test: schema enforces bounds on node count, max fan-out, loop iteration caps, and declared cost ceiling; a manifest exceeding the principal's resource budget is rejected at compile time. Fuzz with high-fanout/looping manifests.
- **Status:** PARTIAL — run & per-cap budget ceilings + maxNodeVisits loop bound enforced; explicit fan-out/trigger-rate caps still to add.

### `CM-29` — Manifest references external schema or include URLs ($ref to a remote document), and the compiler fetches and inlines them at compile time.
- **Consequence:** SSRF and remote-content injection; the effective manifest depends on a mutable/attacker-controlled remote resource.
- **Guard:** Design rule: no remote $ref resolution; all includes are content-hash-pinned and resolved from a trusted local store. Test that any external/remote reference is rejected and that pinned includes verify by hash.
- **Status:** UNADDRESSED

### `CM-30` — Backward-compat: an OLD runtime loads a NEWER manifest version it doesn't fully understand and ignores newly-added authority-restricting fields (forward-incompatible).
- **Consequence:** A constraint added in a new schema version is silently ignored by an older runtime, executing with looser authority.
- **Guard:** Design rule: manifests carry a minimum-runtime-version requirement; a runtime older than that refuses to load (fail-closed) rather than ignoring unknown restricting fields. Test old-runtime + new-manifest = refusal, not best-effort execution.
- **Status:** UNADDRESSED

### `CM-31` — Secrets or scoped-token references accidentally get embedded into the manifest body by the compiler (LLM echoes a secret it saw in context) and are then signed and stored on the append-only ledger.
- **Consequence:** Secret leaks permanently into an immutable, content-addressed log; non-revocable disclosure.
- **Guard:** Test: a pre-sign scanner rejects any manifest containing high-entropy/secret-shaped strings or known token prefixes; assert the compiler context never includes raw secrets (only broker references). Add a redaction test on every ledger write path.
- **Status:** UNADDRESSED

### `CM-32` — The eval-gate, monotonicity check, and signing happen as separate steps without a single atomic commit, so a manifest can pass evals, then be mutated, then signed (TOCTOU).
- **Consequence:** The signed/promoted manifest is not the one that passed the gate.
- **Guard:** Test: the artifact that is evaluated, the artifact that is monotonicity-checked, and the artifact that is signed are identified by the SAME content hash recorded on the ledger; assert hash equality across all three stages and reject on mismatch.
- **Status:** UNADDRESSED

<a name="memory"></a>
## Memory Planes

### `MEM-01` — Distillation (summarizing episodic->semantic) runs as a non-deterministic LLM call but its output is written to the semantic plane without emitting an EffectRequested/EffectResult pair, so replay regenerates a different summary.
- **Consequence:** Verification replay diverges; cost reconciliation is off; the ledger no longer equals what executed, violating the core invariant.
- **Guard:** Test: run a distillation, then verification-replay the same ledger and assert byte-identical semantic-plane writes are re-fed from captured EffectResult (never recomputed). Add a kernel assertion that any write to semantic/SOUL planes MUST carry a referencing captured-effect event id.
- **Status:** PARTIAL — distillation is modeled as a captured (non-deterministic) EffectResult; the explicit EffectRequested pairing is engine-wiring still to add.

### `MEM-02` — Retrieval (vector/keyword search over episodic/semantic) returns a result set that is captured by reference (ids only) rather than by content hash; the underlying records mutate between capture and replay.
- **Consequence:** Counterfactual and verification replay see different retrieved content than the live run; decisions can't be reproduced.
- **Guard:** Capture retrieval result-sets as a content hash of the full ordered payload (id+content+score). Test: mutate a retrieved record after capture, replay, and assert the replay uses the hashed snapshot, not the mutated row.
- **Status:** GUARDED — retrieve() returns a deterministic, stably-ordered set the engine captures by content (re-fed on replay); test proves order-independence.

### `MEM-03` — Untrusted-inbound provenance tag is set at ingestion but lost when a fact passes through distillation (summary of untrusted episodic memory is written as 'clean' semantic fact).
- **Consequence:** Laundered untrusted content reaches spend/irreversible-decision paths; the provenance bar is silently bypassed.
- **Guard:** Provenance must be a monotone join: any derived fact inherits the most-untrusted tag of ANY input. Test: distill a set containing one untrusted record, assert output carries untrusted-inbound tag and is rejected by the spend gate.
- **Status:** GUARDED — distilledProvenance(): untrustedness PROPAGATES; any untrusted source taints the fact (test).

### `MEM-04` — Retrieval mixes trusted and untrusted memories into one prompt context with no per-chunk provenance, so downstream the LLM-produced action can't be traced to whether untrusted text influenced it.
- **Consequence:** Cannot enforce the laundering bar at the decision point; prompt-injection from episodic memory escalates to real effects.
- **Guard:** Tag every retrieved chunk with provenance and carry it into the captured context. Rule: if any untrusted chunk is in-context when an irreversible/spend EffectRequested is proposed, the kernel HALTs and re-gates with step-up. Test injection payload in episodic memory attempting to trigger a payment.
- **Status:** UNADDRESSED

### `MEM-05` — SOUL plane (identity/values/long-term policy) is mutated by an LLM-proposed distillation without a distinct higher-authority admission gate; SOUL changes use the same authority as working-memory writes.
- **Consequence:** A single compromised or hallucinated distillation rewrites the agent's identity/policy, widening capability silently.
- **Guard:** SOUL mutations require a dedicated AdmissionDecision with elevated authority + human/step-up confirmation, and capability monotonicity check. Test: an untrusted intent proposing a SOUL edit is denied; assert no SOUL write event without the elevated admission event.
- **Status:** GUARDED — SOUL update only applies when authorityOk (owner authority); unauthorized edit ignored (test).

### `MEM-06` — SOUL mutation is applied in place (overwrite) rather than append-only versioned events, so prior SOUL state is unrecoverable and not auditable.
- **Consequence:** Cannot audit how identity drifted; no rollback; ledger-is-runtime invariant broken for the most sensitive plane.
- **Guard:** SOUL stored as append-only versioned events; current SOUL is a fold. Test: apply three SOUL mutations, assert all three are reconstructable from the ledger and that fold(ledger@v2) reproduces the v2 state exactly.
- **Status:** GUARDED — SOUL is append-only versioned events; re-distillation supersedes by version, never mutates in place (test).

### `MEM-07` — Decay/retention deletes (hard-deletes) episodic events from the ledger to save space, breaking the append-only content-addressed chain.
- **Consequence:** Hash chain breaks; replay of older runs fails; audit gaps; crash-hole detection mistakes pruned events for tampering.
- **Guard:** Retention must never delete ledger events; it may only mark planes as non-retrievable via a tombstone event or tier to cold storage with chain intact. Test: run retention, then verify the hash chain still validates end-to-end and replay of a pre-decay run still succeeds.
- **Status:** UNADDRESSED

### `MEM-08` — Decay policy is evaluated with wall-clock time from the local machine rather than the monotonic notarized clock, so an attacker advancing the clock can force premature expiry of inconvenient memories.
- **Consequence:** Targeted memory deletion (e.g., evidence of a prior decision) by clock manipulation; retention becomes an attack surface.
- **Guard:** All retention/TTL comparisons use the notarized monotonic clock; reject decay decisions if clock attestation is stale. Test: skew local clock forward, assert decay does not fire without notarized-clock confirmation.
- **Status:** UNADDRESSED

### `MEM-09` — Embedding model version used for retrieval is not pinned/recorded; an embedding-model upgrade changes vector space, so replay retrieval returns different neighbors.
- **Consequence:** Replay non-determinism in retrieval; counterfactual results are not comparable; silent behavior drift.
- **Guard:** Record embedding model id+version+dimension in each retrieval capture. Test: change embedding model, replay, assert the captured result-set (by hash) is re-fed rather than re-embedded; live runs must re-index before using a new model version.
- **Status:** UNADDRESSED

### `MEM-10` — Working-memory plane leaks across branches: a park-until-event suspension resumes and reads working memory mutated by a sibling branch (no branch-scoping).
- **Consequence:** Cross-branch contamination; non-reproducible state; correlation-id reuse compounds it.
- **Guard:** Working memory keyed by branch-scoped correlation id; reads are isolated per branch. Test: fork two branches, mutate working memory in branch A, resume branch B from park, assert B sees only its own scoped state.
- **Status:** UNADDRESSED

### `MEM-11` — Retrieval cost (embedding API calls, vector DB queries) is not reserved-then-settled; it is recorded after the fact or not at all.
- **Consequence:** Cost reconciliation to the cent fails; budget gates can be bypassed by hammering retrieval; verification replay can't reconcile.
- **Guard:** Each retrieval and embedding call goes through reserve-then-settle with real cost in EffectResult. Test: assert sum of settled retrieval costs over a run equals the reconciled total within zero cents during verification replay.
- **Status:** UNADDRESSED

### `MEM-12` — Distillation effect is captured but its non-determinism tag is set to 'deterministic', so the supervisor allows recompute on replay.
- **Consequence:** Replay recomputes a different summary while believing it's safe; divergence undetected.
- **Guard:** Distillation EffectResult MUST carry determinism=non-deterministic (LLM). Kernel rule: LLM-backed effects cannot be tagged deterministic. Test: attempt to register a distillation plugin claiming determinism, assert rejection.
- **Status:** UNADDRESSED

### `MEM-13` — Memory plugin (e.g., vector store) self-signs the EffectResult asserting what it stored/returned, instead of the supervisor co-signing mechanically-observed I/O.
- **Consequence:** A malicious or buggy memory plugin can lie about retrieval contents; laundering and tampering undetectable.
- **Guard:** Supervisor co-signs the observed request/response bytes for memory effects; plugin signature alone is insufficient. Test: plugin returns content differing from supervisor-observed bytes, assert the fact is rejected.
- **Status:** UNADDRESSED

### `MEM-14` — Semantic-plane upsert uses a natural key (e.g., 'fact about user X') and overwrites, so contradictory facts silently replace each other with no conflict event.
- **Consequence:** Lost information, no audit of contradictions, agent acts on stale overwritten belief.
- **Guard:** Upserts emit a supersede event linking old->new with reason; conflicts surface as events. Test: write fact, write contradicting fact, assert both are in the ledger and current fold shows the supersede chain.
- **Status:** UNADDRESSED

### `MEM-15` — Retrieval injects the top-K by similarity without a recency/decay weighting cap, and a flood of near-duplicate untrusted ingests dominate the result set (poisoning).
- **Consequence:** Attacker controls context via volume; trusted memories crowded out; decisions steered.
- **Guard:** Dedup by content hash before ranking; cap per-provenance and per-source share of any result set; record the dedup/cap parameters in capture. Test: ingest 1000 duplicate untrusted notes, assert they occupy at most the configured share of top-K.
- **Status:** UNADDRESSED

### `MEM-16` — SOUL/semantic memory is shared across tenants because retrieval queries omit the RLS tenant predicate (tenant-of-one assumption violated in hosted mode).
- **Consequence:** Cross-tenant memory leak — catastrophic data breach.
- **Guard:** All memory reads/writes go through RLS-enforced access with tenant id bound from the principal, not from query params. Test: as tenant A, attempt retrieval that would match tenant B's vectors, assert zero cross-tenant rows even with crafted filters.
- **Status:** UNADDRESSED

### `MEM-17` — Distillation reads across planes and writes a SOUL value derived from untrusted episodic content, and the SOUL elevated gate only checks the immediate proposer's authority, not the provenance of inputs.
- **Consequence:** Untrusted content launders into identity via a privileged distillation step.
- **Guard:** SOUL admission must check provenance of ALL transitive inputs (via captured retrieval hashes), not just proposer authority. Test: untrusted-tagged input in a SOUL distillation is blocked even when the distiller is privileged.
- **Status:** UNADDRESSED

### `MEM-18` — Park-until-event correlation ids for memory-driven resumptions are reusable, so a replayed or duplicated event resumes a branch twice, double-writing memory.
- **Consequence:** Duplicate memory facts, double cost, non-idempotent resume.
- **Guard:** Correlation ids are single-use and branch-scoped; second use is rejected. Test: deliver the same correlation event twice, assert exactly one resume and one memory write.
- **Status:** UNADDRESSED

### `MEM-19` — Retrieval result hash is computed over content but not over ORDER, while the prompt is order-sensitive; replay re-feeds same set in different order.
- **Consequence:** LLM output differs on replay despite 'identical' retrieval; verification diverges.
- **Guard:** Hash includes the exact ordered sequence and any truncation applied. Test: same set, different order, assert different capture hash and that replay preserves exact order fed to the model.
- **Status:** UNADDRESSED

### `MEM-20` — Working-memory entries containing freshly minted scoped secrets/tokens are persisted into episodic/semantic memory and later retrieved into prompts.
- **Consequence:** Secrets leak into the ledger and into plugin-visible context, violating 'secrets never touch plugins' and creating durable credential exposure.
- **Guard:** Memory write path scrubs/forbids secret-class values (tagged by the broker); secrets are referenced by handle only. Test: place a broker token in working memory, attempt to distill/persist, assert it is redacted and never appears in any ledger event or retrieval payload.
- **Status:** UNADDRESSED

### `MEM-21` — Decay re-feeds: on counterfactual replay (which forks and re-gates), retention/decay effects are re-executed against live storage, deleting current data.
- **Consequence:** Counterfactual replay (meant to be analytical) destroys real memory state.
- **Guard:** Counterfactual replay operates on a copy-on-write fork of all planes; retention effects in counterfactual mode target the fork only. Test: run counterfactual replay including a decay step, assert live planes unchanged.
- **Status:** UNADDRESSED

### `MEM-22` — Semantic memory stores derived confidence/embeddings but not the source episodic event ids, so provenance and audit trail are broken at the semantic layer.
- **Consequence:** Cannot trace a belief to its evidence; laundering bar cannot evaluate transitive provenance.
- **Guard:** Every semantic fact stores back-references to source event ids + their provenance tags. Test: pick any semantic fact and assert the chain to its originating episodic events resolves and carries provenance.
- **Status:** UNADDRESSED

### `MEM-23` — The retrieval capture stores only what was returned, not the QUERY, so on replay the system cannot prove the captured set corresponds to the query that was actually issued.
- **Consequence:** A swapped query with a stale captured set passes verification; tampering window.
- **Guard:** Capture binds (query text + filters + model version + result-set hash) under one signed event. Test: alter the query post-hoc, assert the signature/binding check fails on replay.
- **Status:** UNADDRESSED

### `MEM-24` — Distillation prompt includes the entire raw episodic plane unbounded, blowing context limits and cost, and silently truncating which records are summarized.
- **Consequence:** Non-deterministic truncation, runaway cost, incomplete/biased summaries with no record of what was dropped.
- **Guard:** Distillation input is an explicit, captured, hashed selection with a recorded budget and truncation policy. Test: assert the set of input event ids is enumerated in the effect request and reconstructable; assert cost stays within reserved budget or HALTs.
- **Status:** UNADDRESSED

### `MEM-25` — Memory plane writes happen before the AdmissionDecision is finalized (write-then-gate), so a denied action still left memory side effects.
- **Consequence:** Gating is cosmetic; denied/halted actions leak state; crash between write and gate creates an inconsistent hole.
- **Guard:** Strict order: AdmissionDecision -> EffectRequested -> EffectResult, then memory commit; no plane write without a preceding finalized admission. Test: inject a crash between request and result, assert HALT and re-gate, and assert no committed memory write exists for the unfinished effect.
- **Status:** UNADDRESSED

### `MEM-26` — SOUL mutation that narrows capability is allowed but one that is presented as 'narrowing' actually widens (e.g., adds an allowed spend category framed as a restriction); monotonicity check only inspects top-level shape.
- **Consequence:** Capability creep through deceptively framed SOUL edits proposed by untrusted compiler output.
- **Guard:** Capability monotonicity is computed structurally over the grant lattice (set of authorities/scopes), not on prose framing; any net-widening from an untrusted intent is denied. Test: a SOUL edit that adds a scope while removing another must be flagged as widening on the added axis.
- **Status:** UNADDRESSED

### `MEM-27` — Embedding/distillation calls are routed to a model not in the manifest's pinned model set (model-agnostic shortcut), so memory derivation uses an unaudited model.
- **Consequence:** Unbounded cost/behavior, provenance of derived facts references a phantom model, replay impossible.
- **Guard:** Memory-effect model must be one of the manifest-declared models; record exact model id in the effect. Test: configure a non-manifest model for distillation, assert the effect is rejected at admission.
- **Status:** UNADDRESSED

### `MEM-28` — Retention archives episodic events to cold storage but the retrieval path silently skips cold-tier records, so older context is invisible without any signal.
- **Consequence:** Agent appears to 'forget' silently; decisions made on incomplete memory with no audit that retrieval was partial.
- **Guard:** Retrieval must record tier-coverage in the capture (e.g., 'cold tier excluded') and surface a degraded-completeness flag. Test: archive records, run retrieval, assert the capture documents the excluded tier and the kernel can require warm-up before high-stakes decisions.
- **Status:** UNADDRESSED

### `MEM-29` — Two concurrent distillations on overlapping episodic ranges both write semantic facts without optimistic-concurrency on the fold, producing duplicate/divergent beliefs.
- **Consequence:** Belief duplication, inconsistent semantic plane, cost double-spent.
- **Guard:** Distillation writes assert an expected fold version (compare-and-append); conflicting writes retry against the new fold. Test: launch two overlapping distillations, assert exactly one commits per fact version and the other rebases or is rejected.
- **Status:** UNADDRESSED

### `MEM-30` — Memory content is content-addressed by hash but the hash excludes provenance/tenant metadata, so an untrusted record and a trusted record with identical text dedup to one entry and inherit the trusted tag.
- **Consequence:** Provenance collision laundering: untrusted text gains trusted status via hash collision on body only.
- **Guard:** Dedup key and content address include provenance + tenant + source, not just body bytes. Test: ingest identical text as trusted and untrusted, assert two distinct entries with their own tags and no tag-merge.
- **Status:** UNADDRESSED

### `MEM-31` — SOUL read is cached in working memory at run start and not re-read after a mid-run SOUL mutation, so the agent acts on a stale identity within the same run.
- **Consequence:** Policy changes don't take effect until next run; an approved restriction is bypassed for the remainder of the current run.
- **Guard:** SOUL reads are versioned; any in-core decision gate re-validates against the latest SOUL fold version before an irreversible effect. Test: mutate SOUL mid-run, attempt a now-forbidden effect, assert it is blocked using the new SOUL version.
- **Status:** UNADDRESSED

### `MEM-32` — Retrieval re-feed on replay is keyed only by a step index, so if the manifest/graph changed between original run and replay the captured set is mismatched to the wrong retrieval node.
- **Consequence:** Replay feeds wrong memory to wrong step; silent corruption of verification.
- **Guard:** Re-feed keyed by the stable node identity + capture event id bound in the ledger, not positional index. Test: edit an unrelated graph node, replay, assert retrieval captures still map to the correct nodes by identity.
- **Status:** UNADDRESSED

## Identity, Secrets & Time

### `IST-01` — Local root signing key is stored on disk unencrypted (or encrypted with a key derived from a constant/empty passphrase) so a single host read exfiltrates the ledger's authenticity root.
- **Consequence:** An attacker who reads one file can forge arbitrary signed events (AdmissionDecision/EffectResult) that replay-verify as authentic; the entire 'what you see is what executed' guarantee collapses silently.
- **Guard:** Test: assert the root private key never exists in plaintext at rest — key material must be wrapped by an OS keystore (Keychain/DPAPI/TPM/SE) or a KDF over a non-empty operator secret; add a unit test that scans the data dir and key files for any 32/64-byte raw Ed25519/secp256k1 key and fails CI if found.
- **Status:** UNADDRESSED

### `IST-02` — Key rotation generates a new key but does not record a signed, chained rotation event (old key signs an endorsement of the new key), so verifiers can't establish continuity of authority across the rotation boundary.
- **Consequence:** After rotation, events signed by the new key are indistinguishable from a key swap by an attacker who stole disk access; replay verification of pre-/post-rotation segments cannot prove they belong to the same principal.
- **Guard:** Design rule + test: every rotation emits a RotationEvent where the outgoing key signs (new_pubkey, effective_at, seq). Verification replay MUST reject a key change that is not endorsed by an unrevoked predecessor key; test a rotation chain of length 3 and a forged unsigned swap that must be rejected.
- **Status:** UNADDRESSED

### `IST-03` — Revocation is enforced only at signing time, not at verification/replay time — a revoked key's previously-signed events keep verifying, and there is no signed RevocationEvent with an effective timestamp folded into replay.
- **Consequence:** A stolen-then-revoked key continues to authenticate forged or malicious events created after the compromise window; audit cannot distinguish legitimate pre-revocation facts from post-compromise forgeries.
- **Guard:** Test: replay verifier must consult a signed revocation list and reject any event whose signing key was revoked with effective_at <= event notarized-time. Property test: events signed after revocation effective time fail; events before pass.
- **Status:** UNADDRESSED

### `IST-04` — The monotonic notarized clock falls back to the local wall clock when the notary/time-attestation source is unreachable, with no tag distinguishing notarized from local timestamps.
- **Consequence:** An attacker controlling local time (the documented offline authenticity gap) can backdate or reorder events to defeat revocation windows, token expiry, and replay reconciliation — undetectably.
- **Guard:** Design rule: every timestamp carries a source tag {notarized|local-unattested} and a monotonic counter. Test: in offline mode all timestamps are tagged local-unattested; any spend/irreversible-effect gate that requires notarized time MUST halt-and-re-gate rather than accept local time. Assert clock counter is strictly monotonic and never decreases across restarts.
- **Status:** UNADDRESSED

### `IST-05` — Monotonic counter for the clock is held only in memory and resets to zero (or re-reads wall clock) after a process crash/restart, allowing timestamp regression.
- **Consequence:** Two distinct events can receive the same or decreasing (time, counter) tuple, breaking total ordering of the ledger and enabling replay/reorder attacks across a crash boundary.
- **Guard:** Test: persist the monotonic counter and last-issued time to durable storage on every increment (or use a HLC). After a simulated crash + restart, assert the next issued timestamp tuple is strictly greater than the last persisted one; fuzz with backward wall-clock jumps.
- **Status:** UNADDRESSED

### `IST-06` — Scoped-token broker mints tokens whose scope is broader than the requesting effect needs (e.g., full read/write to a resource when only one record read was requested), or with no audience binding.
- **Consequence:** A compromised or buggy plugin replays the token against other resources/audiences, escalating from a narrow grant to broad access — violating SECRETS NEVER TOUCH PLUGINS in spirit by handing out over-broad capability.
- **Guard:** Design rule: broker derives scope strictly from the manifest effect-profile, minimal by construction; token MUST carry aud, resource, action, and a content hash of the granting AdmissionDecision. Test: request a single-record read and assert minted token's claims contain exactly that resource+action and a matching aud; reject any broker code path that defaults to wildcard scope.
- **Status:** UNADDRESSED

### `IST-07` — Broker-minted tokens have long TTLs (hours/days) or no expiry, contradicting 'scoped short-lived'.
- **Consequence:** Token leakage (logs, memory dump, plugin compromise) yields a long window of unauthorized access; revocation lag becomes the de-facto exposure window.
- **Guard:** Test: assert every minted token's exp - iat <= configured max (e.g., 300s) and is non-zero; CI fails if any broker path omits exp. Add a runtime guard that refuses to mint a token without a bounded TTL.
- **Status:** UNADDRESSED

### `IST-08` — Tokens are bearer-only with no proof-of-possession (no DPoP/mTLS binding), so any component that observes the token can use it.
- **Consequence:** A token logged by an effect-runner or sniffed in transit is fully replayable by any party, including a malicious channel adapter, breaking the deny-by-default trust boundary.
- **Guard:** Design rule: bind tokens to a key the legitimate holder proves possession of (DPoP-style cnf claim or mTLS thumbprint). Test: replay a captured token from a different key/connection and assert the resource rejects it.
- **Status:** UNADDRESSED

### `IST-09` — Secrets are passed to plugins as environment variables or process arguments, or appear in the EffectRequested event payload, directly violating SECRETS NEVER TOUCH PLUGINS.
- **Consequence:** Secrets leak into the append-only ledger (permanent, content-addressed, replayable forever), crash dumps, and process listings — an unrecoverable disclosure baked into immutable history.
- **Guard:** Test: scan all EffectRequested/EffectResult payloads and plugin invocation surfaces for known secret patterns and the broker's master secret; assert plugins receive only short-lived scoped tokens, never raw secrets. Add a redaction/forbidden-key linter over event serialization with a known-secret canary that must never appear in the ledger.
- **Status:** UNADDRESSED

### `IST-10` — Plugins self-report their identity/authority (claim 'I am plugin X with authority Y') and the supervisor trusts it, instead of binding identity to a verified MCP plugin signature.
- **Consequence:** A malicious or swapped plugin impersonates a higher-authority plugin and obtains tokens/effects it should be denied — violating 'plugins MUST NOT self-sign facts about their own behavior'.
- **Guard:** Design rule: plugin identity = verified signature over a pinned plugin manifest hash; supervisor resolves authority from the signed registry, never from plugin-asserted claims. Test: a plugin presenting a forged/altered identity claim with a non-matching signature is denied; authority is read only from the signed effect-profile source of truth.
- **Status:** UNADDRESSED

### `IST-11` — Principal binding maps a channel message to a principal using only an unauthenticated identifier (e.g., a Telegram username or email From header) with no cryptographic or assurance-level check.
- **Consequence:** Spoofing a username/From header lets an attacker act as a trusted principal, since 'approval IS authorization' in the Interaction Resolver — an impersonated approval authorizes real spend/irreversible effects.
- **Guard:** Test: assert each inbound principal carries an identity-assurance level derived from verified channel auth (signed webhook, verified sender, OAuth subject), not a free-text field. Reject/quarantine any principal binding below the assurance threshold required by the effect; spoofed From/username test must fail to authorize.
- **Status:** UNADDRESSED

### `IST-12` — Step-up authentication is requested but the resulting elevated assurance is cached/reused beyond its intended single decision or branch scope.
- **Consequence:** One step-up unlocks many subsequent irreversible actions the user never re-approved; an attacker who triggers actions right after a legitimate step-up rides the elevated session.
- **Guard:** Design rule: step-up assurance is bound to a single-use, branch-scoped correlation id (the documented park-until-event primitive) and a specific effect hash. Test: a second effect attempting to consume the same step-up correlation id is rejected; assurance does not persist across decisions.
- **Status:** UNADDRESSED

### `IST-13` — The single-use branch-scoped correlation id for park-until-event is guessable/sequential or not cryptographically random, and is not invalidated after first use.
- **Consequence:** An attacker forges or replays a resume event to wake a parked branch with attacker-chosen data, or double-resumes to execute an effect twice (defeating idempotency).
- **Guard:** Test: correlation ids are >=128 bits CSPRNG; a resume with a wrong/forged id is rejected; a second resume with a valid-but-already-consumed id is rejected (single-use enforced atomically). Fuzz with replayed resume events.
- **Status:** UNADDRESSED

### `IST-14` — Channel adapters (stateless sandboxed) are trusted to assert the assurance level instead of the in-core Interaction Resolver computing it from verifiable evidence.
- **Consequence:** A compromised adapter inflates assurance ('this user passed MFA') and the core authorizes accordingly, collapsing the trusted/untrusted boundary the architecture depends on.
- **Guard:** Design rule: adapters forward raw verifiable evidence (signed tokens, provider assertions); the in-core Resolver alone computes assurance. Test: an adapter-supplied assurance claim is ignored; tampering with forwarded evidence (bad signature) yields the lowest assurance, not the claimed one.
- **Status:** UNADDRESSED

### `IST-15` — Key custody has no separation between the ledger-signing key and the supervisor co-signing key, so one stolen key can both originate an effect request and co-sign the observed result.
- **Consequence:** An attacker forges complete three-event atomic facts (Admission -> EffectRequested -> supervisor-signed EffectResult) that look mechanically observed and real, defeating the supervisor co-signing safeguard.
- **Guard:** Design rule: supervisor key MUST be distinct, separately custodied, and ideally on a different trust domain (HSM/SE) from the requester key. Test: signing both the request and the supervisor-attested result with the same key fails verification; verifier asserts requester_key != supervisor_key per fact.
- **Status:** UNADDRESSED

### `IST-16` — Offline mode accepts attestations (notarized clock, identity provider assertions) cached from before going offline without checking their freshness/expiry against the monotonic counter.
- **Consequence:** Stale attestations are honored indefinitely offline; an attacker who can keep the node offline extends the validity of revoked keys or expired tokens (the authenticity gap weaponized).
- **Guard:** Test: cached attestations carry max-age; after the monotonic counter advances past max-age, offline gates requiring fresh attestation HALT and queue for re-gating on reconnect rather than proceeding. Simulate offline + counter advance and assert halt.
- **Status:** UNADDRESSED

### `IST-17` — Time skew between the issuing broker and the verifying resource is not bounded, so tokens validated with a generous leeway (e.g., +/- minutes) effectively extend short TTLs.
- **Consequence:** A 60s token with 5-minute clock leeway is usable for ~6 minutes, and backdated local clocks (the gap) widen this further.
- **Guard:** Test: assert clock-skew leeway in token validation is <= a small bound (e.g., 5s) and is configurable but capped; property test that a token cannot be made valid beyond exp + leeway even under adversarial local clock within the offline tag.
- **Status:** UNADDRESSED

### `IST-18` — Revocation propagation has no bound or heartbeat — a revoked key/token is not detected by long-running effect-runners or parked branches until they next contact the core.
- **Consequence:** Compromised credentials remain usable on isolated workers for an unbounded window; revocation is advisory rather than enforced.
- **Guard:** Design rule: effect-runners require a fresh, signed revocation epoch (short TTL) to act on irreversible effects; missing/stale epoch -> halt. Test: revoke a key, advance time past the epoch TTL, assert a worker refuses the next irreversible effect without a refreshed revocation epoch.
- **Status:** UNADDRESSED

### `IST-19` — Identity assurance levels are an unordered set of strings (e.g., 'mfa', 'sso', 'verified') with no defined ordering, so the manifest's 'requires assurance >= X' comparison is ambiguous or string-compared.
- **Consequence:** Authorization checks pass/fail inconsistently; a lower assurance can be accepted for a high-authority effect due to lexical comparison, enabling under-authorized irreversible actions.
- **Guard:** Test: assurance is a typed monotone lattice/integer; the gate uses numeric ordering. Property test: for all a<b, an effect requiring b rejects a; CI lint forbids string comparison of assurance.
- **Status:** UNADDRESSED

### `IST-20` — The scoped-token broker is itself reachable by plugins directly (plugin -> broker), letting a plugin request tokens for scopes it was never granted by an AdmissionDecision.
- **Consequence:** Plugins self-escalate by asking the broker for more than the kernel authorized, bypassing capability monotonicity.
- **Guard:** Design rule: broker mints ONLY against a signed AdmissionDecision presented by the trusted effect-runner, never on a plugin's direct request; token scope is the intersection of (admission grant, manifest profile). Test: a plugin-originated mint request with no backing AdmissionDecision is rejected; request scope exceeding the admission grant is clamped/denied.
- **Status:** UNADDRESSED

### `IST-21` — Master secret / signing key has no rotation or revocation procedure at all (rotation 'will be added later'), so a single compromise is permanent.
- **Consequence:** When the inevitable leak happens there is no recovery path; the platform's trust root is unrecoverable, which is fatal for a 'most-trustworthy' positioning.
- **Guard:** Test: there exists an automated, tested rotation+revocation runbook executed in CI against a test ledger that proves continuity (IST-02) and post-rotation verification. Absence of a passing rotation integration test fails the release gate.
- **Status:** UNADDRESSED

### `IST-22` — Time used for cost reserve-then-settle (token/budget windows) is derived from local unattested time, letting a manipulated clock reset rate/budget windows.
- **Consequence:** An attacker resets spend windows by moving the clock, defeating reserve-then-settle and enabling unbounded spend that still 'reconciles to the cent' on a forged timeline.
- **Guard:** Test: budget/rate windows key off the monotonic notarized clock (or monotonic counter) only; clock rollback does not reset a window. Simulate wall-clock rollback and assert the spend window does not reopen.
- **Status:** UNADDRESSED

### `IST-23` — Principal identity is bound at admission but not re-bound/verified at effect-settlement, so a long-parked branch resumes and executes under the original principal even after that principal's authorization was revoked.
- **Consequence:** Revoked or de-provisioned users still cause irreversible effects when stale parked branches wake, because authority was checked once at admission only.
- **Guard:** Design rule: irreversible effects re-gate principal authorization at settlement time against the current revocation epoch. Test: park a branch, revoke the principal, resume, assert the effect halts and re-gates rather than executing.
- **Status:** UNADDRESSED

### `IST-24` — Token and key material are logged for debugging (DEBUG level prints token claims/JWT, or signing happens via a logged subprocess), leaking into the observability/ledger pipeline.
- **Consequence:** Secrets/tokens land in logs that may be shipped, replayed, or folded into observability reducers; permanent exposure given append-only design.
- **Guard:** Test: a log-scrubbing test feeds known token/key canaries through all log levels and asserts they are redacted; CI greps the codebase for logging of token/secret variables and fails on match (aligns with project rule: never print()/never hardcode secrets).
- **Status:** UNADDRESSED

### `IST-25` — Signature verification uses a non-constant-time comparison or accepts multiple signature encodings (malleability), enabling signature forgery or timing side-channels on the verification path.
- **Consequence:** Attacker crafts a malleable variant of a signed event that passes verification, or extracts key bits via timing — undermining ledger authenticity.
- **Guard:** Test: use a vetted library with constant-time verify and strict canonical encoding; add a malleability test (alter S in (R,S), re-encode) asserting rejection, and verify no custom byte-compare of signatures exists.
- **Status:** UNADDRESSED

### `IST-26` — Event canonicalization before signing is non-deterministic (JSON key order, float formatting, unicode normalization), so the content-addressed hash and signature differ across nodes/versions.
- **Consequence:** Replay verification on another node fails legitimately OR an attacker exploits canonicalization ambiguity to make two different payloads share a signature/hash, breaking content-addressing.
- **Guard:** Test: define one canonical serialization (e.g., RFC 8785 JCS or deterministic CBOR); round-trip and cross-version golden tests assert identical bytes/hash. Fuzz with reordered keys, NFC/NFD unicode, and -0.0 to confirm stable hashing.
- **Status:** UNADDRESSED

### `IST-27` — Nonce/idempotency keys for EffectRequested are derived from the principal-controlled or plugin-controlled input rather than the kernel, allowing collision/prediction.
- **Consequence:** Two different effects collide on an idempotency key (one silently dropped) or an attacker predicts a future key to pre-seed a fraudulent EffectResult — violating the crash-hole HALT/re-gate guarantee.
- **Guard:** Test: idempotency key = hash(kernel-controlled fields incl. monotonic counter + branch correlation id), never attacker-influenced alone. Property test: distinct effects never collide; key is unpredictable without kernel state.
- **Status:** UNADDRESSED

### `IST-28` — RLS tenant-of-one isolation relies on an application-supplied tenant id that is not cryptographically bound to the authenticated principal/key, so a bug or forged context reads another tenant's keys/tokens.
- **Consequence:** Cross-tenant secret/key disclosure in the hosted (and even multi-profile self-host) deployment — catastrophic for a self-host-first trust story.
- **Guard:** Test: tenant context is set from the verified principal binding, not a request parameter; Postgres RLS policies tested with a forged/mismatched tenant id assert zero rows. Add an integration test that attempts cross-tenant key/token read and expects denial.
- **Status:** UNADDRESSED

### `IST-29` — The notarized clock's notary trust is single-source (one external time-attestation provider) with no quorum or pinned cross-check, so compromising/spoofing it forges notarized time.
- **Consequence:** The defense against the documented clock-attack collapses to a single point of compromise; 'notarized' time is no stronger than one vendor's availability/honesty.
- **Guard:** Design rule: notarized time requires agreement of >=2 independent roughtime/notary sources within a bound, with pinned keys. Test: feed two sources that disagree beyond bound and assert time is tagged unattested + gates halt; feed one spoofed source and assert it cannot alone produce notarized time.
- **Status:** UNADDRESSED

### `IST-30` — Short-lived tokens are not revocable before expiry (no jti deny-list), so a known-compromised token stays valid for its full TTL.
- **Consequence:** Even with short TTLs, an actively exploited token cannot be killed mid-window; incident response cannot contain a leak in progress.
- **Guard:** Test: every token carries a jti; the resource checks a signed short-TTL deny-list epoch. Revoke a jti and assert the next use is rejected before exp. Ensure deny-list epoch staleness causes halt for irreversible effects (ties to IST-18).
- **Status:** UNADDRESSED

### `IST-31` — Key generation uses a weak or non-CSPRNG source (e.g., Math.random, seeded RNG, or RNG before entropy is available at first boot of the self-host artifact).
- **Consequence:** Predictable keys/correlation ids allow forgery of the root of trust on freshly provisioned self-host instances — the worst possible time, at first boot.
- **Guard:** Test: assert all key/nonce generation uses the platform CSPRNG (crypto.getRandomValues/node:crypto) and that first-boot blocks until sufficient entropy. Statistical test on generated keys; CI lint bans Math.random in security paths.
- **Status:** UNADDRESSED

### `IST-32` — Untrusted-inbound provenance tag is dropped or not propagated through principal binding, so a laundered fact (data that entered via an untrusted channel) is later treated as authorized to drive spend/irreversible decisions.
- **Consequence:** An attacker injects content through an untrusted channel that, after passing through memory/distillation, gains authority it should be barred from — the exact laundering the architecture says to prevent.
- **Guard:** Test: provenance.untrusted_inbound is a sticky taint that propagates across binding, memory planes, and distillation; assert any spend/irreversible gate rejects an input whose taint set includes untrusted_inbound. Trace a tainted value end-to-end and confirm it cannot authorize an effect.
- **Status:** UNADDRESSED

## Channels & Observability

### `CH-01` — A channel adapter (Telegram/Slack/email) caches conversation state, queued messages, or a partial reply buffer in adapter-local memory instead of round-tripping through the ledger, then the adapter process restarts.
- **Consequence:** Approval context is lost; a user reply that authorized an effect vanishes or is double-delivered, breaking 'what you see is exactly what executed'.
- **Guard:** Statelessness conformance test: run every adapter under a chaos harness that kills the process between every inbound and outbound event; assert zero behavioral difference vs. no-kill run. Static lint: adapter modules may not declare mutable module-level/instance state except connection handles.
- **Status:** UNADDRESSED

### `CH-02` — The interaction resolver treats a channel-delivered 'yes' as authorization without binding it to the specific EffectRequested correlation id, so a stale or out-of-order approval authorizes a different pending effect.
- **Consequence:** User approves action A, but approval is applied to action B (approval confusion / authorization replay).
- **Guard:** Every approval event MUST carry the single-use branch-scoped correlation id of the exact EffectRequested it answers; resolver rejects approvals whose correlation id is unknown, already-consumed, or scoped to a different branch. Test: inject approval with mismatched/expired/reused correlation id -> must be denied and ledger-logged.
- **Status:** GUARDED — resolver binds the reply to a single-use, branch-scoped correlation id; stale/replayed token → ALREADY_RESOLVED (test).

### `CH-03` — Approval message rendered to the user describes a different action, amount, or recipient than the EffectRequested it is bound to (template drift, truncation, or a later mutation of the effect after the prompt was sent).
- **Consequence:** User authorizes something other than what executes; 'approval is authorization' is violated because consent didn't match the act.
- **Guard:** The approval prompt MUST be derived by a pure function from the content-addressed EffectRequested hash, and that hash MUST be re-verified at consumption time; if the effect's content hash changed after the prompt, invalidate the approval. Test: mutate effect post-prompt -> approval rejected.
- **Status:** UNADDRESSED

### `CH-04` — Inbound channel content (an email body, a message) is ingested without the untrusted-inbound provenance tag, or the tag is dropped as the content flows through the resolver into manifest variables.
- **Consequence:** Laundered untrusted text reaches spend/irreversible decisions (prompt injection -> 'wire $10k'), defeating the provenance bar.
- **Guard:** Provenance tag is part of the event's content-addressed payload and is unforgeable/sticky through every fold; kernel refuses any spend/irreversible EffectRequested whose authorizing inputs transitively carry untrusted-inbound. Test: tainted inbound -> attempt spend -> kernel HALT + gate.
- **Status:** UNADDRESSED

### `CH-05` — Identity assurance level for a channel principal is assumed constant; a low-assurance channel (unauthenticated webhook, SMS spoofable) is allowed to grant the same authority as a high-assurance, step-up-verified session.
- **Consequence:** Attacker forges a low-assurance message and authorizes a high-authority effect with no step-up.
- **Guard:** Each effect's authority axis declares a minimum identity-assurance level; resolver compares the binding principal's per-channel assurance and forces step-up if below threshold. Test matrix: every (channel assurance level x effect authority) pair asserts allow/step-up/deny.
- **Status:** GUARDED — per-effect-class assurance requirement; low-assurance channel cannot authorize spend without step-up (test).

### `CH-06` — Step-up challenge can be satisfied on a different channel/device than the one that initiated, with no binding between initiator and verifier.
- **Consequence:** Phishing: attacker initiates on victim's account, victim completes step-up on their phone thinking it's their own action.
- **Guard:** Step-up correlation id binds the originating branch + principal + initiating channel; the verifying channel must present matching binding and the resolver shows the actual pending action during step-up. Test: cross-channel step-up with mismatched binding -> denied.
- **Status:** UNADDRESSED

### `CH-07` — Channel adapter runs effect-bearing code (sends money, calls APIs) directly instead of only emitting events for the effect-runner, blurring the channel/effect boundary.
- **Consequence:** Side effects occur that never pass admission gating and never appear as the AdmissionDecision->EffectRequested->EffectResult triple; ledger is no longer the runtime.
- **Guard:** Channels are in a deny-by-default sandbox with NO outbound network/effect capability except posting signed events to the core. Test: adapter attempting any non-event side effect is blocked by sandbox policy and the attempt is logged.
- **Status:** UNADDRESSED

### `CH-08` — Duplicate inbound delivery from a channel (webhook retried, user double-taps approve) is processed twice because the adapter does not deduplicate by provider message id.
- **Consequence:** An effect is authorized/triggered twice; double-spend or duplicate notification.
- **Guard:** Inbound events carry a channel-scoped idempotency key (provider message id); resolver folds them idempotently. Test: replay same provider message id N times -> exactly one logical effect.
- **Status:** UNADDRESSED

### `CH-09` — An approval has no expiry; a 'yes' from days ago is still consumable when the same effect is re-proposed.
- **Consequence:** Stale consent authorizes an action the user no longer intends.
- **Guard:** Approvals carry a TTL and are single-use; park-until-event for approval auto-expires. Test: consume approval after TTL -> rejected, must re-prompt.
- **Status:** UNADDRESSED

### `CH-10` — Resolver maps an inbound reply to a pending request by loose heuristics (most recent prompt, fuzzy text match) rather than explicit correlation, when multiple approvals are pending in one chat.
- **Consequence:** Reply 'yes' resolves the wrong pending approval (ambiguity attack / honest mistake).
- **Guard:** When >1 approval is pending on a channel that lacks per-message reply threading, resolver MUST require an explicit disambiguating token in the prompt and reject bare yes/no. Test: two concurrent approvals + bare 'yes' -> resolver refuses and re-asks.
- **Status:** GUARDED — reply mapped by EXPLICIT correlation id only (no fuzzy/most-recent heuristic); unknown id rejected (test).

### `CH-11` — Channel adapter signs facts about delivery/read receipts as if authoritative (self-signs 'user saw this'), violating the plugins-must-not-self-sign rule.
- **Consequence:** Forgeable audit claim that a user was notified/approved.
- **Guard:** Delivery/read facts are co-signed by the supervisor based on mechanically observed transport ACKs only; adapter signature alone is insufficient. Test: adapter-only-signed delivery fact is treated as unverified by all reducers.
- **Status:** UNADDRESSED

### `CH-12` — Per-channel principal identity is keyed on a mutable handle (phone number, @username, email display) that can be reassigned/recycled by the provider.
- **Consequence:** A recycled phone number gives a new person the prior owner's authority bindings.
- **Guard:** Bind principal to a provider-stable immutable subject id plus a rotation/revocation event when reassignment is detected; require re-enrollment on identity-id change. Test: simulate handle recycling -> old bindings revoked, step-up forced.
- **Status:** UNADDRESSED

### `CH-13` — Approval UI uses default-affirmative / pre-checked / single-tap that triggers on accidental interaction, or batches many effects under one 'approve all'.
- **Consequence:** User grants broad authority unintentionally; capability over-grant.
- **Guard:** High-authority/irreversible effects require explicit per-item affirmative with the human-readable consequence shown; no 'approve all' for irreversible class. Test: assert irreversible effects cannot be batch-approved.
- **Status:** UNADDRESSED

### `OB-01` — Verification replay (zero side effects) accidentally invokes a real effect because a code path doesn't distinguish replay-mode from live, or a reducer has a hidden side effect (logging to external sink, metric emit with cost).
- **Consequence:** 'Zero side effects' replay actually mutates the world; reconciliation is wrong and possibly harmful.
- **Guard:** Replay runs the kernel with the effect-runner physically swapped for a null/record-only stub at the type level (effects are data, not calls); a side-effect detector asserts zero outbound syscalls/network during verification replay. Test: instrument and assert 0 effects.
- **Status:** UNADDRESSED

### `OB-02` — Verification replay reconciles cost to the cent using recomputed/estimated cost instead of the supervisor-signed real cost in EffectResult, so it 'reconciles' against itself.
- **Consequence:** Cost drift goes undetected; the replay falsely claims the ledger is consistent.
- **Guard:** Reconciliation compares the recorded supervisor-signed EffectResult.cost against the reserved amount and against an independent re-derivation; mismatch beyond zero tolerance fails replay. Test: tamper one EffectResult cost by 1 cent -> replay FAILS.
- **Status:** UNADDRESSED

### `OB-03` — Counterfactual replay forks but reuses cached non-deterministic results (LLM outputs, retrieval sets) from the original run instead of re-gating and re-capturing, when the fork changes upstream inputs.
- **Consequence:** Counterfactual is contaminated by original-run data; conclusions are invalid; effects that should re-gate are skipped.
- **Guard:** Counterfactual mode marks the fork point; any captured non-deterministic effect downstream of a changed input is invalidated and MUST be re-gated/re-executed (or explicitly stubbed). Test: change an input pre-fork -> assert all downstream captured ND results are re-requested, not reused.
- **Status:** UNADDRESSED

### `OB-04` — Counterfactual replay re-executes a real irreversible effect against production because re-gating routes to the live effect-runner.
- **Consequence:** A 'what if' analysis sends real money/emails.
- **Guard:** Counterfactual forks run in an isolated effect domain with all side-effect-class!=read effects either denied or routed to a sandbox; re-gating in counterfactual mode requires explicit human authorization per irreversible effect. Test: counterfactual fork attempting irreversible effect -> blocked unless explicitly authorized in fork context.
- **Status:** UNADDRESSED

### `OB-05` — Reserve-then-settle leaks reservations: an effect reserves budget, then crashes between EffectRequested and EffectResult, and the reservation is never released or settled.
- **Consequence:** Budget is permanently consumed by phantom reservations; agent starves and stops.
- **Guard:** Reservations are tied to a correlation id with a lease/TTL; the crash-hole HALT path releases or settles via a compensating event on re-gate. Test: kill between request and result -> reservation reconciled (released or settled) on recovery, never orphaned.
- **Status:** UNADDRESSED

### `OB-06` — Settle-after-reserve allows actual cost to exceed reserved amount with no enforcement, because the cap is only checked at reserve time.
- **Consequence:** Runaway effect (huge token bill, paginated API storm) blows the budget; reserve becomes advisory.
- **Guard:** Effect-runner enforces a hard ceiling = reserved amount; effects that would exceed it are aborted mid-flight and settle at the cap with an overrun event. Test: effect that tries to consume 2x reserve -> aborted at reserve ceiling.
- **Status:** UNADDRESSED

### `OB-07` — Double counting / under counting in cost folds: the same EffectResult is summed twice (retry created two results) or a settled-down adjustment is ignored.
- **Consequence:** Cost dashboard and budget enforcement disagree with reality.
- **Guard:** Cost reducer is idempotent over EffectResult by correlation id; settlement supersedes reservation rather than adding. Property test: cost fold over any permutation/duplication of events yields identical total = sum of distinct settled results.
- **Status:** UNADDRESSED

### `OB-08` — The autonomy ladder auto-upgrades to a higher autonomy tier based on eval scores that were computed on stale, cherry-picked, or self-selected traces (the agent grades its own homework).
- **Consequence:** Unsafe autonomy escalation; agent gains spend authority it shouldn't have.
- **Guard:** Auto-upgrade gates require compiled evals run on an independent, held-out, provenance-clean trace set; eval results are supervisor-signed and the trace selection is deterministic/auditable. Test: feed cherry-picked traces -> upgrade gate rejects due to selection-bias check / minimum coverage.
- **Status:** UNADDRESSED

### `OB-09` — Evals are run by the same untrusted LLM/agent that produced the behavior, or eval prompts/thresholds are mutable by the agent, so the autonomy gate can be talked past.
- **Consequence:** Eval-gated autonomy is theater; LLM grades itself up.
- **Guard:** Evals are part of the signed manifest under a separate authority section that untrusted intents cannot widen (capability monotonicity); eval execution uses a pinned, separate evaluator with its own key. Test: untrusted intent attempting to lower an eval threshold -> rejected by monotonicity check.
- **Status:** UNADDRESSED

### `OB-10` — Retrieval/memory result-sets are captured by reference (id, query) instead of by content hash, and the underlying store mutates before replay.
- **Consequence:** Replay re-feeds different data than the original run; verification falsely passes/fails; 'what you see' diverges from what executed.
- **Guard:** Captured retrieval result-sets are stored by content hash of the materialized set and re-fed verbatim on replay; replay verifies the live store's current hash differs is tolerated (replay uses captured), but counterfactual re-runs query. Test: mutate store post-run -> verification replay still uses captured set and reconciles.
- **Status:** UNADDRESSED

### `OB-11` — Observability reducers are not actually pure — they read wall-clock time, random, env, or current store state, so two folds of the same ledger produce different views.
- **Consequence:** Canvas/monitor/cost/audit are non-reproducible; replay determinism breaks.
- **Guard:** Reducers are pure functions of (state, event) only; time/random come from notarized clock events in the ledger, not Date.now(). Property test: fold(ledger) == fold(ledger) byte-identical across runs/machines; lint bans Date.now/Math.random/process.env in reducer modules.
- **Status:** UNADDRESSED

### `OB-12` — Determinism tag on EffectResult is set by the plugin and trusted, so a non-deterministic effect is mislabeled deterministic and not re-captured.
- **Consequence:** Replay reuses a value that wasn't actually reproducible; silent divergence.
- **Guard:** Determinism classification is by effect profile (3-axis) declared in the single source of truth, co-signed by supervisor based on observed behavior, not plugin self-report. Test: plugin claims deterministic but supervisor observes output variance across probe -> tag forced to non-deterministic, captured.
- **Status:** UNADDRESSED

### `OB-13` — Crash hole (event between EffectRequested and EffectResult is missing) is silently filled by re-running the effect, violating 'never silently re-executed'.
- **Consequence:** Idempotency assumed but not guaranteed -> duplicate side effect (double charge).
- **Guard:** Missing EffectResult HALTs the branch and re-gates with the idempotency key; if the downstream system can't confirm prior execution, require human decision. Test: simulate missing EffectResult -> branch halts, no auto re-execution, re-gate event emitted.
- **Status:** UNADDRESSED

### `OB-14` — Replay cost reconciliation only checks totals, not per-effect attribution, so an over-charge on effect A and a missing charge on effect B cancel out.
- **Consequence:** Net total looks correct; per-action audit is wrong; fraud/bug hidden.
- **Guard:** Reconcile per correlation id, not just the sum; every reserved id must have exactly one settled result within tolerance. Test: offset two effects by +/- equal amounts -> reconciliation FAILS on per-id check.
- **Status:** UNADDRESSED

### `OB-15` — Counterfactual fork shares the same correlation id namespace as the live run, so a fork's EffectRequested collides with or consumes the live run's idempotency key / park-until-event.
- **Consequence:** Fork unparks a live suspension or dedupes against live effects; counterfactual leaks into reality.
- **Guard:** Forks get a distinct branch-scoped correlation namespace; single-use correlation ids are branch-qualified. Test: run live + counterfactual concurrently -> assert no id collision and live suspensions are not resolved by fork events.
- **Status:** UNADDRESSED

### `OB-16` — Autonomy ladder has no downgrade/kill path; once upgraded, a regression in evals doesn't pull authority back.
- **Consequence:** Agent keeps elevated spend authority after it starts misbehaving.
- **Guard:** Autonomy tier is continuously gated; a failing eval or budget-overrun emits an automatic downgrade event that re-narrows authority on the next admission decision. Test: inject eval regression -> next effect requiring elevated authority is denied/step-up.
- **Status:** UNADDRESSED

### `OB-17` — Reserve-then-settle ignores concurrency: two parallel branches each reserve against the same budget without atomic decrement, both pass the check.
- **Consequence:** Combined spend exceeds budget (TOCTOU over-spend).
- **Guard:** Budget reservation is an atomic, serializable ledger operation (compare-and-append) so concurrent reserves can't both succeed past the cap. Test: N concurrent reserves of (budget/2 + 1) -> exactly one succeeds.
- **Status:** UNADDRESSED

### `OB-18` — Eval result and provenance are tracked in separate systems that can disagree (an eval passes on a trace whose provenance is later found tainted).
- **Consequence:** Autonomy granted on tainted evidence; unified evals+provenance promise broken.
- **Guard:** Evals reference the exact content-addressed trace events and inherit their provenance; an eval over any untrusted-inbound-tainted trace cannot satisfy an autonomy-upgrade gate. Test: eval over tainted trace -> gate rejects on provenance grounds.
- **Status:** UNADDRESSED

### `OB-19` — Notarized monotonic clock is only used for some events; some cost/eval/effect timestamps use local wall clock, which an attacker or NTP skew can move backward.
- **Consequence:** Reservations 'expire' early/late, replay ordering wrong, attacker rewinds time to reuse a single-use approval.
- **Guard:** All ordering/expiry decisions use the monotonic notarized clock event sequence, never local timestamps; reject events whose notarized time goes backward. Test: feed a backward-dated event -> rejected; expiry logic uses notarized seq only.
- **Status:** UNADDRESSED

### `OB-20` — Replay of a partially-failed run does not reproduce the EXACT failure (error, timeout, partial output) because the failure details weren't captured as an event.
- **Consequence:** Verification replay diverges at the failure point; can't reproduce incidents.
- **Guard:** Failures, timeouts, and partial results are first-class captured EffectResult variants (with error payload hashed); replay re-injects the captured failure. Test: a run that timed out replays to the same timeout state deterministically.
- **Status:** UNADDRESSED

### `OB-21` — Cost dashboard (a fold) and the budget gate (kernel) use two different cost reducers that drift over versions.
- **Consequence:** Users see a cost number that doesn't match what the gate enforced -> trust collapse ('what you see is NOT what executed').
- **Guard:** Single shared cost reducer module used by both the gate and the view; versioned and pinned per ledger; replay uses the reducer version recorded in the ledger. Test: gate total == dashboard total for every run, byte-equal.
- **Status:** UNADDRESSED

### `OB-22` — Distillation (memory consolidation) runs during replay as if deterministic, re-deriving SOUL/semantic memory differently than originally.
- **Consequence:** Replayed agent has different memory than the real run; behavior diverges.
- **Guard:** Distillation is a captured non-deterministic effect (per spec); replay re-feeds the captured distillation output by hash, never re-runs it in verification mode. Test: verification replay does not invoke the distiller; output matches captured.
- **Status:** UNADDRESSED

### `OB-23` — Approval-is-authorization decisions are recorded in the ledger but the actual authority scope granted is not, so audit can't show WHAT the approval permitted.
- **Consequence:** Replay/audit can't prove the effect stayed within the approved scope; over-execution invisible.
- **Guard:** AdmissionDecision records the exact authority scope (resource, amount cap, expiry) derived from the approval; the effect-runner enforces and the auditor checks the EffectResult stayed within it. Test: effect exceeds approved scope -> denied at runtime and flagged in audit.
- **Status:** UNADDRESSED

### `OB-24` — Counterfactual 'what if I'd used model X' replay changes the effect-profile (authority/determinism) but reuses the original AdmissionDecision, so it doesn't re-gate against the new model's higher risk.
- **Consequence:** Counterfactual underestimates risk/cost of an alternative; misleading analysis used to justify autonomy.
- **Guard:** Any change to model or effect profile in a fork invalidates the original AdmissionDecision and forces re-gating with the new 3-axis profile. Test: swap model in fork -> new admission decision computed, not reused.
- **Status:** UNADDRESSED

### `OB-25` — Eval thresholds for autonomy upgrade are absolute and ignore sample size, so a tier upgrades on 3 lucky passing traces.
- **Consequence:** Statistically unjustified autonomy escalation.
- **Guard:** Upgrade gate requires minimum coverage/sample-size and confidence bound, not just a pass rate; encoded in the signed manifest. Test: pass rate 100% on n=3 -> upgrade denied for insufficient coverage.
- **Status:** UNADDRESSED

### `OB-26` — Reserve-then-settle does not handle currency/unit mismatch (tokens vs USD vs API call quota) and reconciles across incomparable units.
- **Consequence:** Cost-to-the-cent reconciliation is meaningless; budget enforcement wrong.
- **Guard:** Cost is a typed multi-dimensional quantity (per-resource units + a priced USD projection at a captured rate); reconcile per dimension and on the captured price. Test: effect priced at a captured FX/token rate reconciles to the cent using that captured rate, not a live one.
- **Status:** UNADDRESSED

### `OB-27` — The autonomy ladder's gate is evaluated once at session start, not per-effect, so authority granted early persists even as context (provenance taint, budget burn) degrades within the session.
- **Consequence:** Mid-session a tainted input or budget overrun should block elevated effects but doesn't.
- **Guard:** Autonomy authority is recomputed at every AdmissionDecision from current ledger state (provenance, budget, recent eval/incidents), never cached for the session. Test: taint introduced mid-session -> next elevated effect re-gated and blocked.
- **Status:** UNADDRESSED

### `OB-28` — Replay reconstructs canvas/monitor from the ledger but the manifest version used at execution time was not recorded, so the fold uses the current manifest and renders a graph that never ran.
- **Consequence:** Canvas shows a topology different from what executed; violates the core promise.
- **Guard:** Every run pins the content-addressed manifest hash in its first event; all folds (canvas/monitor/audit) load that exact manifest version. Test: change manifest after a run -> replaying that run still renders the original topology.
- **Status:** UNADDRESSED

### `CH-14` — Outbound channel send is fire-and-forget: the adapter posts to the provider but no EffectResult with transport ACK is captured, so the ledger can't prove a notification/approval-request was actually delivered.
- **Consequence:** Audit claims user was asked for approval when the message never arrived; or replay can't reconcile delivery.
- **Guard:** Every outbound send is an effect with reserve-then-settle and a supervisor-co-signed delivery EffectResult containing the provider's transport ACK/message id. Test: provider returns no ACK -> effect settles as failed-delivery, not success.
- **Status:** UNADDRESSED

## Self-Host, Data Layer & Multi-Tenancy

### `SH-01` — The content-addressing hash is computed over a non-canonical serialization (e.g. JSON with unstable key order, float formatting, or unicode normalization differences between the SQLite and Postgres JSON codecs), so the same logical event hashes differently on the two backends.
- **Consequence:** A ledger exported from SQLite and re-imported into Postgres (or replayed cross-backend) shows hash mismatches, breaking the chain and making 'what you see is exactly what executed' provably false; signature verification fails on migrated data.
- **Guard:** Define ONE canonical bytes encoding (e.g. RFC 8785 JCS or deterministic CBOR) computed in application code BEFORE the DB touches it; never hash a value round-tripped through a DB JSON type. Add a cross-backend test that writes 10k events to both SQLite and Postgres and asserts byte-identical content hashes.
- **Status:** GUARDED — SQLite adapter: real on-disk durability across close/reopen (test 47)

### `SH-02` — RLS is the ONLY tenant-isolation mechanism, but a background job, migration runner, or admin task connects as the table owner / a BYPASSRLS or SUPERUSER role, for which Postgres silently ignores row security policies.
- **Consequence:** A single misconfigured connection leaks or cross-writes every tenant's signed events; tenant-of-one guarantee is broken without any error.
- **Guard:** Force every application connection (including migrations and jobs) to a role that is NOT the owner and has NOBYPASSRLS; add a startup assertion querying pg_roles for rolbypassrls=false and rolsuper=false on the app role, and a test that confirms a wrong-tenant SELECT returns 0 rows even for the migration role.
- **Status:** GUARDED — UNIQUE(tenant,offset) PK = atomic CAS append; concurrent offset → OptimisticConflict

### `SH-03` — SQLite has no row-level security, so the embedded default relies on application-level tenant filtering, but a query path (raw SQL, an ORM 'find by id', or a fold/reducer) forgets the tenant predicate.
- **Consequence:** On the self-hosted SQLite default — the path most users run — tenant isolation is enforced only by remembering a WHERE clause, and one omission cross-reads tenants.
- **Guard:** Make the data layer refuse to issue any query lacking a bound tenant_id (wrap the connection so every statement is parameterized through a tenant-scoped session object; reject raw SQL without the tenant binding). Add a test harness that runs every repository method and asserts the emitted SQL contains the tenant predicate.
- **Status:** GUARDED — offset = prev+1 in the txn, not a DB autoincrement (no skips)

### `SH-04` — app.current_tenant is set via a Postgres session GUC (set_config) for RLS, but the connection pool reuses physical connections across requests and the GUC is not reset, so a request inherits the previous request's tenant context.
- **Consequence:** Intermittent, load-dependent cross-tenant reads/writes that are nearly impossible to reproduce and pass all single-request tests.
- **Guard:** Use SET LOCAL inside an explicit transaction per request (auto-reset at commit/rollback), never SET. Add a concurrency test: interleave two tenants' requests on a 1-connection pool and assert no bleed; add a pool 'on-checkout' hook that asserts the GUC is empty.
- **Status:** GUARDED — on-disk tamper caught by verify()/selfCheck (HashMismatch test)

### `SH-05` — A schema migration runs ALTER/UPDATE against the append-only ledger table to 'fix' or backfill historical events, mutating bytes that were signed.
- **Consequence:** Every signature and hash-chain link after the mutated row becomes invalid; verification replay HALTs permanently; the ledger is corrupted and unrecoverable.
- **Guard:** Mark ledger tables as physically immutable: revoke UPDATE/DELETE from the app role, add an AFTER UPDATE/DELETE trigger that RAISES, and store schema evolution as a separate projection/view layer. Migration test must assert no migration ever touches rows in a hash-chained table.
- **Status:** UNADDRESSED

### `SH-06` — Migrations are applied in different orders / with different DDL on SQLite vs Postgres (e.g. SQLite lacks real ALTER COLUMN, JSONB, partial indexes, or CHECK semantics), so the two backends drift into different schemas with subtly different constraints.
- **Consequence:** A manifest or ledger valid on the SQLite default is rejected (or silently coerced) on Postgres after a user 'upgrades' their backend, producing data loss or non-reproducible folds.
- **Guard:** Maintain a single migration definition compiled to both dialects, and run a schema-diff test in CI that snapshots the resulting schema (columns, types, constraints, indexes) on both backends and asserts semantic equivalence after each migration.
- **Status:** UNADDRESSED

### `SH-07` — There is no migration version gate, so a newer binary opens an older on-disk SQLite/Postgres database (or vice-versa) and starts reading/writing against a schema it doesn't fully understand.
- **Consequence:** Silent partial reads, missing columns defaulting to null, or a downgrade writing data the older code can't parse — corrupting a user's only self-hosted copy.
- **Guard:** Store a schema_version and min_compatible_binary_version row; on startup, refuse to run (clear error, no writes) if DB version > binary expects, and run migrations forward only with an explicit confirmation. Test: launch old binary against new DB and assert hard refusal, not partial operation.
- **Status:** UNADDRESSED

### `SH-08` — A migration is applied without a transaction (or SQLite's implicit DDL transaction is broken by a non-transactional step), and the process is killed mid-migration.
- **Consequence:** The single self-hosted DB is left half-migrated with no atomic rollback; the user has no DBA and no replica; the product won't start and data may be unreadable.
- **Guard:** Wrap each migration in a transaction; for SQLite take an exclusive transaction and write a 'migration in progress' marker plus an automatic pre-migration backup file. Test: SIGKILL the process at randomized points during a migration and assert the DB is either fully old or fully new, never in between.
- **Status:** UNADDRESSED

### `SH-09` — Backup is a naive copy of the live SQLite file (cp) or a pg_dump taken without a consistent snapshot while the ledger is being appended.
- **Consequence:** The backup captures a torn write / partial WAL state; on restore the hash chain has a gap or a half-written event, and the supposedly authoritative backup fails verification.
- **Guard:** Backups MUST use the engine's consistent mechanism (SQLite Online Backup API / VACUUM INTO; Postgres pg_dump with a serializable snapshot or pg_basebackup). Restore-then-verify is part of the backup job: after every backup, replay-verify the whole hash chain and fail the backup if it doesn't reconcile.
- **Status:** UNADDRESSED

### `SH-10` — Restore does not preserve or re-validate signatures against the historical key state, so a backup restored after a key rotation can't verify events signed by the now-rotated/revoked supervisor key.
- **Consequence:** A legitimate restore appears tampered (all old events 'fail' verification) or, worse, the system accepts any signature because key history was dropped — destroying the audit guarantee.
- **Guard:** Include the full key history (with validity intervals and revocation timestamps) in the backup, content-addressed alongside the ledger. Verification must check each event's signature against the key that was valid at the event's notarized time. Test: rotate key, back up, restore, and assert all pre- and post-rotation events verify.
- **Status:** UNADDRESSED

### `SH-11` — The 'one binary' embeds a specific SQLite/Postgres client library version, and an upgrade bumps the embedded engine to a version with a different on-disk format, WAL format, or default page size.
- **Consequence:** After auto-update the binary cannot open the user's existing data file, or silently upgrades the file format so they can never downgrade — bricking offline installs.
- **Guard:** Pin and assert the storage engine format version on open; on a format-incompatible bump, run an explicit, backed-up, one-way migration with user confirmation. Test the full upgrade matrix (N-2 -> N) by opening real databases created by prior released binaries in CI.
- **Status:** UNADDRESSED

### `SH-12` — Offline minute-one depends on the embedded SQLite, but the build/runtime quietly falls back to needing a network call (license check, telemetry, remote key/clock service, schema fetch, or a CDN-hosted WASM module) before the first write succeeds.
- **Consequence:** The headline 'offline minute one' promise fails for air-gapped/self-host users; the product won't initialize without internet, contradicting the core pitch.
- **Guard:** Add a CI job that runs the artifact in a network-namespace with ALL egress blocked and asserts a full create-tenant -> append-event -> read-fold cycle succeeds. Lint the dependency graph for any startup-path network call.
- **Status:** UNADDRESSED

### `SH-13` — Postgres-for-everything means working/episodic/semantic/SOUL memory planes and the ledger share one database, but a heavy semantic-retrieval or vector query (or an unbounded memory table) exhausts connections / locks / disk shared with the ledger.
- **Consequence:** Memory workload starves or deadlocks the ledger append path; effects can't be recorded, halting the runtime and making the audit log lossy under load.
- **Guard:** Separate connection pools (and ideally schemas with per-pool statement_timeout) for ledger-append vs memory/retrieval; reserve a guaranteed connection for ledger writes. Load test: saturate retrieval and assert ledger append latency/availability stays within SLO.
- **Status:** UNADDRESSED

### `SH-14` — RLS policies are written for SELECT and the obvious tables but a new feature adds a table (e.g. a new memory plane, cost-reserve ledger, or correlation-id index) without a corresponding policy, defaulting Postgres to no-policy => owner-visible / app-role-blocked depending on FORCE settings.
- **Consequence:** Either a new table leaks across tenants or it's invisible to the app — both ship silently because nothing fails loudly.
- **Guard:** CI policy-coverage test: enumerate all tables in the tenant schema and assert each has FORCE ROW LEVEL SECURITY plus SELECT/INSERT/UPDATE/DELETE policies referencing current_tenant; fail the build if any table is uncovered.
- **Status:** UNADDRESSED

### `SH-15` — tenant_id is a guessable/sequential integer and is trusted from a request header or JWT claim that the data layer sets into the RLS GUC without re-binding it to the authenticated principal.
- **Consequence:** A user who flips the tenant id in a token or header reads another tenant's signed ledger — a direct IDOR-class cross-tenant breach in a 'tenant-of-one' product.
- **Guard:** Use opaque UUIDv4/v7 tenant ids; derive the RLS tenant binding ONLY from the verified principal-to-tenant mapping in core (the trusted Interaction Resolver), never from a raw client-supplied value. Test: forge a token with another tenant's id and assert zero rows returned.
- **Status:** UNADDRESSED

### `SH-16` — The monotonic notarized clock is sourced from the host wall clock for the embedded/offline case, and an attacker (or NTP step / DST / suspend-resume) moves the system clock backward.
- **Consequence:** Event timestamps go non-monotonic; the hash/sequence ordering disagrees with timestamps; cost-window and park-until-event correlation break; the known 'attacker clock' hole materializes in the data layer.
- **Guard:** Persist a monotonic high-water-mark (last_notarized_time) in the DB and refuse to admit any event with a timestamp <= it; advance only via the notarized clock source. Test: set host clock backward and assert appends with stale times are rejected, not silently accepted.
- **Status:** UNADDRESSED

### `SH-17` — The hash-chain head (the 'latest event' pointer used to link the next event) is read and written without serialization, so two concurrent appends both link to the same parent.
- **Consequence:** The append-only log forks into two heads; replay/verification can't determine the canonical sequence; 'what executed' is ambiguous.
- **Guard:** Enforce a single linear chain via a unique constraint on (tenant_id, parent_hash) or an explicit advisory lock / SELECT ... FOR UPDATE on the chain-head row, with append in a serializable transaction. Concurrency test: fire N simultaneous appends and assert exactly one linear chain with no duplicate parent links.
- **Status:** UNADDRESSED

### `SH-18` — SQLite is run with default journaling (rollback journal, synchronous=NORMAL on some platforms) and the OS/power fails after the runtime acked an EffectResult append.
- **Consequence:** An acked, supervisor-signed effect record is lost on crash recovery, creating a 'crash hole' the architecture says must HALT — but here it's silently gone, so a real side-effect (money spent) has no ledger record.
- **Guard:** Force synchronous=FULL (or EXTRA) and WAL with checkpointing for the ledger; only ack an effect after the append is durably fsync'd. Crash test: kill power (simulated via fault injection) after ack and assert the event survives recovery; otherwise the runtime must detect the hole and halt+re-gate.
- **Status:** UNADDRESSED

### `SH-19` — Upgrades change the canonical serialization or the set of fields included in the hash (e.g. adding a new field to EffectResult), without a versioned hashing scheme.
- **Consequence:** All historical events fail to re-verify under the new binary because they're re-canonicalized with the new rules; replay-verification of pre-upgrade data breaks after every feature that touches event shape.
- **Guard:** Embed a hash_scheme_version inside each event and select the canonicalization/verification algorithm by that version forever; never re-canonicalize old events with new rules. Test: events written by every prior released schema version must still verify under the current binary.
- **Status:** UNADDRESSED

### `SH-20` — Restore/import accepts a backup file without verifying its provenance (the offline 'authenticity' hole), so a tampered or swapped ledger file is imported and trusted because it's internally self-consistent.
- **Consequence:** An attacker who can write the backup directory substitutes a forged-but-self-consistent ledger; the system replays it as authoritative truth.
- **Guard:** Sign the backup manifest with a custody key separate from the per-event keys and verify on restore; record an external anchor (e.g. periodic notarized digest) so a wholesale swap is detectable. Test: tamper one event and re-sign the chain with a local key; restore must reject due to missing/invalid custody anchor.
- **Status:** UNADDRESSED

### `SH-21` — The ledger and its derived projections (canvas/monitor/cost/audit/memory materialized views or cache tables) drift because a projection is updated outside a fold (a write path mutates a projection table directly).
- **Consequence:** 'Canvas/monitor/cost are pure reads' becomes false; the UI shows state that no event produced, and reconcile-to-the-cent cost verification fails.
- **Guard:** Make all projection tables strictly derived: rebuildable purely by replaying the ledger, with no direct writers (revoke write grants except to the fold worker). Test: drop and rebuild every projection from the ledger and assert byte-equality with the live projection.
- **Status:** UNADDRESSED

### `SH-22` — Cost reserve-then-settle stores reservations in a table that is not crash-consistent with the EffectRequested append (two separate transactions), so a crash between them leaks a reservation or settles without a reserve.
- **Consequence:** Budget accounting drifts; reserved-but-never-settled amounts accumulate (false 'out of budget') or effects run with no reservation (overspend), violating reconcile-to-the-cent.
- **Guard:** Write the reservation and the EffectRequested event in the SAME transaction (or as one composite event). Test: inject crash between reserve and settle and assert post-recovery the reserve is either fully present and re-gated or fully absent.
- **Status:** UNADDRESSED

### `SH-23` — The single-use branch-scoped correlation id for park-until-event is stored without a uniqueness/consumed guarantee, so a duplicated or replayed inbound event resumes a parked branch twice.
- **Consequence:** An effect tied to a suspension fires twice (double payment, double message); the 'single durable suspension primitive' is not actually single-use.
- **Guard:** Enforce a unique constraint on correlation_id with an atomic compare-and-set 'consumed' flag in the resume transaction. Test: deliver the same resume event twice concurrently and assert exactly one branch resumes and the second is rejected as already-consumed.
- **Status:** UNADDRESSED

### `SH-24` — Multi-tenancy uses shared tables with tenant_id, but indexes (and the query planner stats) are not tenant-aware, so one large tenant's data degrades query plans / autovacuum for all tenants in the self-hosted single DB.
- **Consequence:** Noisy-neighbor: one tenant's huge ledger makes another tenant's folds slow or causes lock/bloat issues; 'tenant-of-one' isolation is logical but not performance-isolated.
- **Guard:** Lead every index with tenant_id, set per-tenant statement timeouts, and (for Postgres) consider partitioning by tenant; for the embedded default, document and test the single-tenant assumption. Load test with skewed tenant sizes asserting per-tenant fold latency SLOs.
- **Status:** UNADDRESSED

### `SH-25` — Encryption-at-rest for the embedded SQLite file is assumed but not implemented (or the key is stored next to the DB file), while the product markets self-host as secure.
- **Consequence:** Anyone with filesystem access reads the entire signed ledger including memory planes and any captured sensitive effect payloads; a stolen laptop = full data breach.
- **Guard:** Either encrypt the file with a key from the OS keystore / passphrase (not co-located), or explicitly document no-at-rest-encryption. Test: assert the on-disk file does not contain known plaintext event payloads when encryption is enabled.
- **Status:** UNADDRESSED

### `SH-26` — The migration runner and the fold/projection rebuild both run on startup, and on a large ledger the projection rebuild blocks readiness for many minutes with no progress, looking like a hang.
- **Consequence:** Self-hosters kill the process mid-rebuild (see SH-08), or the platform is unavailable after every upgrade for an unbounded time, eroding trust.
- **Guard:** Make projection rebuilds incremental/resumable (checkpoint the last-folded event seq) and serve reads from the existing projection while rebuilding in the background. Test: upgrade with a 1M-event ledger and assert readiness within SLO and resumability after a kill.
- **Status:** UNADDRESSED

### `SH-27` — tenant deletion (or data-export for GDPR/right-to-be-forgotten) is expected, but the ledger is append-only and content-addressed, so there is no defined way to erase one tenant's signed events without breaking everyone's chain (if chains are shared) or leaving orphaned hashes.
- **Consequence:** Legal/compliance failure: cannot honor deletion requests, OR deletion silently breaks hash continuity / cross-references; both are launch-blocking for a self-host product that promises to avoid legal risk.
- **Guard:** Chain per-tenant (separate hash chains, never cross-tenant links) and implement crypto-shredding: per-tenant data key whose destruction renders that tenant's events unreadable while leaving structure intact. Test: delete a tenant key and assert that tenant's events are unreadable and no OTHER tenant's chain verification breaks.
- **Status:** UNADDRESSED

### `SH-28` — The migration baseline assumes a fresh DB, so users who installed an early pre-1.0 self-host build have no clean upgrade path (missing baseline migration, or migrations not idempotent / not re-runnable).
- **Consequence:** Early adopters — the most vocal community for a self-host tool — get stuck on an old version or must wipe data; word-of-mouth adoption (a stated goal) is poisoned.
- **Guard:** Make every migration idempotent and provide a tested baseline-detection that maps any released schema state into the migration sequence. CI: upgrade-from-every-released-version test starting from real DBs produced by each prior tag.
- **Status:** UNADDRESSED

### `SH-29` — Determinism tags and replay assume deterministic folds, but a projection/reducer uses a non-deterministic input from the DB itself (NOW(), random(), row order without ORDER BY, or backend-specific collation/sort) so the same ledger folds to different state on SQLite vs Postgres or across runs.
- **Consequence:** Verification replay 'reconciles' differently per backend; counterfactual replay diverges; 'what you see is exactly what executed' is backend-dependent.
- **Guard:** Ban non-deterministic SQL in folds (no NOW()/random()/implicit order); force explicit ORDER BY on (seq) and a fixed collation. Test: fold the same ledger on SQLite and Postgres and assert identical resulting state hash.
- **Status:** UNADDRESSED

### `SH-30` — Postgres connection-string / SSL / sslmode defaults differ from the embedded path, and on first 'graduate to Postgres' the app connects with sslmode=prefer to a remote DB, silently allowing unencrypted tenant data over the wire.
- **Consequence:** Self-hosters who point at a managed Postgres leak signed ledger data in plaintext on the network without any warning.
- **Guard:** Require sslmode=verify-full (or explicit opt-out with a logged warning) for any non-localhost Postgres host; assert at startup. Test: configure a remote host with sslmode unset and assert the app refuses or warns loudly, never silently connects plaintext.
- **Status:** UNADDRESSED

### `SH-31` — The fold worker reads the ledger with READ COMMITTED while appends continue, so a long-running projection rebuild sees a moving target and produces a projection that corresponds to no single ledger prefix.
- **Consequence:** Projection reflects a torn view (some later events included, some earlier excluded), so cost/audit numbers are internally inconsistent and irreproducible.
- **Guard:** Run rebuilds at a fixed snapshot: SERIALIZABLE/REPEATABLE READ transaction or fold strictly up to a pinned max seq captured at start. Test: append continuously during a rebuild and assert the resulting projection exactly equals folding the ledger truncated at the pinned seq.
- **Status:** UNADDRESSED

### `SH-32` — Disk-full on the single self-hosted volume during an append leaves SQLite/Postgres in a state where the ack path partially succeeded, or the runtime keeps trying to run effects it can no longer record.
- **Consequence:** Effects execute (money/messages) but cannot be logged because the disk is full, producing untracked side-effects — the worst class of ledger hole for an audit-first product.
- **Guard:** Reserve headroom and treat 'cannot durably append' as a hard HALT before any effect is requested (check writability/space pre-flight in the effect-runner gate). Test: fill the volume and assert the runtime halts and requests no new effects rather than running un-loggable ones.
- **Status:** UNADDRESSED

<a name="cost-trust"></a>
## Cost Predictability & Trust Legibility

### `COST-01` — Pre-flight estimate is computed only for the first node of a graph, not for fan-out/loop/recursion nodes, so a manifest with a while-loop edge or a map-over-N-items node reports an estimate that is a tiny fraction of actual spend.
- **Consequence:** User approves a $0.40 estimate, the loop runs 300 iterations, and the real bill is $120. Trust in the estimator collapses on the first real workload.
- **Guard:** Estimator must walk the manifest graph and for every cycle or fan-out edge require either a declared static upper bound (max_iterations / max_fanout) or refuse to estimate (return UNBOUNDED). Test: a manifest with an unbounded loop and no declared bound MUST fail estimation, not return a finite number.
- **Status:** UNADDRESSED

### `COST-02` — The pre-flight estimate uses per-token prices from a hardcoded table that drifts from the model provider's actual current pricing, and there is no provenance/timestamp on the price snapshot used.
- **Consequence:** Estimates are systematically wrong after any provider price change; reserve-then-settle reconciliation shows persistent drift that nobody can attribute to a stale table.
- **Guard:** Every estimate must embed the price-table version hash and fetch timestamp; settlement compares EffectResult real cost against the price-table version used and emits a PriceTableDrift event when the implied unit price differs beyond epsilon. Test: changing the price table without bumping its version hash fails CI.
- **Status:** UNADDRESSED

### `COST-03` — Hard ceiling is checked AFTER the effect is requested rather than before, so an effect that overshoots the budget still executes and only the NEXT effect is blocked.
- **Consequence:** Every ceiling is effectively soft by one effect; a single expensive call (e.g. a 200k-token completion) blows the cap before any block fires.
- **Guard:** AdmissionDecision must reserve the effect's MAX possible cost against the ceiling BEFORE EffectRequested is emitted; if reserve would exceed the ceiling, park-before-spend. Test: a single effect whose max-cost exceeds remaining budget must produce a Parked event and zero EffectRequested events.
- **Status:** UNADDRESSED

### `COST-04` — Reservation uses the EXPECTED cost (mean token estimate) rather than the MAXIMUM possible cost (max_tokens * output price + full input), so variance regularly pushes actual spend past the ceiling.
- **Consequence:** Ceilings are routinely breached by 20-50% on high-variance workloads; 'hard ceiling' is a misnomer and a compliance/billing liability.
- **Guard:** Reserve = worst-case cost computed from the effect's declared max_tokens and max_input, not a point estimate. Test: property test asserting settled_cost <= reserved_cost for 10,000 randomized completions; any violation fails.
- **Status:** UNADDRESSED

### `COST-05` — Settlement (EffectResult real cost) is not subtracted from the reservation atomically; the reserve is released and the actual debited in two steps, and a crash between them leaves the budget double-counted or un-released.
- **Consequence:** After crashes, the available budget silently drifts up (lost reservations) allowing overspend, or down (lost releases) blocking legitimate work; cost-to-the-cent reconciliation fails.
- **Guard:** Reserve-release and actual-debit must be one append-only ledger transition (single signed event carrying both reserved_delta and actual_delta). Test: kill the process between reserve and settle; on recovery the budget fold must equal sum(actual) for completed effects + sum(reserve) for in-flight, exactly.
- **Status:** UNADDRESSED

### `COST-06` — Verification replay (the zero-side-effect mode) re-invokes the model/effect runner instead of replaying the captured EffectResult, causing real API spend during what is documented as a free, side-effect-free replay.
- **Consequence:** Users run audits/replays and get billed; the core promise 'verification = zero side effects' is false and erodes all trust.
- **Guard:** In verification mode the effect-runner MUST be hard-disabled at the transport layer (no network egress); replay reads EffectResult from the ledger by content hash. Test: run verification replay with a network namespace that blocks all egress and a billing-meter assertion that spend == 0; any outbound call fails the test.
- **Status:** UNADDRESSED

### `COST-07` — Counterfactual replay (the forking, re-gating mode) is accidentally triggered from a UI control labeled the same as verification replay, or defaults to counterfactual, so users incur real spend expecting a free audit.
- **Consequence:** Accidental real spend on what the user believed was a read-only review; surprise bills.
- **Guard:** The two replay modes must be distinct API endpoints with distinct types; counterfactual replay requires an explicit spend-acknowledgement token and shows a fresh pre-flight estimate before any effect. Test: counterfactual replay without the acknowledgement token returns 403 and emits zero effects.
- **Status:** UNADDRESSED

### `COST-08` — Cost is tracked per-effect but the cost of the orchestration kernel's own LLM-compiler calls (NL->manifest, distillation, eval-judging LLM) is excluded from the ceiling and the estimate.
- **Consequence:** A user's $5 ceiling governs only 'agent' calls while compiler/distillation/judge calls spend another $8 invisibly; total bill is unexplained.
- **Guard:** Every LLM invocation, including compiler, memory distillation, and eval-judge, must emit an EffectResult with a cost attributed to a named budget bucket; no code path may call a model without a reservation. Test: static check that the model client is only reachable through the metered effect-runner; grep/AST test forbidding direct SDK imports outside the runner.
- **Status:** UNADDRESSED

### `COST-09` — Currency and unit mismatches: provider returns micro-cents or different currency, but the ledger stores floating-point dollars, accumulating rounding error over thousands of effects.
- **Consequence:** Reconciliation 'to the cent' fails by accumulated float drift; long-running tenants show cents-to-dollars of unexplained variance.
- **Guard:** Store all costs as integer minor units (e.g. nano-USD) with explicit currency code; forbid float arithmetic in the cost fold. Test: sum 1,000,000 randomized nano-USD costs and assert exact integer equality against an independent big-integer computation.
- **Status:** UNADDRESSED

### `COST-10` — The pre-flight estimate does not account for retries: an effect that fails and is retried 3x costs ~3x, but the estimate assumes one attempt.
- **Consequence:** Workloads with flaky tools or rate-limit retries blow past estimate and ceiling because retries weren't reserved.
- **Guard:** Reservation must multiply worst-case cost by the configured max-retry count (or reserve incrementally and re-gate before each retry). Test: inject 100% transient failures up to max retries and assert settled cost <= reserved cost including retries.
- **Status:** UNADDRESSED

### `COST-11` — Parked-before-spend effects, when later resumed by a park-until-event, do not re-check the ceiling against the budget state AT RESUME TIME, using the stale reservation made at park time.
- **Consequence:** An effect parked for an hour resumes after other effects have consumed the budget, and overspends because its old reservation is honored.
- **Guard:** On resume, the kernel must re-run AdmissionDecision against current budget; the original reservation is voided at park. Test: park effect A, spend the budget via effect B, resume A; A must re-park or be denied, not execute.
- **Status:** UNADDRESSED

### `COST-12` — Budget is enforced per-run but not per-tenant/per-day aggregate, so a user can launch 1000 runs each under the per-run ceiling and spend unboundedly in total.
- **Consequence:** Self-hosted operator gets a five-figure provider bill despite every individual run being 'within budget'.
- **Guard:** Ceilings must be enforceable at multiple scopes (effect / run / principal / tenant / time-window) with the most restrictive binding; all scopes reserve from the same fold. Test: 1000 sub-ceiling runs against a daily tenant cap must park the run that crosses the daily cap.
- **Status:** UNADDRESSED

### `COST-13` — The estimate shown to a non-expert is in tokens or model-internal units rather than money, and uses jargon (input/output tokens, context window) the user cannot map to a decision.
- **Consequence:** Non-expert cannot tell whether to approve; either rubber-stamps everything or abandons the product. Trust legibility fails for the target user.
- **Guard:** Approval UI must show money (in the tenant's currency), a worst-case ceiling, and a plain-language scope sentence ('up to $X, sending up to N emails, reading your calendar'). Test: snapshot/UX test asserting the approval surface contains a currency figure and an effect-class summary, never raw token counts as the primary number.
- **Status:** UNADDRESSED

### `COST-14` — The displayed cost in the monitor is a fold over EffectRequested events (intent) rather than EffectResult events (settled), so the live number shows intended/reserved spend that may never actually occur or may differ from settled.
- **Consequence:** Live cost disagrees with the final invoice; 'what you see is exactly what executed' is violated for the cost projection.
- **Guard:** Cost projection must clearly separate reserved (pending) from settled (final) and label them; the headline number reconciles to settled at run end. Test: the settled-cost fold at completion must equal the sum of EffectResult costs and equal the provider invoice line for that run.
- **Status:** UNADDRESSED

### `COST-15` — Supervisor-signed real cost in EffectResult is trusted to come from the plugin itself rather than from mechanically observed provider response headers/usage, so a buggy or malicious plugin under-reports its cost.
- **Consequence:** Plugins can hide spend; ceilings are bypassed by lying plugins; the 'plugins must not self-sign facts about their own behavior' rule is violated for cost.
- **Guard:** Cost in EffectResult must be derived by the supervisor from the provider's authenticated usage response, never from a plugin-supplied number. Test: a plugin that returns a fabricated low cost must be overridden by supervisor-observed cost; assert ledger cost == supervisor-observed, not plugin-claimed.
- **Status:** UNADDRESSED

### `COST-16` — Streaming responses are billed on output tokens but the run is cancelled mid-stream; the partial tokens already generated are not captured as cost, or are double-counted.
- **Consequence:** Cancelled runs either undercount (free partial generation) or overcount; ceiling math and refunds are wrong.
- **Guard:** On cancellation, settle the EffectResult with the provider's reported usage-so-far (or worst-case if unavailable) and tag it cancelled. Test: cancel mid-stream and assert settled cost equals provider partial-usage, with a determinism/cancelled tag present.
- **Status:** UNADDRESSED

### `COST-17` — The estimate ignores tool/effect costs that are non-LLM but metered (vector DB queries, external paid APIs, Modal sandbox compute seconds), counting only model tokens.
- **Consequence:** Compute- or API-heavy manifests show a near-zero estimate; the dominant cost is invisible until the bill arrives.
- **Guard:** Effect profile's 3-axis declaration must include a cost-model hook per side-effect class; estimator sums all metered classes, not just LLM. Test: a manifest using only a paid vector search and zero LLM must produce a non-zero estimate.
- **Status:** UNADDRESSED

### `COST-18` — Reserve-then-settle leaves orphaned reservations when a branch is abandoned by a conditional edge (the not-taken branch reserved budget that is never released).
- **Consequence:** Available budget slowly leaks down across runs with branching; legitimate effects get falsely park-blocked.
- **Guard:** Reservations must be branch-scoped and released when a conditional edge selects away from a branch; the fold treats not-taken branches as released. Test: a conditional that always takes branch A must leave zero residual reservation from branch B after the run.
- **Status:** UNADDRESSED

### `COST-19` — Cached LLM responses (prompt caching / response cache) are billed at full price in the estimate and reserve, but settle at the discounted cached price, so reservations are wildly over-conservative and block legitimate work.
- **Consequence:** Users hit ceilings that don't reflect real (cached) spend; throughput drops; or operators raise ceilings dangerously to compensate.
- **Guard:** Estimator and reservation must model cache-hit probability conservatively but settlement must use the provider's cache-discounted usage; emit a CacheSavings event reconciling the two. Test: a fully-cached run settles below its reservation and the difference is released, not lost.
- **Status:** UNADDRESSED

### `COST-20` — The 'hard ceiling park' uses a wall-clock or polling check that races with concurrent effects in parallel branches, so two effects each pass the check simultaneously and jointly exceed the ceiling.
- **Consequence:** Parallel workloads breach the ceiling under concurrency; the breach is non-deterministic and hard to reproduce.
- **Guard:** Reservation against the budget must be a serialized compare-and-reserve (single-writer ledger append or DB row lock / optimistic version), not a read-then-check. Test: fire N concurrent effects whose combined max-cost exceeds the ceiling; assert sum of admitted reservations never exceeds the ceiling.
- **Status:** UNADDRESSED

### `COST-21` — Replay reconciliation compares totals but not per-effect costs, so two offsetting errors (one effect overcounted, another undercounted) net to zero and hide systematic mis-attribution.
- **Consequence:** Audit says 'reconciles to the cent' while individual line items are wrong, defeating cost legibility and per-node attribution.
- **Guard:** Verification replay must reconcile cost PER EffectResult by content hash, not just the grand total. Test: inject a +X error on effect 1 and -X on effect 2; the per-effect reconciliation must flag both, not pass on the net.
- **Status:** UNADDRESSED

### `COST-22` — The pre-flight estimate is generated by the untrusted LLM compiler itself (e.g. 'the model guesses how much it will cost'), creating a conflict of interest and unbounded inaccuracy.
- **Consequence:** Estimates are arbitrary; capability monotonicity is irrelevant if the cost gate itself is set by the untrusted component.
- **Guard:** Estimation must be a pure kernel function over the manifest + price table, never an LLM call. Test: estimator must run offline with no model client available; presence of a model call in the estimate path fails CI.
- **Status:** UNADDRESSED

### `COST-23` — When the budget runs out mid-run, the run halts but leaves the system in a partially-applied irreversible state (half-sent emails, partial DB writes) with no indication of cost-vs-completion tradeoff to the user.
- **Consequence:** User sees 'budget exceeded' but the world has been half-mutated; trust and data integrity both suffer.
- **Guard:** Park-before-spend must fire BEFORE the irreversible effect, and the parked state must surface the remaining-cost-to-finish so the user can top up and resume the SAME suspension. Test: a run that parks on budget must have emitted zero irreversible EffectRequested beyond the park point.
- **Status:** UNADDRESSED

### `COST-24` — Top-up / ceiling-raise while a run is parked is applied to the wrong run or globally, so raising one run's budget silently raises others, or a raise is not bound to the specific parked correlation id.
- **Consequence:** Cross-run budget contamination; an attacker or careless user uplifts unrelated runs; spend authorization leaks.
- **Guard:** A ceiling raise must be a signed event scoped to a specific run/correlation id and re-gated on resume. Test: raise budget for run A while run B is also parked; B's available budget must be unchanged.
- **Status:** UNADDRESSED

### `COST-25` — The non-expert approval shows worst-case ceiling but the run almost always costs far less, training users that the number is meaningless theater, so when a run actually approaches worst-case they ignore it.
- **Consequence:** Cry-wolf: legible-cost UI becomes noise; users approve a genuinely expensive run by habit.
- **Guard:** Show BOTH expected and worst-case with calibrated language, and after each run record estimate-vs-actual to display a running accuracy badge. Test: estimator calibration test asserting actual falls within [expected_low, worst_case] for >=95% of a benchmark corpus; persistent miscalibration alerts.
- **Status:** UNADDRESSED

### `COST-26` — Provider returns usage in a different tokenizer/accounting than the estimator assumed (e.g. image tokens, tool-call tokens, system-prompt caching boundaries), so even single-shot estimates are off by a large constant factor.
- **Consequence:** Estimates for multimodal or tool-heavy calls are systematically wrong; reserve math undershoots and ceilings breach.
- **Guard:** Estimator must use the provider's official token-counting endpoint/library per modality, not a generic heuristic; estimate includes image/tool/system components. Test: estimate vs actual for a corpus of multimodal+tool calls must be within a tight tolerance per provider.
- **Status:** UNADDRESSED

### `COST-27` — Cost events are append-only but a compromised or buggy reducer for the cost projection silently drops events (e.g. unrecognized event version), undercounting spend without erroring.
- **Consequence:** New event schema versions cause invisible undercounting; ceilings stop working after an upgrade.
- **Guard:** The cost fold must be total over all EffectResult events and FAIL CLOSED (halt the run, block new spend) on any unrecognized/unparseable cost event rather than skipping it. Test: feed an unknown-version cost event; the fold must error and the kernel must park, not continue.
- **Status:** UNADDRESSED

### `COST-28` — Distillation/memory-write effects (a captured non-deterministic effect) incur model cost that is attributed to a background system budget the user never sees, so user-facing cost understates true cost of using the product.
- **Consequence:** User believes a task cost $0.30 but background memory distillation added $0.50; the legible number is a lie of omission.
- **Guard:** Background effects triggered by a user run must be attributed to that run's cost view (clearly labeled 'memory upkeep') or to a separately disclosed budget the user has acknowledged. Test: total disclosed cost across all buckets must equal sum of all EffectResult costs for the period; any unattributed effect fails the audit.
- **Status:** UNADDRESSED

### `COST-29` — The monotonic notarized clock is not used for time-windowed budgets, so an attacker-controlled or skewed local clock lets a daily/hourly cap reset early, enabling unlimited spend by manipulating time.
- **Consequence:** Time-window ceilings are bypassable on self-hosted nodes with a movable clock; the known offline-authenticity weakness directly enables overspend.
- **Guard:** Time-windowed budget resets must key off the notarized monotonic clock, not local wall time; a clock that jumps backward must not reset a window. Test: set local clock back 1 day; the daily budget window must NOT reset and spend must remain blocked.
- **Status:** UNADDRESSED

### `COST-30` — Untrusted-inbound-provenance data (e.g. a webhook/email payload) influences the effect parameters that drive cost (e.g. 'summarize this 2M-token document'), letting an external party inflate spend up to the ceiling.
- **Consequence:** Third parties weaponize the agent to burn the tenant's budget (cost-DoS) by sending huge or expensive inputs.
- **Guard:** Effects whose cost is driven by untrusted-inbound data must have a separate, lower per-effect cost cap and require step-up approval to exceed it; provenance tag bars laundering into unbounded spend. Test: an inbound payload sized to consume the full ceiling must park for approval, not auto-spend.
- **Status:** UNADDRESSED

### `COST-31` — Reservations are made in the agent's display currency but the provider bills in USD; FX rate at reserve time differs from settle time, so reconciliation drifts and ceilings denominated in local currency are inaccurate.
- **Consequence:** Tenants on non-USD ceilings see breaches/under-utilization from FX drift; 'to the cent' reconciliation impossible.
- **Guard:** Hold the canonical ledger cost in the provider's billing currency; convert for display only with a recorded FX snapshot; ceiling comparisons happen in canonical currency. Test: reconcile canonical-currency totals exactly; display conversions carry an FX-snapshot id and never feed the ceiling check.
- **Status:** UNADDRESSED

<a name="ip-legal"></a>
## IP / Legal / Licensing Hygiene

### `IP-01` — The NL->manifest compiler LLM emits a conditional-edge expression, retry/backoff scheduler, or consensus-voting topology that is a verbatim reproduction of a technique covered by an active software patent (e.g. a patented agent-orchestration or RAG-reranking method), because no patent-clearance gate exists between generation and persistence into the ledger.
- **Consequence:** Krelvan (and every self-hoster) ships infringing functionality; a patent holder issues cease-and-desist or sues, forcing a recall of the artifact and tainting the signed-ledger reproductions that now embed the infringing manifest hash.
- **Guard:** Maintain a curated denylist of patent-claim fingerprints (normalized algorithm signatures) and run every compiled manifest section through a structural-similarity check before it can be signed into the ledger; block + require human review on match, and record the clearance decision as a signed event.
- **Status:** UNADDRESSED

### `IP-02` — A built-in or community MCP plugin embeds source code copied from an AGPL-licensed project (e.g. an AGPL scraping or DB tool), and Krelvan distributes it inside the single downloadable self-host artifact without offering corresponding source.
- **Consequence:** AGPL Section 13 network-use clause triggers for every hosted/self-host operator; license violation forces source disclosure of the whole linked artifact or removal, and copyright holders can sue operators who never knew.
- **Guard:** CI gate that runs an SPDX/license scanner (e.g. scancode/licensee) over every plugin and dependency; fail the build on GPL/AGPL/SSPL/BUSL/CC-BY-NC/CC-BY-SA and any license not on an explicit allowlist (MIT/BSD/Apache-2.0/ISC/MPL-2.0); require a recorded waiver for exceptions.
- **Status:** UNADDRESSED

### `IP-03` — A transitive npm dependency changes its license between minor versions (relicense to BUSL or SSPL) and an unpinned/automated dependency bump pulls it in unnoticed.
- **Consequence:** The shipped artifact silently becomes non-redistributable; a later audit reveals months of releases violated the new license.
- **Guard:** Lockfile is mandatory and committed; CI diffs the license of every dependency against the previously-recorded license-manifest and fails on any change; the license-manifest itself is a signed artifact attached to each release.
- **Status:** UNADDRESSED

### `IP-04` — The LLM compiler reproduces, verbatim or near-verbatim, pseudocode or an algorithm from a research paper whose license is non-commercial or whose terms forbid implementation without attribution (e.g. CC-BY-NC, or a paper with a patent grant restriction).
- **Consequence:** Krelvan implements restrictively-licensed research; commercial use is barred, and the academic authors or their institution assert claims.
- **Guard:** Design rule: the compiler is forbidden to ingest paper text/pseudocode into prompts; prompt-provenance logging captures all compiler inputs, and a periodic audit asserts no paper-derived content entered the generation path. Test: scan generated kernel/eval code for known paper-snippet hashes.
- **Status:** UNADDRESSED

### `IP-05` — Generated agent code or manifest is attributed to the user/operator as sole author, but actually contains substantial LLM-emitted code derived from training on copyleft repos, so attribution and license declarations on user-exported manifests are false.
- **Consequence:** Users distribute manifests under MIT claiming sole authorship while embedding copyleft-derived fragments; downstream license fraud and copyright exposure.
- **Guard:** Every exported manifest carries a machine-readable provenance/attribution block listing compiler model+version and a disclaimer that generated content is not warranted clean; export is blocked until the user acknowledges the provenance record (signed event).
- **Status:** UNADDRESSED

### `IP-06` — Krelvan uses the Anthropic/OpenAI API to GENERATE training data or to build a competing model/eval-distillation, violating provider ToS clauses that forbid using outputs to train competing models.
- **Consequence:** Provider terminates API access for Krelvan and all hosted users; potential breach-of-contract claim.
- **Guard:** Explicit policy + code guard: distillation/memory-capture effects that persist LLM outputs are tagged with the source provider, and a lint rule forbids any pipeline that feeds provider outputs into model-training jobs; document the no-train-on-outputs rule in the effect-runner.
- **Status:** UNADDRESSED

### `IP-07` — The model-agnostic abstraction lets a user point Krelvan at a provider whose ToS forbids the specific use (e.g. automated high-volume agents, or use in regulated domains), but Krelvan surfaces no per-provider ToS constraint, so the manifest schedules disallowed usage.
- **Consequence:** Provider bans the user; Krelvan is seen as facilitating ToS violations across providers.
- **Guard:** Each model plugin ships a declarative ToS-constraint descriptor (rate caps, prohibited categories, no-PII flags); the kernel refuses to admit effects that violate the active provider's descriptor and records the rejection as a signed AdmissionDecision.
- **Status:** UNADDRESSED

### `IP-08` — Krelvan bundles model weights or tokenizer files (for a 'local' embedded model) whose license forbids redistribution or commercial use (e.g. Llama community license restrictions, or a non-commercial embedding model).
- **Consequence:** Redistribution of the self-host artifact violates the weight license; takedown of releases.
- **Guard:** No model weights are bundled in the artifact by default; weights are fetched at runtime from the vendor with the user accepting the vendor license; CI asserts the release tarball contains zero weight/tokenizer blobs.
- **Status:** UNADDRESSED

### `IP-09` — The product name 'Krelvan' (or logo/wordmark) collides with an existing registered trademark in the software/AI class, discovered only after launch and adoption.
- **Consequence:** Forced rebrand of a now-adopted product; loss of domain, package names, and SEO; possible damages.
- **Guard:** Pre-launch: clearance search across USPTO/EUIPO/WIPO in relevant Nice classes (9, 42) plus npm/PyPI/GitHub org name availability; record the cleared name + date; re-check before each major-market expansion.
- **Status:** UNADDRESSED

### `IP-10` — Memory planes (episodic/semantic/SOUL) persist user PII and content harvested from external sources (emails, calendars, drive files) without a lawful basis or retention limit, and there is no deletion path, violating GDPR right-to-erasure.
- **Consequence:** Data-subject erasure requests cannot be honored because the ledger is append-only and content-addressed; regulator fines up to 4% global turnover; self-host operators inherit unbounded liability.
- **Guard:** Design rule: PII is stored only by reference/encrypted with a per-subject key in a crypto-shreddable keystore (NOT inline in the immutable ledger); erasure = key destruction (crypto-shredding) recorded as a signed event; test that destroying the subject key renders all referencing ledger entries unreadable.
- **Status:** UNADDRESSED

### `IP-11` — The append-only signed ledger embeds raw PII (names, emails, message bodies) directly in event payloads because 'what you see is what executed' demands full fidelity, making the ledger itself a GDPR-non-compliant immutable PII store.
- **Consequence:** Immutability and erasure are in direct conflict; no compliant way to delete; the core architecture is legally unshippable in the EU.
- **Guard:** Architectural invariant enforced by schema validation: ledger events store only hashes/encrypted-blob-references for any field classified as personal data; a CI test feeds known PII through a sample run and asserts no plaintext PII appears in serialized ledger bytes.
- **Status:** UNADDRESSED

### `IP-12` — Channel adapters (Telegram, email, etc.) ingest third-party messages and the untrusted-inbound content is later surfaced/redistributed by the agent, but Krelvan stores and republishes copyrighted third-party content (forwarded articles, images) without rights.
- **Consequence:** Copyright/DMCA claims against operators for storing and redistributing scraped or forwarded content.
- **Guard:** Inbound content carries the untrusted-provenance tag AND a copyright-risk tag; effects that redistribute externally are gated and require an explicit operator policy; retention TTL enforced on inbound copyrighted blobs.
- **Status:** UNADDRESSED

### `IP-13` — The deny-by-default plugin sandbox is bypassed by a plugin that statically links a GPL native library (e.g. a GPL PDF or crypto lib via FFI), creating a derivative work of the entire process.
- **Consequence:** GPL copyleft propagates to the Krelvan core process, forcing the whole runtime to be GPL — incompatible with the intended permissive distribution.
- **Guard:** Plugins run as separate processes communicating only over the MCP wire protocol (no in-process linking); build gate forbids native FFI linking of any GPL-licensed shared object into the core; license scan covers native deps, not just JS/TS.
- **Status:** UNADDRESSED

### `IP-14` — License attribution/NOTICE files for Apache-2.0 and BSD dependencies are not aggregated and shipped, violating the attribution clauses of otherwise-permissive licenses.
- **Consequence:** Even fully-permissive dependencies become a compliance violation; bad-faith plaintiffs (copyright trolls) target the missing NOTICE.
- **Guard:** Build step auto-generates a THIRD_PARTY_NOTICES file from the dependency tree (license + copyright + required notice text) and the release fails if any dependency's required-notice is absent; the file ships inside the artifact and is linkable from the UI.
- **Status:** UNADDRESSED

### `IP-15` — The compiler's system prompt, eval prompts, or 'SOUL' persona content was copied from a copyrighted prompt library, blog post, or another product's leaked prompts.
- **Consequence:** Copyright infringement on prompt text; competitor IP claims; embarrassment if leaked prompts are traced.
- **Guard:** All shipped prompts live in-repo with a provenance header (author + 'original work' assertion); a scan compares prompt hashes against a corpus of known public prompt-leaks; PR template requires asserting prompts are original or properly licensed.
- **Status:** UNADDRESSED

### `IP-16` — Krelvan copies UI/UX patterns, icons, or component code from a copyleft or proprietary design system (e.g. an icon set under CC-BY-ND, or paid Figma components used beyond license) into the canvas/monitor UI.
- **Consequence:** Design-asset license violation; asset vendor sends invoice or takedown.
- **Guard:** Asset manifest tracks every icon/font/illustration with its license and source URL; CI fails if any asset lacks a license entry or uses a No-Derivatives/Non-Commercial license; fonts are self-hosted only if license permits embedding.
- **Status:** UNADDRESSED

### `IP-17` — The 'optional sandboxed LangGraph adapter' links LangGraph (or any framework) whose license terms change, or whose use implies acceptance of telemetry/ToS that conflicts with the offline-first promise.
- **Consequence:** Offline-first claim is false (framework phones home); or a relicense breaks redistribution.
- **Guard:** Pin and license-scan the adapter dependency; integration test runs the LangGraph adapter with network fully blocked and asserts zero outbound connections; the adapter is isolated so it can be dropped without core changes.
- **Status:** UNADDRESSED

### `IP-18` — Scoped-token broker mints tokens for third-party SaaS APIs and Krelvan automates actions in ways the SaaS ToS prohibit (scraping, rate-limit evasion, ToS-forbidden automation of consumer accounts).
- **Consequence:** SaaS providers ban users and pursue CFAA/ToS breach claims; Krelvan labeled an abuse tool.
- **Guard:** Each broker-integration declares the target API's automation policy; the kernel rate-limits and refuses actions flagged ToS-prohibited; a per-integration compliance descriptor is required before a broker connector can be registered.
- **Status:** UNADDRESSED

### `IP-19` — Telemetry, crash reports, or 'anonymous usage' from self-host instances transmit PII or content to Krelvan servers without consent, breaking GDPR consent and the offline-first promise.
- **Consequence:** Unlawful processing; loss of trust; the offline guarantee is provably false.
- **Guard:** Telemetry is OFF by default and requires explicit opt-in recorded as a signed consent event; a network test asserts a default install makes zero outbound calls except user-configured providers; telemetry payloads are schema-validated to contain no free-text/PII fields.
- **Status:** UNADDRESSED

### `IP-20` — Cross-border transfer: a self-host operator in the EU configures a US-based model provider, and Krelvan transfers personal data to the US with no SCCs/adequacy mechanism surfaced or recorded.
- **Consequence:** Unlawful international transfer under GDPR Chapter V; regulator action against operators.
- **Guard:** Each model/effect plugin declares its data-residency/processing region; the kernel warns and records a signed acknowledgement when an EU-tagged data subject's data would cross to a non-adequate region; provide a region-pinning policy that blocks such effects.
- **Status:** UNADDRESSED

### `IP-21` — Krelvan copies API client code, SDK snippets, or example code from a provider's docs that are under a restrictive docs license (e.g. some docs are not Apache/MIT) into the core.
- **Consequence:** Provider-docs license infringement; subtle but real copyright claim.
- **Guard:** Rule: no copy-paste from vendor docs into shipped code; client integrations are written from the OpenAPI spec or generated by a license-clean generator; PR review checklist item asserts integration code is not doc-derived.
- **Status:** UNADDRESSED

### `IP-22` — The eval/test suite bundles copyrighted benchmark datasets (e.g. a dataset under research-only or NC license) to gate the autonomy loop, and these ship inside the artifact.
- **Consequence:** Dataset license violation distributed to every self-hoster; dataset owner C&D.
- **Guard:** Eval datasets in the artifact must be self-authored or permissively licensed (CC0/CC-BY); a manifest lists each dataset's license; CI blocks bundling any dataset with NC/research-only/ND terms; restricted datasets are fetched by the user under their own acceptance.
- **Status:** UNADDRESSED

### `IP-23` — Distillation (the 'captured non-deterministic effect') stores provider LLM outputs verbatim into semantic memory, and those outputs are themselves reproductions of copyrighted training data (e.g. song lyrics, article text), persisted and later re-served.
- **Consequence:** Krelvan becomes a redistribution vector for memorized copyrighted text; copyright claims against operators.
- **Guard:** Distillation outputs are tagged with provider+model provenance and a copyright-risk flag; a guard caps verbatim-length storage and runs a long-substring/known-corpus check before persisting; flagged content requires operator policy to retain.
- **Status:** UNADDRESSED

### `IP-24` — Patent marking / FOSS patent-grant mismatch: Krelvan ships under MIT (no patent grant) but a contributor holds a patent reading on their contribution, and there is no contributor patent grant.
- **Consequence:** A contributor (or their employer) later asserts a patent against users of code they themselves contributed; MIT gives users no defense.
- **Guard:** Adopt Apache-2.0 (explicit patent grant + termination on litigation) for the core, or require a DCO+CLA with a patent license; CI checks every contribution is covered by the signed contributor agreement before merge.
- **Status:** UNADDRESSED

### `IP-25` — Right-to-erasure vs replay: counterfactual replay re-feeds captured retrieval result-sets and memory that contain a now-erased data subject's data, resurrecting deleted PII during a replay months later.
- **Consequence:** Deleted personal data is re-processed and possibly re-exposed, defeating the erasure and violating GDPR.
- **Guard:** Replay reads PII only through the live keystore; if a subject's key is crypto-shredded, replay substitutes a tombstone and records 'redacted-on-replay'; test: erase a subject, run verification replay, assert no plaintext of that subject appears and cost still reconciles.
- **Status:** UNADDRESSED

### `IP-26` — The local-root signing key is treated as proof of authenticity, but a self-host operator's key signs facts asserting compliance/consent that were never actually obtained (e.g. fabricated GDPR consent events), and Krelvan presents these as authoritative.
- **Consequence:** Forged consent/audit records mislead regulators and downstream parties; Krelvan's 'trustworthy audit' claim is legally hollow and could be construed as facilitating fraud.
- **Guard:** Consent and other legally-significant events require a second, externally-anchored attestation (e.g. data-subject-side signature or notarized clock + assurance level), not just the local root key; UI/audit clearly marks single-key-signed legal claims as 'self-asserted, not independently verified'.
- **Status:** UNADDRESSED

### `IP-27` — Generated agents process special-category data (health, biometric, political) pulled from connected channels without recognizing the heightened GDPR Art. 9 / sensitive-data obligations, because the manifest has no data-classification concept.
- **Consequence:** Processing sensitive data without explicit consent/legal basis; severe regulatory exposure for operators in regulated verticals.
- **Guard:** Manifest sections carry a data-classification field; the kernel refuses to admit effects touching special-category-tagged data unless an explicit lawful-basis policy is attached; a classifier flags inbound sensitive content and forces gating.
- **Status:** UNADDRESSED

### `IP-28` — Krelvan re-implements a competitor's or prior-employer's proprietary agent-orchestration design that one of the founders/contributors had confidential access to, creating a trade-secret misappropriation claim.
- **Consequence:** Trade-secret litigation; injunction halting the project; the most existential legal risk.
- **Guard:** Contributor onboarding requires a signed clean-room/no-confidential-IP attestation; sensitive subsystems (kernel reducer, effect-profile model) are developed clean-room with documented independent provenance; maintain a design-decision log proving independent derivation.
- **Status:** UNADDRESSED

### `IP-29` — Community plugins in a marketplace carry no license metadata, and Krelvan installs/executes them without recording or surfacing their license, so operators unknowingly run/redistribute non-redistributable or virally-licensed code.
- **Consequence:** Operators inherit license violations from third-party plugins; Krelvan is the distribution channel and shares liability.
- **Guard:** Plugin registry rejects any plugin whose signed manifest lacks an SPDX license id; the installer displays the license and blocks install of denylisted licenses unless explicitly overridden; the license is recorded in the ledger at install time.
- **Status:** UNADDRESSED

### `IP-30` — DMCA/takedown unaddressed: as a plugin/manifest-sharing platform (hosted later), Krelvan hosts user-uploaded content that infringes, with no registered DMCA agent or takedown workflow.
- **Consequence:** Loss of safe-harbor protection; direct liability for hosted infringing content.
- **Guard:** Before any hosted sharing feature ships, register a DMCA agent, implement a notice-and-takedown endpoint with logged actions, and a repeat-infringer policy; gate the hosted-sharing feature flag on these existing.
- **Status:** UNADDRESSED

### `IP-31` — License compatibility within the artifact: an MPL-2.0 file is modified and combined with proprietary code without keeping the MPL file separately available, breaching MPL's per-file copyleft.
- **Consequence:** MPL file-level copyleft violated; source of those files must be disclosed; subtle and easy to miss.
- **Guard:** License scanner classifies weak-copyleft (MPL/EPL/LGPL) deps and the build asserts they are dynamically used / file-separable and that modified MPL files have their source published; document each weak-copyleft dependency's compliance method.
- **Status:** UNADDRESSED

### `IP-32` — Generated SQL/CEL/JSONLogic for conditional edges happens to reproduce a patented database or rules-engine optimization, or the JSONLogic/CEL library itself is under a license incompatible with redistribution.
- **Consequence:** Either patent exposure on the generated logic or license violation on the evaluation library bundled in the artifact.
- **Guard:** Pin the CEL/JSONLogic implementation to an Apache-2.0/MIT library and license-scan it; for generated logic, restrict the compiler to a bounded expression grammar that cannot express patented optimizations, and document the grammar as the single source of truth.
- **Status:** UNADDRESSED

<a name="adversary"></a>
## Adversarial / Cross-Cutting Sweep

### `XC-01` — The ledger is the single source of truth for cost, audit, AND identity/key-state, but key-rotation/revocation events and cost-settlement events are folded by independent reducers that disagree on event ORDERING when two events share the same notarized timestamp (tie). One reducer breaks ties by offset, another by content-address lexical order, so an effect signed near a revocation boundary is judged valid by the cost fold but invalid by the audit fold.
- **Consequence:** Two 'pure reads' of the same ledger render contradictory truth ('what you see is what executed' is violated across panels); audit says tampered, cost says fine.

The supervisor co-signs the OBSERVED effect but uses the author-declared class to decide WHETHER to require approval — so the author's lie is laundered through a genuine supervisor signature: the supervisor truthfully observes an effect it was told didn't need gating.
- **Consequence:** A money-transfer mislabeled 'read-only' executes without approval and gets a valid supervisor co-signature, making the bypass look fully legitimate in audit.
- **Guard:** Side-effect class must be derived from OBSERVED behavior (did egress/state-change occur?) and operator-reviewed at admission, not trusted from the author's declaration; flag and quarantine plugins whose observed class exceeds declared; test a lying-manifest plugin.
- **Status:** UNADDRESSED

### `XC-40` — All cross-area times (parks, budgets, decay, TTLs, eval windows) depend on the notarized clock, but the notarized clock's NOTARY is reached via the same egress proxy that plugins use. A plugin that can saturate or interfere with egress (within its allowlist) can delay/deny the notary heartbeat, indirectly driving the whole system into the degraded clock mode of XC-04 — using a low-authority capability to manipulate the global time substrate.
- **Consequence:** A constrained plugin weaponizes egress contention to attack the clock and thereby every time-gated guard system-wide.
- **Guard:** Notary/time-attestation egress must use a dedicated, prioritized, plugin-inaccessible channel; detect and alarm on notary-heartbeat delay independent of plugin egress; test plugin-induced egress saturation vs notary liveness.
- **Status:** UNADDRESSED

### `XC-41` — Krelvan is model-agnostic and lets operators point at any provider, but the cross-border data-transfer and provider-ToS constraints (IP-07/IP-20) are evaluated at compile time against the manifest, NOT at the per-EFFECT level. An untrusted-inbound task can, at runtime, cause a node to send EU personal data to a US/disallowed provider chosen by a fallback/model-routing rule that the compile-time check never saw.
- **Consequence:** Runtime model-routing breaches data-residency/ToS even though the manifest passed compile-time legal checks — a legal hole at the integration seam between routing and compliance.
- **Guard:** Per-effect, re-evaluate data-classification x destination-jurisdiction before egress; block disallowed routes at the effect-runner with a recorded compliance event; test a fallback-route that would send classified data to a disallowed region.
- **Status:** UNADDRESSED

### `XC-42` — The eval-gated autonomy loop, the cost reducer, and provenance are 'unified' in observability, but they're unified by JOINING on run/effect ids across separately-evolving schemas. A schema change to one (e.g., provenance adds a sub-tag) without a coordinated migration of the join logic causes the unified view to DROP rows that don't join, silently shrinking the population an autonomy upgrade is evaluated on (OB-25 sample-size, but caused by a join-miss not luck).
- **Consequence:** Autonomy upgrades evaluated on a silently-truncated, biased sample due to a schema/join seam; the gate is bypassed by an integration bug.
- **Guard:** Unified views must use OUTER joins with explicit null-accounting and assert row-count conservation against source tables; CI fails if a schema change reduces join coverage; test a provenance schema bump against the unified eval view.
- **Status:** UNADDRESSED
